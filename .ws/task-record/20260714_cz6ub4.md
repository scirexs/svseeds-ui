# Task: add a page-change hook (`onchange`) to Pagination

Target file: `src/lib/_svseeds/Pagination.svelte`

References (read before editing):
- `src/lib/_svseeds/NumberInput.svelte` — the sibling convention to follow:
  destructures `onchange: onchangeProp` out of `...rest` and fires a synthesized
  `new Event("change")` at the value-commit seam (`emitChange`, `hchange`).
- `src/lib/_svseeds/DateInput.svelte` — same convention (`const ev = new
  Event("change"); onchangeProp?.(ev as any);`).
- `.ws/policy/coding-style.md` §8–9 (handler typing, `as any` for event
  re-dispatch), §11 (`@component` doc), §16 (tests).
- `tests/Pagination.svelte.test.ts` — existing test conventions (direct
  `MouseEvent` dispatch via `fireEvent.click`, `getButton(label)` helper).

## Goal

Give callers an imperative hook that fires when the user switches the page, so
the content re-render that must accompany a page change can be triggered without
watching `value` from an `$effect`. Follow the library's established convention:
expose `onchange` and fire a native `change` `Event` — do not invent a new
callback shape.

## Done when

- `Pagination` invokes the caller's `onchange` with a native `change` `Event`
  exactly once for each **user-initiated** page change (any of the four nav
  buttons that actually moves, or a ComboBox commit that lands on a different
  page), and never otherwise. (C-1)
- `onchange` is destructured out of `...rest` so it is no longer passed through
  to `<nav>` as a bubbled native handler. (C-1)
- The `@component` doc reflects the hook consistently with sibling components. (C-2)
- Tests cover firing on each user source and the no-fire cases. (C-3)
- `bun run check`, the scoped test run, and scoped `bun run fmt:check` pass.

## Background / constraints

- Pagination's root is `<nav>` extending `HTMLAttributes<HTMLElement>`, so
  `onchange` is **already part of the prop type** — no new field needs to be
  added to `PaginationProps`. Today it is spread onto `<nav>` via `{...rest}`,
  where it only fires from native `change` events that bubble up from the
  embedded ComboBox's `<input>` — unreliable and silent for button clicks. This
  task replaces that pass-through with an explicit, controlled invocation.
- The page value settles at exactly two user-initiated seams: the four nav
  button handlers (`htop`/`hleft`/`hright`/`hbottom`) and `commit()` (invoked by
  the embedded ComboBox through `ctx.commit`). `sync()` (the `$effect.pre` clamp
  on `min`/`max`/`value` change) and any caller `bind:value` assignment are
  **programmatic**, not user-initiated.
- Must keep working: existing clamping, `bind:value`, ComboBox commit-on-blur /
  Enter / select, at-bound button suppression, and the options-following pattern
  documented in the `@component` Behavior.

## Detailed changes

### (C-1) Fire `onchange` on user-initiated page change

**Where:** the `$props()` destructure (line ~84), plus the value-mutation seams:
the four button handlers `htop`/`hleft`/`hright`/`hbottom` and `commit()`.

**Required behavior:**
- The `$props()` destructure MUST take `onchange: onchangeProp` (renamed to avoid
  shadowing) so `onchange` is **removed from `...rest`** and no longer wired to
  `<nav>`.
- The component MUST invoke `onchangeProp` with a **native `Event` of type
  `"change"`** (`new Event("change")`) — no custom payload; callers read the new
  page from bound `value`.
- `onchangeProp` MUST be invoked **once** whenever a user action changes the
  committed `value`:
  - a nav button (First = `htop` → `lo`, Prev = `hleft`, Next = `hright`, Last =
    `hbottom` → `hi`) that actually moves the page;
  - a ComboBox `commit()` whose parsed+clamped result differs from the current
    `value`.
- `onchangeProp` MUST NOT be invoked when:
  - a nav button is at its bound and performs no move (the existing `atMin` /
    `atMax` early-returns);
  - a `commit()` resolves to the **same** page, or reverts invalid/empty input;
  - `value` changes programmatically via the caller's `bind:value`;
  - `sync()` clamps `value` because `min`/`max` changed.

**Suggested implementation** (illustrative, non-binding):

```ts
// destructure: … onchange: onchangeProp, … , class: c, ...rest }
function emit() {
  onchangeProp?.(new Event("change") as any); // §9: as any for re-dispatch
}
```

Fire only on a real change by comparing before/after. E.g. in `commit()`:

```ts
function commit() {
  const prev = value;
  const next = parse(text);
  if (next !== undefined) value = clamp(next);
  text = String(value);
  if (value !== prev) emit();
}
```

and in each button handler, `emit()` after the assignment when the value moved,
e.g.:

```ts
const hright: MouseEventHandler<HTMLButtonElement> = () => {
  if (atMax) return;
  const prev = value;
  value = clamp(value + 1);
  if (value !== prev) emit();
};
```

A single small helper that clamps, compares, assigns, and emits (routing all
four buttons and `commit()` through it) is acceptable if it stays localized and
does not alter the existing at-bound suppression or clamping behavior.

**Notes:** `emit()` runs synchronously inside the handler, before the batched
`$effect.pre`/`sync()` runs; `value` is already the settled target at that point.
`onchange` inherited from `HTMLAttributes<HTMLElement>` is typed as a change
handler, so the `new Event("change")` argument needs `as any` (coding-style §9,
the gold-standard re-dispatch case) — do not widen or re-type the prop.

**Risk:** an existing caller that passed `onchange` expecting the *bubbled*
native change from the inner input will now receive the controlled invocation
instead (fires on buttons too, and only on real page changes). This is the
intended correction — see Accepted risks.

### (C-2) Document the hook

**Where:** the leading `@component` comment.

**Required behavior:** add a Behavior bullet stating that `onchange` fires a
native `change` `Event` on user-initiated page changes (nav buttons and ComboBox
commit) and that the new page is read from bound `value`; note it does **not**
fire on programmatic `value` changes or bound clamping. Keep wording consistent
with sibling components. Do not remove the existing options-following Behavior
bullet (it remains accurate). Optionally note `onchange` in the Types comment
block; if added there, keep the `@component` Types block and `PaginationProps` in
sync per coding-style §11 (no new interface field is required, since it is
inherited).

## Invariants / expected behavior

- Clamping, `bind:value`, ComboBox commit paths, at-bound suppression, and the
  options-following pattern are unchanged apart from the added `onchange` firing.
- `onchange` fires **after** `value` has settled, so a handler reading bound
  `value` sees the new page.

## Out of scope

- No new field added to `PaginationProps` (`onchange` is inherited).
- No custom event name or typed page-payload callback (e.g. `onpagechange`).
- No change to `ComboBox.svelte`, `_core.ts`, or the ComboBox context contract.
- No firing on programmatic/clamp value changes.
- No refactor of `thin()`, `sync()`, or the button/ARIA structure beyond adding
  the emit.

## Accepted risks / tradeoffs

- The hook is scoped to user-initiated commits (native-`change` semantics),
  deliberately not a general "value changed" signal — callers needing to react to
  programmatic `value` changes still use `bind:value` reactivity. This avoids the
  echo/clamp noise that effect-watching produces and is the reason the hook was
  requested.
- Repurposing the inherited `onchange` from a bubbled pass-through to a controlled
  invocation is a behavior change for any caller relying on the old bubbling; this
  is intended.

## Tests

`render` / `getButton` / `fireEvent.click` helpers:

- `onchange` fires once per moving nav button — First, Prev, Next, Last — with an

  page at call time (e.g. spy pushes `value`).
- `onchange` fires when a ComboBox commit lands on a different page (set the
  combobox input value then blur / Enter, per existing commit tests).

  - a ComboBox commit resolves to the current page or to reverted invalid input;
  - `value` is changed via `rerender`/props (programmatic), and when a `max`
    decrease clamps `value` through `sync()`.
