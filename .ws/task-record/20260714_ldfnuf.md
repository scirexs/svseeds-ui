## Goal

The calendar overlay (`.bottom`) is absolutely positioned but, when it does **not**
overflow the viewport, its inline style is bare `position:absolute;` with no
offset ‚Äî so it renders at its static position, overlapping `.whole` instead of
dropping below the input. Fix the default so the calendar appears **below the
input, left-aligned**, without the caller having to add any CSS; flip to
above/right only on viewport overflow (unchanged trigger). Make the offsets
**overridable in plain CSS without `!important`** and expose the flip state so
callers can also style the overflow case. Separately, register the document
listeners through Svelte's `on()` and one-box their teardown into the `$effect`
cleanup.

## Done when

- With no caller styling, opening the calendar renders it **below** the input
  (default anchor `top` + `left`), not overlapping `.whole`.
- On viewport overflow the overlay still flips: bottom-overflow ‚Üí renders above
  (`bottom`), right-overflow ‚Üí right-aligned (`right`); the two axes flip
  independently.
- The offset values are routed through caller-overridable CSS custom properties
  (default names `--svs-position-x` / `--svs-position-y`), renameable via a new
  `cssvar` prop, and the overlay carries `data-svs-flip-x` / `data-svs-flip-y`
  when the respective axis is flipped.
- Document `pointerdown` / `keydown` listeners are registered via `on()` from
  `svelte/events`; `stopListening` and `onDestroy(stopListening)` are gone and
  teardown runs from the `$effect` cleanup.
- `bun run check`, the scoped DateInput test, and the scoped format check pass.

## Background / constraints

- `.whole` keeps its inline `style="position:relative;"` ‚Äî it is the overlay's
  containing block and must stay. Do not change it.
- The overflow detection path is unchanged: `observeOverflow()` still resets
  `overflow` to `{ x: false, y: false }`, `await tick()`, then measures via
  `_detectOverflow(overlayElem)` (SSR-guarded by the existing
  `typeof window === "undefined"` check ‚Äî keep it).
- `_core.ts` needs **no** change: `_cssVar` and `_detectOverflow` already exist
  there.
- Keyboard/Escape and outside-pointerdown dismissal behavior must keep working
  exactly as today; only the registration mechanism changes.
- The `$effect` here is genuinely reactive (gated on `open`): listeners are added
  when `open` becomes true and must be removed on every `open ‚Üí false` transition
  **and** on destroy. The `$effect` cleanup covers both (Svelte runs effect
  teardown on unmount), so per coding-style ¬ß5 this reactive teardown belongs in
  the `$effect` cleanup, **not** in `onDestroy`.

## Detailed changes

### (C-1) Default anchor + overridable placement

**Where:** `<script module>` exports, the instance `<script>` `$props()` +
Reactive Handlers, and the overlay `<div class={cls(PARTS.BOTTOM,‚Ä¶)}>` in the
template.

**Required behavior:**

- The overlay MUST default to anchoring **below and left** (`top` + `left`) when
  neither axis overflows, and MUST flip **per axis** to `bottom` (y-overflow) and
  `right` (x-overflow), matching the existing `overflow` state.
- Each offset MUST be emitted as a CSS custom property with a computed fallback so
  a caller can override it in ordinary CSS without `!important`. Default property
  names MUST be `--svs-position-x` (fallback `0%`) and `--svs-position-y`
  (fallback `100%`), and MUST be renameable via a new optional `cssvar` prop of
  shape `cssvar?: Partial<Record<DateInputCssVar, string>>` where
  `DateInputCssVar = "x" | "y"`. Resolve names with `_cssVar` (direct-consume
  form, per coding-style ¬ß13.1), exactly like Slider.
- The overlay element MUST carry `data-svs-flip-x` when the x axis is flipped and
  `data-svs-flip-y` when the y axis is flipped, and MUST omit each attribute when
  that axis is not flipped (presence/undefined, per Popover's idiom).
- `DateInputCssVar` MUST be exported from `<script module>` (like `SliderCssVar`)
  and added to the `@component` Types block (C-3).

**Suggested implementation (illustrative):**

- Add the export near the other module exports:
  ```ts
  export type DateInputCssVar = "x" | "y";
  ```
- Add `cssvar` to `DateInputProps` (both the real interface and the `@component`
  Types copy), among the optional props next to the styling-related tail
  (mirroring Slider's `cssvar, attach, element, styling, variant` grouping):
  ```ts
  cssvar?: Partial<Record<DateInputCssVar, string>>; // custom-property names for the overlay x/y offsets; absent key uses default name
  ```
- Add `cssvar` to the `$props()` destructure (before `attach`).
- Import `_cssVar` from `./_core` (add to the existing value import from `./_core`).
- In **Reactive Handlers**, resolve the names and rebuild `overlayStyle`:
  ```ts
  const xVar = $derived(_cssVar(cssvar, "x", "--svs-position-x"));
  const yVar = $derived(_cssVar(cssvar, "y", "--svs-position-y"));
  const overlayStyle = $derived(
    `position:absolute;${overflow.y ? "bottom" : "top"}:var(${yVar},100%);${overflow.x ? "right" : "left"}:var(${xVar},0%);`,
  );
  ```
- On the overlay `<div>`, add the two data attributes alongside `style={overlayStyle}`:
  ```svelte
  data-svs-flip-x={overflow.x ? "" : undefined}
  data-svs-flip-y={overflow.y ? "" : undefined}
  ```

**Notes:** the fallback literals (`100%` / `0%`) reproduce today's flipped offsets,
so the default visual gains a below/left anchor while the overflow offsets are
unchanged in magnitude. `_cssVar(map, key, fallbackName)` returns the caller's
name when `cssvar.x`/`cssvar.y` is set, else the default name ‚Äî the value is read
as `var(<name>, <fallback>)`.

**Risk:** low. Existing overlay-flip tests assert the old bare
`position:absolute;` default and must be updated (T-1).

### (C-2) `on()`-based listener registration

**Where:** the `$effect` that runs while `open` is true, the `stopListening`
function, and the `onDestroy(stopListening)` call.

**Required behavior:**

- Document `pointerdown` (`houtside`) and `keydown` (`hkey`) listeners MUST be
  registered with `on()` from `svelte/events` while `open` is true, and MUST be
  removed on every `open ‚Üí false` transition and on destroy.
- The teardown MUST run from the `$effect` cleanup. `stopListening` and
  `onDestroy(stopListening)` MUST be removed, and `onDestroy` MUST be dropped from
  the `svelte` import if it becomes unused (it does).
- `observeOverflow()` and the `tick().then(() => focusCalendar())` call MUST keep
  running each time the effect (re)runs on open, as today.

**Suggested implementation (illustrative):**

```ts
$effect(() => {
  if (!open) return;
  const off = [on(document, "pointerdown", houtside), on(document, "keydown", hkey)];
  observeOverflow();
  tick().then(() => focusCalendar());
  return () => off.forEach((x) => x());
});
```

- Import: `import { on } from "svelte/events";` (svelte sub-path block, per ¬ß1).
- Change `import { onDestroy, tick, untrack } from "svelte";` to
  `import { tick, untrack } from "svelte";`.
- Delete the `stopListening` function and the `onDestroy(stopListening)` line.

**Risk:** low. `houtside` / `hkey` are referenced only inside the effect callback,
which runs after component init (when `open` is true), so no temporal-dead-zone
issue ‚Äî same as the current `addEventListener` code.

## Invariants / expected behavior

- Default (no overflow): overlay style resolves to
  `position:absolute; top:var(--svs-position-y,100%); left:var(--svs-position-x,0%);`
  and no `data-svs-flip-*` attribute is present.
- y-overflow: `bottom` instead of `top`, `data-svs-flip-y` present.
- x-overflow: `right` instead of `left`, `data-svs-flip-x` present.
- Dismissal (Escape, outside pointerdown), focus handling, `closeOnSelect`,
  `openOnFocus`, and all existing behavior unchanged.

## Out of scope

- `.whole`'s inline `position:relative;` ‚Äî leave as is.
- `Calendar.svelte`, `DateField.svelte`, and any other sibling ‚Äî do not touch.
- `_core.ts` ‚Äî no change (`_cssVar` / `_detectOverflow` already exist).
- The overflow detection algorithm and the `transition` prop path.

## Accepted risks / tradeoffs

- The overlay's chosen **side** (top/bottom, left/right) remains component-owned
  (automatic collision avoidance); callers tune the **offset magnitude** via the
  `cssvar` custom properties and can branch on the flipped state via
  `data-svs-flip-*`. Fully bespoke placement (overriding which side) still
  requires the caller to supply their own `children` or `!important`; this is an
  accepted bound, consistent with the headless-but-out-of-the-box goal.
