Target file: `src/lib/_svseeds/WheelPicker.svelte`

- `.ws/policy/coding-style.md` — project conventions (§4 branching, §6 naming,

## Goal

Let a user select a **visible but not-yet-selected** option by tapping/clicking it
on the drum, iOS-wheel style — not only by dragging or spinning. Tapping a nearby
row is faster than dragging to it. The behavior is **always on** (no new prop) and
animates to the tapped row, matching the existing selection-change motion.

## Done when

- A pointer press-and-release on `.middle` with negligible travel (a tap) selects
  the option under the release point and animates the drum to it.
- A tap on the already-selected (center) row changes nothing and fires no `change`.
- A tap on a `disabled` row selects the nearest enabled option (same rule as the
  drag-release snap).
- Dragging and wheel-spinning still snap exactly as before (no regression).
- `bun run check` passes and the WheelPicker test suite passes.

## Background / constraints

- `.middle` is the decorative drum (`aria-hidden`). It already owns
  `onpointerdown` / `onpointermove` / `onpointerup` / `onpointercancel` /
  `onwheel`. The sr-only `<select>` stays the source of truth; selection is
  committed by setting `element.selectedIndex` and dispatching a native `change`,
  exactly as `snap()` does today.
- Today a tap (press/release without meaningful drag) barely moves `pos`, so
  `hpointerup` → `snap()` just re-settles on the current selection — i.e. a tap is
  currently a no-op. This task makes a tap resolve to the pressed row instead.
- **What must keep working:** drag-to-snap, wheel-to-snap, native keyboard /
  typeahead selection, two-way `value` binding, `loop` seam clones, reduced-motion
  (jump instead of animate), and the pointer-capture drag flow.

## Scope of changes

- [ ] (C-1) In `hpointerup`, distinguish a **tap** from a **drag** by pointer
  travel and, on a tap, select the row under the release point; otherwise keep the
  existing `snap()`.
- [ ] (C-2) Add two hoisted control helpers: one that resolves a release point to

- [ ] (C-3) Update the `@component` *Behavior* comment to document tap-to-select.

No architecture-overview change (no component/API/layout/toolchain shift). No new

## Detailed changes

### (C-1) Tap-vs-drag in `hpointerup`

**Where:** the `hpointerup` const-arrow handler, and a new small constant next to
`POINTER_RATE` (in *Initialize*).

**Required behavior:**

- On pointer release, if the event is **not** a `pointercancel` **and** the
  vertical travel from press to release is within a small tap threshold, the
  handler MUST attempt a tap: resolve the row under the release point (C-2) and,
  if it yields a valid option index that differs from the current `selected`,
  commit that selection (C-2) and return **without** calling `snap()`.
- In every other case (a real drag, a `pointercancel`, or a tap that resolves to
  no row / to the already-selected row) the handler MUST fall through to the
  existing `snap()` — preserving today's behavior.
- Pointer-capture release MUST still happen on release as it does now.

**Suggested implementation** (illustrative):

```ts
const TAP_SLOP = 8; // px; max pointer travel to treat a release as a tap

const hpointerup: PointerEventHandler<HTMLDivElement> = (ev) => {
  if (!dragging) return;
  dragging = false;
  ev.currentTarget.releasePointerCapture?.(ev.pointerId);
  if (ev.type !== "pointercancel" && Math.abs(ev.clientY - startY) <= TAP_SLOP) {
    const idx = tapIndex(ev.clientX, ev.clientY);
    if (idx != null && idx !== selected) return pick(idx);
  }
  snap();
};
```

`startY` is already captured in `hpointerdown`. `TAP_SLOP`'s value MAY be tuned;
the contract is only that it is clearly below one row height so a deliberate drag
is never misread as a tap.

**Risk:** low — the new branch only fires on a low-travel, non-cancel release; all
other paths reach the unchanged `snap()`.

### (C-2) Hit-test and commit helpers

**Where:** among the hoisted control functions (near `snap` / `nearest`). Keep them
as `function` declarations (they are internal control helpers and are
forward-referenced from the `hpointerup` arrow — see coding-style §8).

**Required behavior:**

- `tapIndex(x, y)` MUST return the target option index for a release at viewport
  point `(x, y)`, or `undefined` when no drum row is hit:
  - It MUST use `document.elementFromPoint(x, y)` and locate the `labelElems`
    entry that **is** or **contains** the hit element; `shown[i]` and
    `labelElems[i]` are index-aligned.
  - From the matched row it MUST derive the option index via the existing
    `nearest(shown[i].virtual)`, so a hit on a `disabled` row resolves to the
    nearest enabled option and a hit on a `loop` seam clone resolves through the
    same modular logic as the drag-release snap.
  - It MUST guard SSR (`typeof document === "undefined"`) and return `undefined`
    when there is no hit or no matching row.
- `pick(index)` MUST commit `index` through the native select so the change routes
  through the existing `hchange` path (which sets `value`, animates to the new
  selection, and re-dispatches `onchange`): set `element.selectedIndex = index`
  and dispatch a bubbling native `change`. It MUST NOT set the `program` guard, so
  the move animates (and honors reduced-motion via the existing `animate` logic).

**Suggested implementation** (illustrative):

```ts
function tapIndex(x: number, y: number): number | undefined {
  if (typeof document === "undefined") return;
  const hit = document.elementFromPoint(x, y);
  if (!hit) return;
  const i = labelElems.findIndex((el) => el && (el === hit || el.contains(hit)));
  if (i < 0) return;
  return nearest(shown[i].virtual);
}
function pick(index: number) {
  if (!element) return;
  element.selectedIndex = index;
  element.dispatchEvent(new Event("change", { bubbles: true }));
}
```

**Notes:** `elementFromPoint` is a geometry query, unaffected by the active
pointer capture on `.middle`, and it returns the visually front-most row — which
is why hit-testing (not Y-to-row math) is used: it stays correct under the
`perspective` 3D transform where row spacing is non-linear. Rows beyond `maxAngle`
are `visibility: hidden` and are skipped by `elementFromPoint`, so they cannot be
tapped (the handler then falls through to `snap()`), which is the desired no-op.

**Risk:** low. Relies on `.aux` / `.extra` being `pointer-events: none` so the tap
reaches the label — already required for the existing drag to work across the
whole drum.

## Invariants / expected behavior

- The native `<select>` remains the single source of truth; focus stays on it
  (`hpointerdown` already focuses `element`).
- Drag-to-snap, wheel-to-snap, keyboard/typeahead selection, `value` binding,
  `loop` clones, and reduced-motion behavior are all unchanged.
- A tap commits via the same native-`change` route as keyboard selection, so a
  single tap fires `onchange` exactly once; a no-op center tap fires it zero times.

## Out of scope

- No new props (tap is always on) and no configurability toggle.
- No momentum / inertia, and no change to `snap`, `animate`, `nearest`,
  `normalize`, or `labelStyle` math.
- No `<style>` block, no CSS additions, and no changes to `_core.ts`.
- No change to the `<select>`, `options` rendering, `cssvar` mirror, or geometry
  measurement.

## Accepted risks / tradeoffs

- In `loop` mode, committing through the native change routes the animation to the
  option's **canonical** index — the same path any programmatic/keyboard selection
  already animates. So tapping a wrap-around **seam clone** row scrolls to the
  canonical position rather than the shortest visual path. This is consistent with
  existing `loop` selection-animation behavior and is an accepted edge case (seam
  clones sit at the drum extremes and are mostly `visibility: hidden`).
