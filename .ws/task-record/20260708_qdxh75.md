## Goal

Two independent improvements, delivered together:

- **Part A (production, direct):** give both drag-and-drop surfaces an **accessible name**:
  (A-1) `SortableGroup`'s `role="group"` wrapper accepts `aria-label` / `aria-labelledby`
  (and other group attributes) via a `{...rest}` passthrough with `class` merged — the exact
  gap the a11y review flagged, fixed the same way `ToggleGroup` was in `64fec3c`; and
  (A-2) `Sortable`'s list `<ul>` gains a `{...rest}` HTMLAttributes passthrough plus an
  explicit `ariaLabel?: string` recommendation prop (and `class` merge), matching the
  `Modal` / `Toggle` / `Drawer` precedent (explicit `ariaLabel` + `aria-label` Omitted from
  rest). Neither changes the Sortable controller or any drag behavior.
- **Part B (verification component):** add a **keyboard drag-and-drop** path (WAI-ARIA
  APG modal pickup/move/drop, incl. `Ctrl+Space` multi-select) in a **new copy**
  `ZSortableA11y.svelte`, so the high-risk keyboard work is validated in isolation without
  adding any keyboard/behavior change to the production `Sortable.svelte`. The copy ships in
  the package (Z-prefixed, not shown in docs/CLI suggestions) so the maintainer can use it as
  a real library component for validation.

The rationale for the copy approach is recorded as an ADR (Part D).

## Done when

- `SortableGroup` renders `<div class={[whole, c]} {...rest} role="group">`; a caller's
  `aria-label` / `aria-labelledby` and other div attributes reach the group element, and

- `Sortable`'s `<ul>` accepts a `{...rest}` passthrough (e.g. `aria-labelledby`), an explicit

- `ZSortableA11y.svelte` exists as a working copy of `Sortable` whose items are
  keyboard-focusable and reorderable via the APG modal grab/move/drop model below, with
  live-region announcements; the existing pointer behavior is unchanged.
- `.ws/architecture/overview.md` lists the new component; the ADR `003-*` exists.
- `bun run check` passes; the scoped tests pass; scoped `fmt:check` is clean.

## Background / constraints

- **Must keep working — production `Sortable`:** the only change to `Sortable.svelte` is the

  merge); its `SortableController`, pointer drag, multi-select followers, clone/swap/move modes,

- Keyboard DnD is genuinely high-risk against this pointer-centric controller (recent

## Detailed changes

### (A-1) SortableGroup accessible name

**Where:** `SortableGroupProps` interface, the `$props()` destructure, and the `<div>` in
`SortableGroup.svelte`. Also update the `@component` *Types* + *Anatomy* comment to match.

**Required behavior:**
- `SortableGroupProps` MUST extend `Omit<HTMLAttributes<HTMLDivElement>, "children" | "role">`
  (keep `class` **out** of the `Omit` so it merges). The existing required `children: Snippet`
  and the feature props (`group`/`duration`/`easing`/`styling`/`variant`) MUST remain.
- The `$props()` destructure MUST capture `class: c, ...rest` (one-line, `// prettier-ignore`,
  per coding-style §2).
- The group `<div>` MUST render `class={[cls(PARTS.WHOLE, variant), c]}` and spread `{...rest}`
  **before** the component-owned `role="group"`, so callers cannot override `role` but any
  `aria-label` / `aria-labelledby` / other div attrs pass through.
- `role="group"` MUST remain present and component-owned.

**Suggested implementation (illustrative):**
```svelte
<!-- interface -->
export interface SortableGroupProps extends Omit<HTMLAttributes<HTMLDivElement>, "children" | "role"> { … }
<!-- import type -->
import type { HTMLAttributes } from "svelte/elements";
<!-- $props -->
let { children, group, duration, easing, styling, variant = VARIANT.NEUTRAL, class: c, ...rest }: SortableGroupProps = $props();
<!-- markup -->
<div class={[cls(PARTS.WHOLE, variant), c]} {...rest} role="group">
```
**Notes:** `SortableGroup` does not re-emit any computed `aria-*`, so a plain `{...rest}`
passthrough is safe — the re-emit clobber gotcha
(`svelte-rest-props-reemitted-aria-undefined-clobber`) does **not** apply here; do not add
explicit `aria-*={undefined}` after `{...rest}`. Follow coding-style §14 (spread rest before
owned attrs; merge caller `class` last).
**Risk:** low; mirrors `ToggleGroup` `64fec3c`.

### (A-2) Sortable accessible name

**Where:** `SortableProps<T>` interface, the type imports, the `$props()` destructure, and the
`<ul>` in `Sortable.svelte`. Update the `@component` *Types* + *Anatomy* comment to match.

Adopt the exact `Modal` / `Toggle` / `Drawer` shape: a `{...rest}` HTMLAttributes passthrough
**plus** an explicit `ariaLabel?: string` prop that both signals "give this list a name" and
serves as the single `aria-label` source (raw `aria-label` is Omitted from rest so the two
cannot collide; `aria-labelledby` and any other attribute still flow through rest).

**Required behavior:**
- `SortableProps<T>` MUST extend
  `Omit<HTMLAttributes<HTMLUListElement>, "children" | "aria-label" | "onpointerdown" | "onpointerover" | "onpointerout" | "onpointerenter" | "onpointerup">`
  (keep `class` **out** of the `Omit` so it merges). Add an explicit optional
  `ariaLabel?: string` (an optional, so after the required `items` / `item` per coding-style
  §12). All existing props MUST remain.
- The `$props()` destructure MUST additionally capture `ariaLabel, class: c, ...rest`
  (one-line, `// prettier-ignore`).
- The list `<ul>` MUST render `class={[cls(PARTS.WHOLE, effVariant), c]}`, then
  `aria-label={ariaLabel}`, then spread `{...rest}` **before** the component-owned pointer
  handlers (`onpointerdown` … `onpointerup`) — so callers cannot override the drag semantics
  but may pass `aria-labelledby` / other attrs. When `ariaLabel` is unset the attribute
  resolves to none (`undefined`).
- No other change to `Sortable.svelte`: the `SortableController`, the pointer-handler bodies,
  the `<li>`/shadow template structure, and drag behavior MUST be untouched.

**Suggested implementation (illustrative):**
```svelte
<!-- interface -->
export interface SortableProps<T = string> extends Omit<HTMLAttributes<HTMLUListElement>, "children" | "aria-label" | "onpointerdown" | "onpointerover" | "onpointerout" | "onpointerenter" | "onpointerup"> {
  items: T[]; item: Snippet<[T, SVSVariant, Attachment]>; …; ariaLabel?: string; styling?: SVSClass; variant?: SVSVariant;
}
<!-- type import (svelte sub-paths group) -->
import type { HTMLAttributes } from "svelte/elements";
<!-- $props: add to the destructure -->
let { items = $bindable(), item, …, ariaLabel, dragging = $bindable(false), styling, variant = VARIANT.NEUTRAL, class: c, ...rest }: SortableProps<T> = $props();
<!-- markup -->
<ul class={[cls(PARTS.WHOLE, effVariant), c]} aria-label={ariaLabel} {...rest}
    onpointerdown={onPointerDown} onpointerover={onPointerOver} onpointerout={onPointerOut}
    onpointerenter={onPointerEnter} onpointerup={onPointerUp}>
```
**Notes:** `Modal`/`Drawer`/`Toggle` are the precedent (Omit `"aria-label"`, explicit
`ariaLabel` prop, `aria-label={ariaLabel}` before `{...rest}`). This is why `Sortable.svelte`
is copied to `ZSortableA11y.svelte` **after** A-2 — the copy inherits the whole passthrough and
names its own `<ul>` for free. `Pagination` (the one `ariaLabel`-without-passthrough exception)
is a separate future task and is **not** touched here.
**Risk:** low; additive interface/passthrough, no drag-behavior change.

### (B) ZSortableA11y.svelte — copy + keyboard DnD

**Where:** new file `src/lib/_svseeds/ZSortableA11y.svelte`, created by copying the
A-2-updated `Sortable.svelte` verbatim, then applying the changes below. Do not add any
keyboard/behavior change back into `Sortable.svelte`. Because the copy adds its own
`onkeydown` handler on the `<ul>` (B-2), its `SortableProps` `Omit` list MUST additionally
include `"onkeydown"` (extending the A-2 Omit set) so the component owns that handler.

**B-1 — Export namespacing (one rename only).**
- The copy MUST rename exactly one public export: the group factory
  `createSortableGroup` → **`createZSortableA11yGroup`** (update its internal call site in
  the instance script's `group ?? context ?? createZSortableA11yGroup()` fallback too).
  Rationale: the generated package index (`prep.ts` `makeIndex`) processes files
  alphabetically and de-dupes by export name, so an identically-named `createSortableGroup`
  in the copy would be silently dropped in favor of `Sortable`'s — leaving no way to create
  the keyboard-capable controller from the package and defeating cross-list verification.
  Renaming the factory keeps it reachable.
- All other identifiers MAY keep their original names. The duplicate **type** exports
  (`SortableProps`, `SortableReqdProps`, `SortableBindProps`, `SortableMode`,
  `SortableGroupController`) de-dupe harmlessly against `Sortable`'s structurally-identical
  originals (see Accepted risks). The `_`-prefixed internals (`_SORTABLE_PRESET`,
  `_getSortableContext` / `_setSortableContext`) are excluded from the index and MAY keep
  their names; the copy's `_createContext()` call already yields a fresh, isolated context
  key. Keep the CSS preset **string value** `"svs-sortable"` so existing Sortable styling
  applies to the copy.
- The default component export becomes `ZSortableA11y` automatically from the filename.

**B-2 — Keyboard drag-and-drop (APG modal grab / move / drop).** Required behavior:

- **Focusable items.** Each item `<li>` MUST be keyboard-focusable using **roving
  tabindex**: exactly one item carries `tabindex="0"` and the rest `tabindex="-1"`; the
  `<ul>` (or a per-item handler) MUST manage moving the `0` tabstop as focus moves. (A flat
  `tabindex="0"` on every item is acceptable only if long-list tab noise is judged
  tolerable; roving is preferred.)
- **Focus navigation (not grabbed).** ArrowUp/ArrowDown MUST move focus between items in the
  list without reordering.
- **Pick up.** Space or Enter on a focused item MUST enter *grab* mode for that item.
- **Move (grabbed).** ArrowUp/ArrowDown MUST move the grabbed item one position, reordering
  `items` through the copied controller (reuse `#moveTo` semantics) and keeping the moved
  item focused after the DOM/FLIP update. At the boundary of a list connected in the same
  group (shared controller, `accept` permitting), the move MUST continue into the adjacent
  connected list.
- **Drop.** Space or Enter while grabbed MUST commit the current position and exit grab mode.
- **Cancel.** Escape while grabbed MUST restore the item to its original position and exit
  grab mode. Losing focus while grabbed SHOULD cancel (restore) to avoid a stuck grab.
- **Live region.** A visually-hidden `aria-live="polite"` region MUST announce pick-up,
  each move, drop, and cancel. Exact wording is *suggested*, not contractual — e.g.
  *"Grabbed {item}, position {i} of {n}. Use arrow keys to move, space to drop, escape to
  cancel."* / *"Moved to position {i} of {n}."* / *"Dropped at position {i} of {n}."* /
  *"Cancelled, returned to position {i} of {n}."*
- **ARIA mechanism.** The implementation MUST NOT use the deprecated `aria-grabbed` /
  `aria-dropeffect` attributes; focusable items + the live region are the mechanism. The
  item `<li>` MAY carry `aria-roledescription` (e.g. `"sortable item"`); instructions MAY be
  wired via `aria-describedby`.
- **Coexistence with pointer.** Keyboard grab and pointer drag MUST be mutually exclusive:
  starting one while the other is active cancels or ignores the second; neither path may
  leave the controller in a half-open state.
- **Modes.** Keyboard reorder MUST work for `mode="move"` within a list and across connected
  lists, respecting `sort` and `accept`. Keyboard support for `mode="clone"` / `mode="swap"`
  MAY be included; if not, those modes MUST still behave as today under pointer and MUST NOT
  throw under keyboard (a no-op or a move-style fallback is acceptable).
- **Multi-select (`Ctrl+Space`).** When `multiple` is true, `Ctrl+Space` on a focused
  (not-grabbed) item MUST toggle that item's selection membership in the existing `selected`
  set (add if absent, remove if present), and the live region SHOULD announce the change
  (e.g. *"{item} selected, {k} items selected"* / *"{item} deselected"*). Picking up
  (`Space`/`Enter`) an item that is part of a multi-selection MUST carry the other selected
  items as **followers** through the move and commit — reusing the existing follower path
  (`prepare(..., followers)` semantics and `#commitFollowers`) exactly as pointer multi-select
  does — so the whole selection moves together. Picking up a non-selected item is a plain
  single-item grab. Selection membership MUST persist across a cancel (mirroring pointer,
  which keeps `selected` across drags). `Ctrl+Space` MUST be inert when `multiple` is false.

**Suggested implementation (illustrative, non-binding):**
- Extend the copied `SortableController` with keyboard state and methods, e.g. a private
  `#grab` record `{ member, key, originKey }` plus `grab(member, key)`,
  `stepKeyboard(dir: -1 | 1)` (reusing `#moveTo` and the connected-member ordering),
  `dropKeyboard()`, `cancelKeyboard()`, and a reactive `message = $state("")` the owning
  list renders into the live region. Add these to the `SortableGroupController` interface in
  the copy.
- In the instance script add a `hkeydown: KeyboardEventHandler<HTMLUListElement>` (`const`
  arrow, typed per coding-style §8) on the `<ul>`; resolve the target `<li>` via
  `closest("[data-svs-key]")` as the pointer handlers do. After a grabbed move, re-focus by
  key: `await tick()` then focus `[data-svs-key="…"]`.
- Render one visually-hidden live region per list (only the list owning the grabbed item
  announces), styled inline (`clip`/`sr-only`-style), reading `controller.message`.
- For `Ctrl+Space`, detect `ev.ctrlKey && ev.key === " "` in `hkeydown`, resolve the `<li>` key,
  and toggle the instance-level `selected` `SvelteSet` (the same set pointer select uses).
  At pick-up, compute `followers = [...selected].filter((k) => k !== key)` and pass them into
  the keyboard grab, mirroring `onPointerDown`'s follower computation.

**Notes:** keep the diff against `Sortable.svelte` as small as the keyboard feature allows —
this is a faithful copy plus the keyboard layer, not a rewrite. Update the copy's
`@component` comment: *Behavior* MUST document the keyboard model and the live region; the
*Exports* line MUST reflect `createZSortableA11yGroup`.
**Risk:** medium–high; contained to the new file.

### (D) ADR 003

**Where:** create `.ws/decision/003-keyboard-dnd-in-verification-copy.md` with **exactly**
this body:

```md
# 003. Keyboard drag-and-drop developed in a Z-prefixed verification copy

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-08
**Affects:** public package surface (new exported component `ZSortableA11y` +
`createZSortableA11yGroup`); `SortableGroup` / `Sortable` accessible name; Sortable keyboard
accessibility

## Context

`Sortable` / `SortableGroup` implement drag-and-drop with pointer events only; there is no
WAI-ARIA APG keyboard path (pick up / move / drop). Keyboard DnD is a wanted improvement but
high-risk: the `SortableController` is pointer-centric and its recent history is a run of
subtle reorder bug fixes (FLIP-induced `pointerover`, multi-select followers). Adding a
parallel keyboard state machine directly to the production component risks regressing shipped
pointer behavior during pre-release.

## Decision

Develop the keyboard DnD path in a **copy**, `ZSortableA11y.svelte` (a faithful copy of
`Sortable.svelte` plus the keyboard layer), instead of modifying `Sortable.svelte`. The copy
ships in the package under a `Z` prefix — it is not surfaced in the docs site or CLI
suggestions — so the maintainer can validate it as a real installed component before deciding
whether to fold the keyboard path back into `Sortable`. The low-risk accessible-name fixes are
applied directly to production: `SortableGroup` and `Sortable` both gain an
`aria-label` / `aria-labelledby` `{...rest}` passthrough (the latter with an explicit
`ariaLabel` recommendation prop, matching `Modal` / `Toggle` / `Drawer`). Because the package
index de-dupes exports by name, the copy renames its one factory export to
`createZSortableA11yGroup`; its duplicate type exports de-dupe harmlessly against the identical
originals.

## Consequences

- Production `Sortable` pointer behavior carries zero regression risk from the keyboard work.
- The package temporarily exposes a near-duplicate component; the `Z` prefix marks it as
  provisional and keeps it out of docs/CLI suggestions.
- A future task may merge the validated keyboard path into `Sortable` and remove
  `ZSortableA11y`; that will be its own decision.
- The `Z` prefix is established as the convention for provisional/verification components.

## Rejected alternatives

- **Modify `Sortable.svelte` directly** — least duplication, but risks regressing shipped
  pointer DnD during pre-release with no isolation to validate against.
- **Exclude the copy from the package (`prep.ts` filter for `Z*`)** — would prevent using it
  as a real installed component, which is the whole point of the verification step.
- **Rename all of the copy's exports to a `ZSortableA11y*` namespace** — larger, noisier diff
  against the source of truth; only the factory actually needs renaming to survive index
  de-dup.
```

## Invariants / expected behavior

- The only change to `Sortable.svelte` is the additive accessible-name surface (A-2: interface
  `extends Omit<HTMLAttributes…>`, `ariaLabel`, `{...rest}`, `class` merge); its
  `SortableController`, pointer-handler bodies, template structure, and other props are unchanged.
- `SortableGroup` `role="group"` always present; callers cannot override `role`; caller
  `class` merges after the part class.
- In `ZSortableA11y`, pointer DnD is identical to `Sortable`; keyboard (incl. `Ctrl+Space`
  multi-select) is additive.
- No use of `aria-grabbed` / `aria-dropeffect`.

## Out of scope

- Any change to `Sortable.svelte` beyond the A-2 accessible-name surface (no controller /
  keyboard / drag-behavior change in the production component).
- `Pagination`'s accessible-name alignment (the one `ariaLabel`-without-passthrough exception)
  — a separate future task, not touched here.
- `_core.ts` changes (none needed).
- Docs-site / CLI wiring for `ZSortableA11y`.

## Accepted risks / tradeoffs

- The package exports a `ZSortableA11y` component that duplicates much of `Sortable`. Accepted
  per ADR 003; the `Z` prefix marks it provisional.
- The copy's duplicate **type** exports (`SortableProps`, `SortableGroupController`, …) are
  de-duped out of the generated index in favor of `Sortable`'s identical originals. This is
  acceptable because the shapes are structurally near-identical — the copy is taken after A-2,
  so both share the accessible-name surface (`ariaLabel` + `{...rest}`); the copy only additionally
  `Omit`s `"onkeydown"` and its controller carries extra keyboard methods, neither of which a
  consumer typing against the shared names is harmed by. The one export that materially differs
  (the keyboard-capable factory) is renamed and kept.
