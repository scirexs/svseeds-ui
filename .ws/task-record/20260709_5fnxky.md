## Background / constraints

`ZSortableA11y.svelte` is the Z-prefixed keyboard-capable copy of `Sortable`
(ADR `003`). Its keyboard layer lets a user pick up an item (Space/Enter),
move it (ArrowUp/ArrowDown), and drop (Space/Enter) or cancel (Escape). Lists are
connected by sharing one `createZSortableA11yGroup()` controller; a grabbed item
can cross a list boundary into a connected, `accept`-permitting list.

**What must keep working:**

- The pointer drag path and all existing keyboard behavior (grab / same-list move
  / cross-list move / drop / Escape-cancel / multi-select followers).
- Same-list `sort={false}` gating already fixed in the prior task
  (`stepKeyboard` calls `#canSort` before `#moveTo`).

**Critical hazard — verified, do not skip.** A **cross-list** keyboard move
destroys the grabbed item's `<li>` in the source list and re-creates it in the
destination list; `focusItem` then re-focuses it after a `tick()`. During that
window the source `<li>` fires exactly **one `blur` / `focusout` with
`relatedTarget === null`**. (Confirmed empirically: a grabbed item stepped across
a boundary emits a single `focusout` whose `relatedTarget` is `null` before focus
lands on the item in the new list.) A **same-list** move keeps the same keyed DOM
node and fires no blur. Therefore a naive "cancel on any blur where relatedTarget
is not a sortable item" would fire on that `null` and wrongly cancel every
cross-list keyboard move — which C-3's tests exercise directly. The cancel-on-blur
logic **MUST** be immune to the blur a programmatic keyboard step causes.

## Detailed changes

### (C-1) Cancel keyboard grab on focus loss — `src/lib/_svseeds/ZSortableA11y.svelte`

**Where:** the `hkeydown` ArrowUp/ArrowDown branch (which calls
`controller.stepKeyboard` then `await focusItem(next)`) and the `<li>` template
(add an `onblur` beside the existing `onfocus`).

**Required behavior (normative):**

- While `controller.keyboardActive` is true, if the grabbed item loses focus to a
  target that is **not** a sortable item (not within an element matching
  `[data-svs-key]`) **and** the focus loss is not part of a programmatic keyboard
  step, the handler **MUST** call `controller.cancelKeyboard()` — restoring the
  pre-grab order and announcing the cancellation (the controller already does
  both).
- A keyboard step's transient blur **MUST NOT** cancel the grab. In particular a
  cross-list ArrowUp/ArrowDown move (`relatedTarget === null` mid-move) MUST
  complete normally.
- On a genuine focus-loss cancel the handler **MUST NOT** re-focus any item
  (focus has legitimately left; stealing it back would trap the user). It only
  cancels.
- The pointer drag path and non-grabbed keyboard navigation **MUST** be
  unaffected (`onblur` is a no-op unless `keyboardActive`).

**Suggested implementation (illustrative — a suppression flag is the reliable
way to satisfy the hazard; a relatedTarget-only guard is not):**

- Add a component-local, non-reactive guard, e.g. `let suppressGrabCancel = false;`
  (read only inside the event handler, so it need not be `$state`).
- In the `hkeydown` ArrowUp/ArrowDown branch, set the guard around the grabbed
  step so the transient blur is ignored:

  ```ts
  if (ev.key === "ArrowUp" || ev.key === "ArrowDown") {
    ev.preventDefault();
    const dir = ev.key === "ArrowUp" ? -1 : 1;
    if (controller.keyboardActive) {
      suppressGrabCancel = true;
      try {
        const next = controller.stepKeyboard(dir);
        if (next) {
          focusKey = next;
          await focusItem(next);
        }
      } finally {
        suppressGrabCancel = false;
      }
    } else {
      const next = nextFocus(key, dir);
      if (next) {
        focusKey = next;
        await focusItem(next);
      }
    }
    return;
  }
  ```

- Add an `onblur` to the `<li>` (beside `onfocus={() => (focusKey = k)}`):

  ```svelte
  onblur={(ev) => {
    if (suppressGrabCancel || !controller.keyboardActive) return;
    const related = ev.relatedTarget;
    if (related instanceof Element && related.closest("[data-svs-key]")) return;
    controller.cancelKeyboard();
  }}
  ```

**Notes:**

- The suppression flag is what makes cross-list moves safe: the `null`-relatedTarget
  blur fires synchronously during `focusItem`'s `tick()`, i.e. strictly between
  setting the flag and clearing it in `finally`. The `related.closest(...)` guard
  is secondary defense for focus moving directly onto another sortable item.
- Do **not** move this logic into the controller or change `stepKeyboard` /
  `focusItem` signatures; keep the guard component-local. Grab (Space/Enter) and
  drop/Escape paths need no flag: grab keeps the same node (no blur), and
  drop/cancel clear `keyboardActive` before `focusItem`, so `onblur` is already a
  no-op.

**Risk:** getting the flag lifecycle wrong (e.g. not resetting in a `finally`)
could leave grabs uncancellable, or a missing flag could break cross-list moves.
C-3's cross-list and focus-loss tests cover both directions.

## Accepted risks / tradeoffs

- The suppression flag couples the `onblur` cancel to the `hkeydown` step
  lifecycle. This is deliberate: it is the deterministic way to distinguish a
  programmatic-step blur (`relatedTarget === null`) from genuine focus loss, which
  `relatedTarget` alone cannot do.
