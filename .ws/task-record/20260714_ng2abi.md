## Goal

Let callers supply the month-year picker declaratively — `<Calendar><MonthPicker … /></Calendar>` —
in addition to the existing `monthPicker` config bag. Achieve it by converting the
Calendar↔MonthPicker relationship from a **Composite** (private child wired by props) to the
library's standard **context coordinator** pattern (as Accordion↔Disclosure): MonthPicker becomes
dual-mode (reads an optional context, falls back to its own props), and Calendar sets that context
and renders a caller-supplied `children` snippet in its picking slot. This is also the shape the
planned `<DateInput><Calendar>…</Calendar></DateInput>` nesting will reuse.

## Done when

- MonthPicker exports a `MonthPickerContext` contract and `[_getMonthPickerContext,
  _setMonthPickerContext]`, and consumes it dual-mode (embedded ⇒ context; standalone ⇒ props,
  behaviour unchanged).
- Calendar sets that context and accepts `children?: Snippet`; while `picking`, `children` (when
  present) renders in the middle/picking slot in place of the default internal MonthPicker.
- `<Calendar><MonthPicker/></Calendar>` behaves like the `monthPicker` bag today: the embedded
  MonthPicker reflects `display` (year-month), respects `min`/`max`/`variant`, and selecting a
  year-month updates Calendar's `display`.
- The `015-…` ADR file, the `.ws/architecture/overview.md` wording, and the
  `.ws/policy/compound-components.md` example are updated as specified.
- `bun run check` and the two scoped test files pass; scoped `fmt:check` over the changed
  `.svelte` files passes.

## Background / constraints

- **Standalone MonthPicker must not change.** With no context present, MonthPicker keeps its
  current behaviour exactly (default value = `Temporal.Now`, ±100-year window, clamp, wheels).
- **Default (no-`children`) Calendar must not change.** With no `children`, Calendar still shows an
  internal MonthPicker while picking, configured by the `monthPicker` bag, wired to `display` /
  `min` / `max` / `variant` — but now *through the context*, not explicit props (the bag's
  `Omit` already excludes `value`/`min`/`max`/`variant`, so no bag key is lost).
- **`children` wins over the bag.** When both `children` and `monthPicker` are supplied, `children`
  renders and the bag is ignored (mirrors Accordion's `children` vs `items`). No merge.
- `_core.ts` needs **no** change: `_createContext` and `SVSContext` already exist there.
- The `@component` Types block lives inside an HTML comment that Prettier does **not** check
  (`.ws/knowledge/svelte-component-comment-block-skips-prettier.md`) — sync it to the interface by
  hand; a green `fmt:check` does not prove it.

## Detailed changes

### (C-1) MonthPicker — dual-mode context child

**Where:** `src/lib/_svseeds/MonthPicker.svelte`, `<script module>` and instance `<script>`.

**Required behavior:**

1. In `<script module>` MUST export a context interface and accessor pair, declared after
   `MonthPickerBindProps`, before `_MONTHPICKER_PRESET`:

   ```ts
   export interface MonthPickerContext extends SVSContext {
     get value(): Temporal.PlainYearMonth | undefined;
     set value(v: Temporal.PlainYearMonth | undefined);
     get min(): Temporal.PlainYearMonth | undefined;
     get max(): Temporal.PlainYearMonth | undefined;
   }
   export const [_getMonthPickerContext, _setMonthPickerContext] = _createContext<MonthPickerContext>();
   ```

   `variant`/`styling` come from `SVSContext`; do not redeclare them.
2. Imports MUST follow `coding-style.md` §1: add `_createContext` to the value import from
   `./_core`, and `SVSContext` to the type import from `./_core`.
3. The instance script MUST read the context once at init (`const ctx = _getMonthPickerContext();`)
   and derive effective inputs: value, `min`, `max`, `variant` come **from `ctx` when present, else
   from the own props**. `styling` resolves as `styling ?? ctx?.styling` in the `cls` derivation.
4. The standalone default (`value ??= Temporal.Now…`) MUST run **only when `ctx` is absent**; when
   embedded, MonthPicker MUST NOT mutate its own `value` prop — it writes through `ctx.value`.
5. All value **writes** (`sync`, `setYear`, `setMonth`) MUST route through a single setter that
   targets `ctx.value` when embedded, else the local `value` prop.
6. The `min`/`max` window defaults (`lo`/`hi`) and clamp MUST use the effective `min`/`max` (context
   when embedded), and the ±100-year fallback MUST derive from the effective current value.
7. The template MUST use the effective `variant` for both the part classes and the two
   `<WheelPicker>`s.

**Suggested implementation** (illustrative — match Accordion/Disclosure's dual-mode style,
`compound-components.md` §4):

```ts
const ctx = _getMonthPickerContext();
// svelte-ignore state_referenced_locally
if (!ctx && value === undefined) value = Temporal.Now.plainDateISO().toPlainYearMonth();
const initial = (ctx?.value ?? value) as Temporal.PlainYearMonth;

const effVariant = $derived(ctx ? ctx.variant : variant);
const effMin = $derived(ctx ? ctx.min : min);
const effMax = $derived(ctx ? ctx.max : max);
const cls = $derived(_fnClass(_MONTHPICKER_PRESET, styling ?? ctx?.styling));
// current(): (ctx ? ctx.value : value) ?? initial
// setValue(v): if (ctx) ctx.value = v; else value = v;   ← used by sync/setYear/setMonth
// lo/hi: effMin ?? {year: initial.year-100, month:1}; effMax ?? {year: initial.year+100, month:12}
```

**Notes:** `current()` is already the single read helper; make writes call the new `setValue`
helper instead of assigning `value` directly. Keep the `state_referenced_locally` ignore.

**Risk:** the `initial` capture and the default-value guard are the only non-mechanical parts;
getting the `!ctx` guard wrong would either mutate the parent's value or skip the standalone
default. Cover both with tests (C-7).

### (C-2) Calendar — coordinator + `children`

**Where:** `src/lib/_svseeds/Calendar.svelte`, `<script module>` interface, imports, instance
script, and the picking branch of the template.

**Required behavior:**

1. `CalendarProps` MUST gain `children?: Snippet;`. Place it among the snippet props (e.g. just
   before `monthPicker`). `Snippet` is already imported. Keep the `@component` Types block in sync (C-3).
2. Imports MUST add `_setMonthPickerContext` to the value import from `./MonthPicker.svelte` and
   `MonthPickerContext` to its type import.
3. Calendar MUST build a `MonthPickerContext` object and call `_setMonthPickerContext(ctx)` at init.
   The context MUST expose:
   - `value` — get returns the current display year-month (`currentDisplay()`); set assigns
     `display` (guard against `undefined`).
   - `min` / `max` — `min?.toPlainYearMonth()` / `max?.toPlainYearMonth()` (the same conversion the
     template does today).
   - `variant` — the Calendar `variant`.
   - `styling` — MUST return `undefined` (Calendar does not push its own styling onto the picker;
     preserves today's behaviour where the picker's styling comes only from the `monthPicker` bag /
     the caller-written child).
   Any `$derived` values the getters read MUST be declared **before** the context object literal
   (`compound-components.md` §5).
4. In the `{#if picking}` branch, the middle slot MUST render `children` when present, otherwise the
   default internal `<MonthPicker {...monthPicker} />`. The internal MonthPicker MUST NOT keep the
   explicit `bind:value` / `min` / `max` / `variant` attributes — those now flow through the
   context. The `{...monthPicker}` spread stays (bag config for the default child).
5. When `children` is present, the `monthPicker` bag MUST have no effect (children path does not
   render the default child).

**Suggested implementation:**

```svelte
<!-- Initialize (after cls/uid, alongside other derived): -->
const ctxMin = $derived(min?.toPlainYearMonth());
const ctxMax = $derived(max?.toPlainYearMonth());
const mpCtx: MonthPickerContext = {
  get value() { return currentDisplay(); },
  set value(v) { if (v) display = v; },
  get min() { return ctxMin; },
  get max() { return ctxMax; },
  get variant() { return variant; },
  get styling() { return undefined; },
};
_setMonthPickerContext(mpCtx);
```

```svelte
{#if picking}
  <div class={cls(PARTS.MIDDLE, variant)} transition:tfn|local={tparams}>
    {#if children}{@render children()}{:else}<MonthPicker {...monthPicker} />{/if}
  </div>
{:else}
  … (grid branch unchanged) …
{/if}
```

**Notes:** `currentDisplay()` is a hoisted function, safe to call from a getter; using it keeps the
getter always in sync with `display`. The setter's `if (v)` guard matches the non-optional
year-month writes MonthPicker actually makes.

**Risk:** low. Because Calendar sets the context unconditionally, the default internal MonthPicker
is *also* embedded and self-wires — so the explicit attributes MUST be removed (leaving them would
be dead, ignored props). Verify the default (no-`children`) calendar still tracks `display`.

### (C-6) ADR — create `.ws/decision/015-calendar-monthpicker-context-coordinator.md`

This file does not exist yet; create it with **exactly** this body:

```md
# 015. Calendar↔MonthPicker as a context coordinator, not a Composite

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-14
**Affects:** Calendar / MonthPicker public API (Calendar `children`; MonthPicker context contract); the date-picker composition family (future DateInput/DateField → Calendar).

## Context

Calendar embedded MonthPicker as a private Composite child, configured only through Calendar's
`monthPicker` bag. Callers could not supply their own MonthPicker declaratively, and the fixed
Composite shape would also block the intended `<DateInput><Calendar><MonthPicker/></Calendar></DateInput>`
nesting. The library already has a context-coordinator convention (`compound-components.md`) used by
Accordion↔Disclosure and each Field↔Input pair.

## Decision

Convert the Calendar↔MonthPicker relationship from Composite to context coordinator. MonthPicker
becomes dual-mode: it reads an optional `MonthPickerContext` (value / min / max / variant / styling)
and falls back to its own props when standalone. Calendar sets that context and gains a generic
`children?: Snippet` rendered in the picking slot; when `children` is present it replaces the
default internal MonthPicker (children wins over the `monthPicker` bag). The default (no-children)
path still renders an internal MonthPicker, now self-wired through the same context. This mirrors
Accordion's items/children duality and generalizes to the future DateInput → Calendar coordination.

## Consequences

- Callers can write `<Calendar><MonthPicker … /></Calendar>` and fully customize the picker; the
  `monthPicker` bag remains for the default path.
- MonthPicker gains a public context surface (`_getMonthPickerContext` / `_setMonthPickerContext`,
  `MonthPickerContext`) that must stay stable.
- The same pattern extends to `<DateInput><Calendar>…` nesting with no special mechanism.
- Calendar still imports MonthPicker for the default child, so the hard packaging dependency is
  unchanged.
- Standalone MonthPicker behaviour is unchanged (context absent ⇒ props path).

## Rejected alternatives

- Keep the Composite and expose more through the `monthPicker` bag: cannot express a caller-written
  child and does not generalize to the DateInput nesting.
- Bespoke prop/clone injection into the child: non-idiomatic in Svelte 5 and inconsistent with the
  existing context-coordinator convention.
- Make Calendar a full dual-mode child now (DateInput as coordinator): out of scope; deferred to a
  later task to keep this change minimal.
```

## Invariants / expected behavior

- Standalone `<MonthPicker/>`: unchanged (default value, window, clamp, wheels, a11y).
- `<Calendar/>` with no `children`: unchanged UX; label toggles the internal MonthPicker; picking
  writes `display`.
- `<Calendar><MonthPicker/></Calendar>`: the embedded MonthPicker shows `display`'s year-month,
  bounded by Calendar's `min`/`max`, in Calendar's `variant`; changing a wheel updates `display`.
- `children` present ⇒ `monthPicker` bag ignored.

## Out of scope

- Making Calendar a dual-mode **child** (a `CalendarContext`) or turning DateInput/DateField into a
  Calendar coordinator — a separate future task. Do not add a Calendar context here.
- Any change to `_core.ts`, WheelPicker, DateInput, DateField, or the calendar grid/keyboard logic.
- Changing MonthPicker's presentational props (`order`, `locale`, `monthLabel`, `year`/`month`
  bags) — they stay caller-controlled on the child.

## Accepted risks / tradeoffs

- MonthPicker now carries a public context contract (new stable surface). Accepted: it is the
  library's standard coordination mechanism and the basis for the planned DateInput nesting.
- Calendar's context intentionally returns `undefined` for `styling`, preserving today's behaviour
  (the picker's styling is owned by the child/bag, not inherited from Calendar).
