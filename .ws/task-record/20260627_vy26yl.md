## Goal

Add a headless, accessible two-pane resizable Splitter: two content panes
(`left`/`right`) with a draggable separator between them that resizes the first
pane. Visual layout (orientation, container/pane sizing) stays the consumer's
CSS; the component owns only structure, the resize behaviour, and ARIA. Nesting
a Splitter inside a pane gives N-way splits. Implements the W3C **Window
Splitter** APG pattern.

## Done when

- `<Splitter orientation left right />` renders `whole → left / separator / right`
  with stable part classes, a focusable `role="separator"`, and the consumer can
  size the first pane from the emitted custom property.
- Pointer drag on the separator resizes the first pane within `[min, max]`,
  honoring `snap` points; `value` is two-way bindable and reflects the size.
- Keyboard (arrow keys per orientation, `Home`/`End`) moves the separator; ARIA
  `valuenow`/`valuemin`/`valuemax`/`orientation` and `aria-controls` are correct.
- `bun run check` passes; new unit + sim tests pass; `.ws/architecture/overview.md`
  lists Splitter.

## Background / constraints

- **Stack & shape:** Svelte 5 runes, headless. Follow `coding-style.md` §14 file
  skeleton and the `@component` doc (§11). The component owns **structure +
  behaviour + ARIA only**; all appearance is the consumer's CSS via `styling` /
  part classes. `_core.ts` MUST NOT change.
- **Orientation drives behaviour, not just CSS.** Because the drag axis, the
  arrow-key mapping, and `aria-orientation` all depend on the split direction,
  orientation is an explicit prop (it cannot be inferred from the consumer's
  `flex-direction`). The consumer sets `orientation` to match their layout CSS.
- **Size model.** The first pane's size is a bindable percentage `value`
  (`0`–`100`) owned by the component. The component does **not** lay the panes
  out; it emits the size as a CSS custom property on the root, and the consumer
  wires it (`flex-basis: var(--svs-splitter)` for flex, or
  `grid-template-columns: var(--svs-splitter) auto 1fr` for grid). This keeps the
  component layout-agnostic (works with flex or grid) per the draft's "sizing via
  CSS" intent.
- **Must keep working:** existing components and tests are untouched; no `_core`
  edit; no new runtime dependency.

## Detailed changes

### `(C-1)` Module exports, props, `@component`

**Where:** `Splitter.svelte` `<script module>`.

**Required behavior:**
- The exported interface MUST be `SplitterProps extends Omit<HTMLAttributes<HTMLDivElement>, "children" | "role" | "style">`
  (root is a `<div>`; `style` is component-owned because the size custom property
  is emitted there; keep `class` out of the `Omit` so it merges). Required props
  first (§12), with these members:
  - `left: Snippet<[string]>` — first pane content; `Snippet<[variant]>`. **Required.**
  - `right: Snippet<[string]>` — second pane content; `Snippet<[variant]>`. **Required.**
  - `children?: Snippet<[number, string]>` — optional separator handle content; `Snippet<[value, variant]>`.
  - `orientation?: "horizontal" | "vertical"` — default `"horizontal"`; the **layout axis**
    (same convention as `Tabs`). `"horizontal"` = panes arranged along the inline axis
    (left | right), drag/arrows move horizontally; `"vertical"` = panes stacked
    (top | bottom), drag/arrows move vertically. `aria-orientation` is the perpendicular
    (a left|right split has a vertical separator bar) — see `(C-4)`.
  - `value?: number` — **bindable**; first-pane size as a percentage `0`–`100`, default `50`.
  - `min?: number` — default `0`; lower clamp for `value` (percentage).
  - `max?: number` — default `100`; upper clamp for `value` (percentage).
  - `step?: number` — default `1`; percentage change per arrow-key press.
  - `snap?: number[]` — optional snap target percentages used during pointer drag.
  - `label?: string` — accessible name applied as `aria-label` on the separator.
  - `cssvar?: Partial<Record<SplitterCssVar, string>>` — rename the emitted size
    custom property; `SplitterCssVar = "position"`.
  - `attach?: Attachment<HTMLDivElement>`.
  - `element?: HTMLDivElement` — **bindable** (the root element).
  - `styling?: SVSClass`.
  - `variant?: SVSVariant` — default `VARIANT.NEUTRAL`.
- MUST also export:
  - `export type SplitterReqdProps = "left" | "right";`
  - `export type SplitterBindProps = "value" | "element";`
  - `export type SplitterCssVar = "position";`
  - `export const _SPLITTER_PRESET = "svs-splitter";`
- Imports MUST follow §1 (value block then type block, ordered by source). Value:
  `{ VARIANT, PARTS, _fnClass, _cssVar } from "./_core"`. Types: `Snippet` (svelte),
  `Attachment` (svelte/attachments), `HTMLAttributes` (svelte/elements),
  `{ SVSClass, SVSVariant } from "./_core"`.
- The `@component` doc MUST follow §11 (`Usage` → `Types` → `Anatomy` → `Behavior`).
  *Types* mirrors `SplitterProps` + `SplitterCssVar` with inline default comments;
  do not list `ReqdProps`/`BindProps`. *Behavior* MUST document: orientation
  drives drag axis/keys; the emitted `--svs-splitter` (renameable via `cssvar.position`,
  default namespaced `--svs-*`); `data-orientation` / `data-dragging` attributes;
  the keyboard model; and the perpendicular `aria-orientation` (a left|right split has
  a vertical separator bar). *Usage* MUST note that the consumer supplies the layout CSS
  matching `orientation` (e.g. `display:flex` consuming `--svs-splitter`); the component
  ships no layout CSS.

**Suggested implementation:** model the file on `Slider.svelte`'s `<script module>`.

**Risk:** prop surface drift between the interface and the `@component` *Types* block — keep them in sync (§14).

### `(C-2)` Structure, parts, emitted size property

**Where:** `Splitter.svelte` template + instance `<script>`.

**Required behavior:**
- Anatomy (parts on each element):
  ```svelte
  <div class="whole" {...rest} style data-orientation data-dragging>
    <div class="left" id={paneId}>{@render left(variant)}</div>
    <div class="main" role="separator" tabindex="0"
         aria-controls aria-orientation aria-valuenow aria-valuemin aria-valuemax aria-label>
      {@render children?.(value, variant)}
    </div>
    <div class="right">{@render right(variant)}</div>
  </div>
  ```
  Use `PARTS.WHOLE` (root), `PARTS.LEFT` (first pane), `PARTS.MAIN` (separator),
  `PARTS.RIGHT` (second pane). `const cls = $derived(_fnClass(_SPLITTER_PRESET, styling))`,
  applied as `cls(part, variant)`; the caller's `class` (`c`) MUST merge onto the
  **root** (`class={[cls(PARTS.WHOLE, variant), c]}`).
- `{...rest}` MUST be spread on the root **before** component-owned identity attrs
  (`style`, `data-*`) so callers cannot override them (§14).
- The first pane MUST carry a stable id (`$props.id()`-derived) and the separator's
  `aria-controls` MUST reference it.
- The root MUST emit the size custom property whose **name** is resolved with
  `_cssVar(cssvar, "position", "--svs-splitter")` and whose value is `` `${value}%` ``
  (e.g. `style="--svs-splitter: 50%"`). Renaming via `cssvar.position` changes the
  emitted property name so consumer CSS reading the renamed token is honored.
- The root MUST expose `data-orientation={orientation}` and, while a pointer drag
  is active, `data-dragging` (present/absent) so consumer CSS can set cursor /
  disable text selection.

**Suggested implementation:** resolve the name once as a `$derived`; build the
`style` string inline (or via `_cssVarStyle([{ name, value: `${value}%` }])`).

**Notes:** the component never sets `flex-basis`/`grid` itself — it only publishes
the custom property; the consumer's CSS consumes it. Document this in *Usage*.

### `(C-3)` Pointer-drag resize, clamp, snap

**Where:** `Splitter.svelte` instance `<script>` (`Event Handlers` banner).

**Required behavior:**
- `min`/`max` MUST be normalized so the lower bound is the smaller of the two
  (mirror Slider's `rmin`/`rmax`); `value` MUST be clamped into `[min, max]` on
  init and reactively when `value`/`min`/`max` change (use `$effect.pre` like
  Slider; extract a named `clamp`/`sync` function if the body exceeds one line, §5).
- Dragging MUST use Pointer Events on the separator with pointer capture
  (`setPointerCapture`) so mouse/touch/pen and out-of-element drags all track.
  On `pointerdown`: capture, measure the root's content box, mark dragging.
  On `pointermove`: convert the pointer position along the active axis
  (`orientation === "vertical"` → vertical/`clientY`/height, else
  horizontal/`clientX`/width) into a percentage of the root's content size, then
  set `value = clampedAndSnapped`. On `pointerup`/`pointercancel`/lost capture:
  release and clear dragging.
- `snap`: when `snap` has entries, after clamping, `value` MUST snap to the
  nearest snap point that lies within a small threshold; otherwise the raw clamped
  value is used. Snapping applies to **pointer drag only**, not keyboard. Define
  the threshold as a module constant (suggested ≈ `3`, percentage points) and
  document it.
- Drag handlers MUST be `hXxx` `const` arrows typed from `svelte/elements` (§8).

**Suggested implementation:** keep a small drag-state holder; helper
`pctFromEvent(ev)` reads the cached rect; `applySnap(v)` returns the snapped value.
Consider a class only if the logic grows heavy (§10) — a couple of handlers + two
helpers does not warrant one.

**Risk:** `getBoundingClientRect()` returns zero-sized boxes under jsdom — pointer
math is verified in the **sim** test, not the unit test (see `(C-5)`).

### `(C-4)` Keyboard + ARIA

**Where:** `Splitter.svelte` separator element + a `keydown` handler.

**Required behavior:**
- The separator MUST be `role="separator"`, `tabindex="0"`, with
  `aria-controls={paneId}`, `aria-valuenow={Math.round(value)}`,
  `aria-valuemin={min}`, `aria-valuemax={max}`, and `aria-label={label}` (omitted
  when `label` is unset).
- `aria-orientation` MUST be the **perpendicular** of the layout orientation
  (a separator between left|right panes is a *vertical* bar): `orientation ===
  "vertical"` → `aria-orientation="horizontal"`, else `"vertical"`. This mirrors
  `MenuSeparator.svelte`'s rule — state the reason in *Behavior*.
- Keyboard on the separator (`keydown`), per orientation:
  - horizontal: `ArrowLeft` decreases, `ArrowRight` increases `value` by `step`.
  - vertical: `ArrowUp` decreases, `ArrowDown` increases `value` by `step`.
  - `Home` → `min`; `End` → `max`. Each handled key MUST `preventDefault()`.
  - Results MUST be clamped to `[min, max]` (snap is not applied to keyboard).

**Suggested implementation:** a single `hkeydown: KeyboardEventHandler<HTMLDivElement>`
switching on `ev.key` and `orientation`.

## Invariants / expected behavior

- `value` always stays within the normalized `[min, max]`; the emitted custom
  property always reflects the current `value` as `` `${value}%` ``.
- With no `orientation` set, behaviour is `horizontal` (side-by-side left | right panes).
- Standalone-only component: no context, no `_core` change, no dependency on other
  components.

## Out of scope

- Collapse/restore, `onResize` callback, per-pane pixel minimums, more than two
  panes in a single instance (use nesting), and any default/opinionated CSS.
- Any change to `_core.ts`, other components, the generated `index.ts`/`dep.json`
  (these regenerate from the new file at build time), or unrelated refactors.

## Accepted risks / tradeoffs

- Orientation is an explicit prop rather than inferred from CSS — accepted because
  the drag axis, key mapping, and `aria-orientation` cannot be derived from
  `flex-direction`. The consumer sets `orientation` to match their layout; the value
  names follow `Tabs`' layout-axis convention (`horizontal` = side-by-side).
- `orientation` defaults to `"horizontal"` (side-by-side left|right) — the common
  case, and matching the minimal functional flex setup (`display:flex` defaults to
  `flex-direction:row`). With *no* layout CSS at all the bare DOM stacks vertically,
  so the default drag axis would mismatch the visual — accepted because the splitter
  is non-functional until the consumer wires `--svs-splitter`, and the component must
  not ship layout CSS (headless). Documented in *Usage*.
- The consumer must wire one line of CSS (consume `--svs-splitter`) for the panes
  to resize — accepted to keep the component layout-engine-agnostic (flex or grid).
