# Task: Add an accessible-name API to the `ToggleGroup` group element

Target files:
- `src/lib/_svseeds/ToggleGroup.svelte`
- `src/lib/_svseeds/ToggleGroupField.svelte`
- `tests/fixtures/ToggleGroupCtxProvider.svelte`
- `tests/ToggleGroup.svelte.test.ts`
- `tests/ToggleGroupField.svelte.test.ts`

References (read before editing):
- `.ws/policy/coding-style.md` §14 (rest-props: keep `class` out of `Omit`, destructure `class: c, ...rest`, spread `{...rest}` **before** component-owned identity attrs) and §15 (context wiring).
- `.ws/policy/compound-components.md` §2 (context coordinator: wiring flows through context; the child stays usable standalone).
- `src/lib/_svseeds/ContextMenu.svelte` — sibling that spreads `...rest` onto its root element and merges `class`.
- `src/lib/_svseeds/TagsInput.svelte` — sibling that destructures specific aria attrs (`"aria-describedby"`, `"aria-invalid"`) out of `rest` as named locals and re-emits a merged value; follow this exact shape for `aria-labelledby`.

## Goal

`ToggleGroup` renders its group element (`<span role="group|radiogroup">`) with no way to give it an accessible name: it accepts no rest-props and exposes no `aria-label` / `aria-labelledby`. Standalone callers therefore cannot name the group, and when embedded in `ToggleGroupField` the field's label is attached only to the outer field wrapper — it never reaches the inner group element.

Add an accessible-name API in two parts:
- **(C-1)** Standalone: let callers pass `aria-label` / `aria-labelledby` (and any other span attribute) through to the group element via rest-props, following the project's established rest-props convention.
- **(C-2)** Embedded: propagate the field's label id into the inner group element through the shared context, so the actual `radiogroup` / `group` is named per WAI-ARIA APG.

## Done when

- A standalone `ToggleGroup` given `aria-label` (or `aria-labelledby`, `id`, `data-*`, …) renders that attribute on the group `<span>`; the caller's `class` is merged onto the group element alongside the component's own classes.
- Component-owned semantics on the group (`role`, `aria-describedby`, `aria-invalid`, `aria-errormessage`) are unchanged and cannot be overridden by rest-props.
- Inside `ToggleGroupField` with a non-empty `label`, the inner group element carries `aria-labelledby` pointing at the field's label id; with no label it carries no `aria-labelledby` (no regression).
- `bun run check` is clean and the toggle test files pass, including new cases for the above.

## Background / constraints

- `ToggleGroup` is a context-coordinated child (`.ws/policy/compound-components.md`): it works standalone and, when wrapped by `ToggleGroupField`, reads a `ToggleGroupContext` and prefers context values over its own props (see the existing `eff*` deriveds).
- `ToggleGroupField` already computes `idLabel` (`label?.trim() ? \`${uid}-label\` : undefined`) and applies it to the outer field wrapper `<div role="group" aria-labelledby={idLabel}>`. **Leave the outer wrapper exactly as-is** — it stays a labelled `role="group"`; C-2 only adds a name to the inner element as well. The intentional (harmless) redundancy of both the outer wrapper and the inner group referencing the same label is an accepted tradeoff.
- Keep working: existing single-/multi-select behavior, roving tabindex + arrow/Home/End navigation, `aria-checked`, the `aria-describedby` / `aria-invalid` / `aria-errormessage` wiring (both standalone via `ariaDescId` / `ariaErrMsgId` and embedded via context), per-option button attributes from `ToggleOption`, and the `attach` forwarding to buttons.

## Scope of changes

- [ ] **(C-2)** `ToggleGroup.svelte` + `ToggleGroupField.svelte`: extend `ToggleGroupContext` with `ariaLabelId`; emit the inner group's `aria-labelledby` from context (embedded) or from the caller's `aria-labelledby` (standalone); have `ToggleGroupField` expose `idLabel` as `ariaLabelId`.
- [ ] **(C-3)** `tests/fixtures/ToggleGroupCtxProvider.svelte`: implement the new `ariaLabelId` context member so the fixture still satisfies `ToggleGroupContext`.

## Detailed changes

### (C-1) Rest-props → group element

**Where:** `ToggleGroup.svelte`, `ToggleGroupProps` (both the `<script module>` `export interface` and the `@component` Types mirror), the instance `$props()` destructure, and the group `<span>`.

**Required behavior:**
- `ToggleGroupProps` MUST extend `Omit<HTMLAttributes<HTMLSpanElement>, "children" | "role" | "aria-describedby" | "aria-invalid" | "aria-errormessage">`. `class`, `aria-label`, `aria-labelledby`, `style`, `id`, and `data-*` MUST remain available to callers (do not Omit them). The existing named props (`ariaDescId`, `ariaErrMsgId`, etc.) MUST be kept as-is — the owned aria attrs are Omitted from the base specifically so those named props remain the single source and are not duplicated.
- The instance script MUST destructure `class: c` and `...rest` (and `"aria-labelledby"` per C-2). `{...rest}` MUST be spread onto the group `<span>` **before** the component-owned identity attrs (`role`, `aria-describedby`, `aria-invalid`, `aria-errormessage`, `aria-labelledby`) so callers cannot override the component's semantics.
- The group `<span>` `class` MUST become `class={[cls(PARTS.WHOLE, effVariant), c]}` (caller `c` merged last).
- Rest-props MUST target the **group `<span>` only**, not the option `<button>`s (per-button attributes continue to come from `ToggleOption`).

**Suggested implementation:**
```svelte
<!-- $props() destructure (prettier-ignore line): add class: c, "aria-labelledby": ariaLabelledbyProp, ...rest -->
<span
  class={[cls(PARTS.WHOLE, effVariant), c]}
  {...rest}
  role={roleGroup}
  aria-labelledby={effAriaLabelId}
  aria-describedby={effAriaDescId}
  aria-invalid={invalid}
  aria-errormessage={effAriaErrMsgId}
>
```

**Notes:** Add `import type { HTMLAttributes } from "svelte/elements";` (the module already imports `HTMLButtonAttributes` from there — extend that import).

**Risk:** Low. `class` handling changes from a bare `cls(...)` to a merged array; verify existing class-based tests/`styling` still pass.

### (C-2) Propagate the field label to the inner group

**Where:** `ToggleGroup.svelte` (`ToggleGroupContext` interface + a new `eff` derived + the `<span>` attr) and `ToggleGroupField.svelte` (the `ctx` object).

**Required behavior:**
- `ToggleGroupContext` MUST gain `get ariaLabelId(): string | undefined;` (place it alongside the existing `get ariaDescId()` / `get ariaErrMsgId()` members).
- `ToggleGroup` MUST emit the group's `aria-labelledby` as: context value when embedded, else the caller's own `aria-labelledby` when standalone. Follow the `TagsInput` pattern — destructure `"aria-labelledby"` out of `rest` into a named local and re-emit a merged value, so that in standalone mode the caller's value survives (it must NOT be clobbered by an explicit `undefined` placed after the spread).
- `ToggleGroupField` MUST expose its `idLabel` through the context as `ariaLabelId`.
- When the field has no label (`idLabel` is `undefined`), the inner group MUST NOT render an `aria-labelledby` attribute (unchanged from today).

**Suggested implementation:**
```ts
// ToggleGroup.svelte — new derived, mirroring effAriaDescId
const effAriaLabelId = $derived(ctx ? ctx.ariaLabelId : ariaLabelledbyProp);
```
```ts
// ToggleGroupField.svelte — add to the ctx object, next to ariaDescId
get ariaLabelId() {
  return idLabel;
},
```

**Notes:** In embedded mode the context value wins, so an `aria-labelledby` a caller passes via `ToggleGroupField`'s `toggleGroup` bag is intentionally overridden by the field's label (consistent with how the field already overrides `variant` / `ariaDescId` / etc.). No change to the `toggleGroup` `Omit` bag is required.

**Risk:** Low. Additive context member; the only touch-point in `ToggleGroupField` is the `ctx` object.

## Invariants / expected behavior

- Standalone `aria-labelledby` passed by the caller reaches the group element; embedded, the field's label id reaches it instead.
- All owned group semantics (`role`, `aria-describedby`, `aria-invalid`, `aria-errormessage`) are unaffected and remain non-overridable by rest-props.
- No change to option-button rendering, selection, keyboard navigation, or event/validation flow.

## Out of scope

- The outer `ToggleGroupField` wrapper (`<div role="group" aria-labelledby>`) — do not change its role or labelling.
- Any change to sibling components or to the shared field anatomy.
- Adding a dedicated `aria-label` named prop (it flows through `...rest` already).
- Restricting the `toggleGroup` `Omit` bag.

## Accepted risks / tradeoffs

- In embedded mode both the outer field wrapper and the inner group reference the same label id (redundant but valid nesting). Accepted: it keeps the field anatomy consistent with sibling fields while making the inner group APG-conformant.
