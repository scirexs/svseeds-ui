**Target file:** `src/lib/_svseeds/ContextMenu.svelte`

- `.ws/policy/coding-style.md` — §4 branching, §6 naming, §8 handlers vs control functions, §11 `@component` comment, §16 tests.

## Goal

iOS (iPad/iPhone) never fires the `contextmenu` event, so `ContextMenu` cannot be opened on those devices at all. Add a 700 ms touch long-press path that opens the menu, while keeping the existing `contextmenu` path intact for mouse and for touch devices that *do* fire `contextmenu` (Android Chrome fires it at ~500 ms mid-press).

## Done when

- A 700 ms touch long-press opens the menu, positioned at the touch point, on devices that never fire `contextmenu`.
- On a touch device that does fire `contextmenu`, the menu opens exactly once, via the native `contextmenu` — the pending long-press timer does not also fire.
- The `click` that ends a touch gesture which opened the menu does not immediately close it, and a later outside tap still closes it.
- Mouse right-click behavior, `lock`, Escape, outside-click dismissal, and focus restore are unchanged.
- `bun run check`, the scoped test run, and the scoped format check all pass (see Validation).

## Background / constraints

`ContextMenu` currently registers `on(target ?? document, "contextmenu", show)` in an `$effect.pre`, and dismisses via `<svelte:document onclick={hide} onkeydown={hkeydown} />`.

Two non-obvious hazards this task must handle:

1. **The trailing click.** The component closes on a document `click`. A touch long-press ends with `pointerup` → `click`, so a menu opened by long-press would be closed instantly by its own gesture's click. Exactly one such click must be swallowed. The suppression flag must not persist: if it stuck, the *next* legitimate outside tap would be swallowed and the menu could never be dismissed.
2. **Double-open on Android.** Android Chrome fires `contextmenu` at ~500 ms while the finger is still down. Without cancelling the pending 700 ms timer, the menu would open on `contextmenu` and then re-open/re-position 200 ms later.

**What must keep working:** every currently passing test in `tests/ContextMenu.svelte.test.ts` and `tests/ContextMenu.svelte.sim.test.ts` — in particular `lock` gating both open and close, `show`'s `preventDefault`/`stopPropagation` (the latter is what stops the document-level `hide` from firing when `target` is set), edge-aware positioning, and focus restore to `prevFocus`.

## Scope of changes

- [ ] **(C-1)** Add touch long-press detection via pointer events; extract the positioning/open body of `show` into a coordinate-taking `showAt`.
- [ ] **(C-2)** Make the `contextmenu` path cancel a pending long-press timer and mark the open as touch-originated.
- [ ] **(C-3)** Swallow the single trailing `click` of a touch gesture that opened the menu, resetting the flag on the next `pointerdown`.
- [ ] **(C-4)** Create the ADR `.ws/decision/017-contextmenu-touch-long-press.md` with the exact body given below.
- [ ] **(C-5)** Update the `@component` comment's *Behavior* section.
- [ ] **(C-6)** Add browser tests for the long-press paths.

## Detailed changes

### (C-1) Touch long-press detection

**Where:** `ContextMenu.svelte` — new module-scope constants; new state next to `position` / `prevFocus`; new listeners in the existing `$effect.pre`; `show` split in the Event Handlers section.

**Required behavior:**

- A `pointerdown` with `pointerType === "touch"` and `isPrimary === true` MUST start a 700 ms timer; when it elapses, the menu MUST open positioned at the `pointerdown` coordinates.
- The timer MUST be cancelled by `pointerup`, `pointercancel`, and by a `pointermove` whose distance from the press origin exceeds 10 px. A `pointermove` within 10 px MUST NOT cancel it (finger jitter must not defeat the gesture).
- A `pointerdown` whose `pointerType` is not `"touch"`, or which is not primary, MUST NOT start a timer.
- `lock` MUST be re-checked when the timer fires (not only at `pointerdown`), so a menu locked mid-press does not open.
- The long-press open MUST reuse the existing edge-aware positioning and `prevFocus` capture — do not duplicate that arithmetic.
- The `$effect.pre` cleanup MUST clear any pending timer in addition to removing listeners.
- The 700 ms threshold MUST be a module-scope constant, not a prop. No public prop is added by this task; `ContextMenuProps` is unchanged.

**Suggested implementation** (illustrative):

```ts
// module scope, near _CONTEXT_MENU_PRESET
const LONG_PRESS = 700;
const MOVE_CANCEL = 10;
```

```ts
// instance script, next to `position` / `prevFocus`
let timer: ReturnType<typeof setTimeout> | undefined;
let origin = { x: 0, y: 0 };
let touched = false;

$effect.pre(() => {
  const cleanups: (() => void)[] = [];
  cleanups.push(on(target ?? document, "contextmenu", show));
  cleanups.push(on(target ?? document, "pointerdown", press));
  cleanups.push(on(document, "pointermove", move));
  cleanups.push(on(document, "pointerup", cancel));
  cleanups.push(on(document, "pointercancel", cancel));
  if (target) {
    cleanups.push(on(document, "contextmenu", hide));
    cleanups.push(on(document, "pointerdown", reset));
  }
  return () => {
    cancel();
    cleanups.forEach((x) => x());
  };
});
```

```ts
// Event Handlers — `show` keeps its guard/preventDefault, delegates the rest
function show(ev: Event) {
  if (lock) return;
  ev.preventDefault();
  ev.stopPropagation();
  if (timer !== undefined) touched = true; // (C-2)
  cancel();
  showAt((ev as MouseEvent).clientX, (ev as MouseEvent).clientY);
}
function showAt(x: number, y: number) {
  const menu = { width: element?.offsetWidth ?? 0, height: element?.offsetHeight ?? 0 };
  position.x = window.innerWidth - x < menu.width ? (x < menu.width ? x : x - menu.width) : x;
  position.y = window.innerHeight - y < menu.height ? (y < menu.height ? y : y - menu.height) : y;
  prevFocus = document.activeElement as HTMLElement | null;
  open = true;
}
function press(ev: PointerEvent) {
  reset();
  if (ev.pointerType !== "touch" || !ev.isPrimary) return;
  origin = { x: ev.clientX, y: ev.clientY };
  timer = setTimeout(() => {
    timer = undefined;
    if (lock) return;
    touched = true;
    showAt(origin.x, origin.y);
  }, LONG_PRESS);
}
function move(ev: PointerEvent) {
  if (timer === undefined) return;
  if (Math.hypot(ev.clientX - origin.x, ev.clientY - origin.y) > MOVE_CANCEL) cancel();
}
function cancel() {
  clearTimeout(timer);
  timer = undefined;
}
function reset() {
  touched = false;
}
```

**Notes:**

- `press` / `move` / `cancel` / `reset` are forward-referenced from the `$effect.pre` above them, so they MUST be hoisted `function` declarations, exactly like the existing `show` / `hide`. This is coding-style §8's stated exception; making them `const` arrows would be a temporal-dead-zone error.
- `on(target ?? document, "pointerdown", press)` type-checks with `press(ev: PointerEvent)` — **verified against `bun run check` during design**. No `as` cast is needed here, unlike the existing `show(ev: Event)`. Leave `show`'s signature alone.
- `timer`, `origin`, and `touched` are plain `let` — not `$state`. Nothing in the template reads them, and making them reactive would re-run the `$effect.pre` and tear down the listeners mid-gesture.
- `move` is registered on `document` and fires at high frequency; the `timer === undefined` guard keeps it cheap in the common case.

**Risk:** `pointermove`/`pointerup`/`pointercancel` are registered on `document` unconditionally rather than per-gesture. This is deliberate — it keeps listener wiring in one place. The guard in `move` and the idempotent `cancel` make them harmless when no gesture is in flight.

### (C-2) Coexistence with a native `contextmenu`

**Where:** `ContextMenu.svelte` — inside `show`.

**Required behavior:**

- When `contextmenu` fires while a long-press timer is pending, `show` MUST cancel that timer, so the menu opens once (via `contextmenu`) rather than twice.
- A pending timer at `contextmenu` time means the gesture is touch-originated, so `show` MUST also set the trailing-click suppression flag (C-3) in that case — and MUST NOT set it for a mouse right-click, where no timer is pending.

**Notes:** the `if (timer !== undefined) touched = true;` + `cancel()` pair in the C-1 sketch is the whole of this change. Order matters: read `timer` before `cancel()` clears it.

A mouse right-click does not emit a `click` event, so leaving `touched` false on that path is correct and preserves current behavior.

### (C-3) Trailing-click suppression

**Where:** `ContextMenu.svelte` — a new `hclick` handler; `<svelte:document>`.

**Required behavior:**

- When the menu was opened by a touch gesture, the first subsequent document `click` MUST NOT close the menu, and MUST clear the flag.
- The flag MUST also be cleared at the start of every `pointerdown`, so it can never persist and swallow an unrelated later tap.
- All other document clicks MUST close the menu exactly as today.

**Suggested implementation** (illustrative):

```ts
const hclick = () => {
  if (touched) return reset();
  hide();
};
```

```svelte
<svelte:document onclick={hclick} onkeydown={hkeydown} />
```

**Notes:** `hclick` is only referenced from the template, so it is a `const` arrow per coding-style §8, matching the existing `hkeydown`. `hide` keeps its own `lock` guard and stays unchanged.

**Risk:** the flag reset depends on `pointerdown` being observed for taps *outside* `target`. That is exactly why the `if (target)` branch registers `on(document, "pointerdown", reset)` in C-1: when `target` is set, `press` only runs for presses on the target, so without this listener a long-press-opened menu could never be dismissed by an outside tap. Do not drop it.

```md
# 017. ContextMenu touch long-press via pointer events

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-15
**Affects:** ContextMenu (internal interaction model); the library's touch long-press convention, shared with Tooltip

## Context

iOS Safari never fires the `contextmenu` event, so `ContextMenu` — which opened
only on `contextmenu` — could not be opened at all on iPad/iPhone. A long-press
gesture has to synthesize the trigger.

Three forces shaped the design:

- Android Chrome *does* fire `contextmenu`, at ~500 ms, while the finger is still
  down. Any timer-based path must not double-open on those devices.
- `ContextMenu` dismisses on a document `click`. A long press ends with
  `pointerup` → `click`, so the gesture that opens the menu also closes it.
- `Tooltip` already solves touch long-press in this library, using pointer events
  with a move threshold.

## Decision

Detect the long press with **pointer events** (`pointerdown` gated on
`pointerType === "touch"` and `isPrimary`, cancelled by `pointerup` /
`pointercancel` / a `pointermove` beyond 10 px), not touch events. The threshold
is a fixed 700 ms module constant, not a prop.

The `contextmenu` path stays authoritative: when `contextmenu` fires while the
long-press timer is pending, it cancels the timer and takes the open. A pending
timer is also what identifies the gesture as touch-originated.

An open caused by a touch gesture sets a flag that swallows exactly one
subsequent document `click`. The flag is cleared on the next `pointerdown`, so it
cannot persist and swallow an unrelated tap.

Suppressing iOS's native selection callout is left to the caller's CSS
(`-webkit-touch-callout: none; user-select: none;` on the target), documented in
the component's `@component` *Behavior*.

## Consequences

- Touch long-press is now a library convention with one shape: pointer events
  plus a move threshold. A third component needing it should follow Tooltip and
  ContextMenu rather than invent a touch-event path.
- One event model covers mouse, pen, and touch, so no touch/mouse event
  bookkeeping is duplicated.
- The trailing-click flag is invisible in the DOM and easy to break: any future
  change to the dismissal path must preserve both the swallow and the
  `pointerdown` reset, or the menu becomes either self-closing or undismissable.
- The fixed 700 ms is not tunable. Making it a prop later is additive and cheap;
  the prop would have to be named `delay` (as on Tooltip), since coding-style
  §13.2 reserves `duration` for motion timing.
- Callers who do not apply the callout CSS still get the menu on iOS, but with
  the native selection UI appearing during the press.

## Rejected alternatives

- **Touch events (`touchstart`/`touchmove`/`touchend`/`touchcancel`).** The shape
  most long-press references use, and it needs no `pointerType` gate. Rejected:
  it would leave the component straddling two event models (mouse via
  `contextmenu`, touch via touch events) and diverge from Tooltip's pointer-event
  precedent for the same gesture.
- **A `delay` prop for the threshold.** Rejected for now as unrequested API
  surface; 700 ms covers the requirement and the prop stays available later.
- **Component-applied inline style on `target`** to suppress the iOS callout.
  Rejected: it mutates a caller-owned element, and it cannot work at all in the
  default case where `target` is unset and the fallback is `document`.
- **Cancelling the long press on any `pointermove`.** Rejected: finger jitter
  fires `pointermove` constantly, so the gesture would rarely complete. Hence the
  10 px threshold, matching Tooltip.
```

## Invariants / expected behavior

- Mouse right-click: unchanged — `contextmenu` opens the menu, `preventDefault` is called, no `click` follows, `touched` stays false.
- `lock` blocks opening via **both** paths and blocks closing, as today.
- Escape and outside click still close and restore focus to `prevFocus`.
- With `target` set, a `contextmenu` outside the target still closes an open menu (via `show`'s `stopPropagation` keeping the target path from reaching the document `hide`).
- No public API change: `ContextMenuProps`, `ContextMenuReqdProps`, `ContextMenuBindProps`, and `_CONTEXT_MENU_PRESET` are untouched.

## Out of scope

- `Tooltip.svelte` — do not refactor its long-press logic, and do not extract a shared helper into `_core.ts`. Two call sites do not yet justify the abstraction; ContextMenu keeps its own local constants.
- Any change to `MenuList` / `MenuItem` / `MenuGroup` / `MenuSeparator`.
- Adding a `delay` (or any other) prop.
- `.ws/architecture/overview.md` — this change adds no component, no intra-library import, and no public API, so none of the overview's update triggers fire.
- Touching caller-owned DOM to suppress the iOS callout.

## Accepted risks / tradeoffs

- **Fixed 700 ms**, not configurable. Settled: matches the request, keeps the public API unchanged, and a `delay` prop stays additive later.
- **The iOS selection callout is the caller's problem.** Settled: the component does not mutate caller-owned elements, and the `target`-unset case (`document`) has nothing to style anyway. Documented in *Behavior* instead.
- **`document`-level pointer listeners** are always registered, not just during a gesture. Accepted for wiring simplicity; guarded to be cheap.

## Tests

Add to `tests/ContextMenu.svelte.test.ts` (browser env per coding-style §16 — jsdom is not needed; these use real timers). Keep the existing helpers and add:

const sleep = (ms: number) => new Promise((r) => setTimeout(r, ms));

async function pointer(type: string, el: Document | HTMLElement = document, init: PointerEventInit = {}) {
  await tick();
  const ev = new PointerEvent(type, {
    pointerType: "touch",
    isPrimary: true,
    clientX: 100,
    clientY: 200,
    bubbles: true,
    cancelable: true,
    ...init,
  });
  el.dispatchEvent(ev);
  await tick();
  return ev;
}

Cases to cover (a new `describe("ContextMenu touch long-press")` block):

1. **Opens after the long press, at the touch point** — `pointerdown` at (300, 400), `await sleep(750)`, `await tick()`; `open === true` and the root has `left: 300px` / `top: 400px`.
2. **Release before the threshold does not open** — `pointerdown`, `sleep(300)`, `pointerup`, `sleep(500)`; `open === false`.
3. **Move beyond 10 px cancels** — `pointerdown` at (100, 200), `pointermove` at (100, 220), `sleep(750)`; `open === false`.
4. **Move within 10 px does not cancel** — `pointerdown` at (100, 200), `pointermove` at (103, 202), `sleep(750)`; `open === true`.
5. **`pointercancel` cancels** — `pointerdown`, `pointercancel`, `sleep(750)`; `open === false`.
6. **Non-touch pointer does not long-press** — `pointerdown` with `pointerType: "mouse"`, `sleep(750)`; `open === false`.
7. **The gesture's trailing click does not close the menu** — `pointerdown`, `sleep(750)`, `pointerup`, then `clickDocument()`; `open === true`.
8. **A later outside tap does close it** — continue case 7 with `pointerdown` → `pointerup` → `clickDocument()`; `open === false`. This is the regression guard for a stuck flag.
9. **`contextmenu` mid-gesture cancels the pending timer** — `pointerdown` at (100, 200), `sleep(300)`, `contextmenu(document, { clientX: 150, clientY: 250 })`, then `sleep(750)`; `open === true` and the position is still `left: 150px` / `top: 250px` (i.e. the timer did not fire and re-position to the press origin).
10. **`lock` blocks long-press opening** — `lock: true`, `pointerdown`, `sleep(750)`; `open === false`.
11. **Long press on `target` opens** — with `target` set (use the existing `appendTarget()`), `pointerdown` on the target, `sleep(750)`; `open === true`.
