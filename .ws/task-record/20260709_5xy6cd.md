## Goal

Bring `Pagination` in line with the library's prop convention: extend its props
from the root `<nav>` element's HTML attributes (like Modal/Drawer/Toggle/Tabs),
and collapse the five flat `aria*Label` props into a landmark name
(`ariaLabel?: string`, unchanged in type/meaning across the library) plus a single
grouped object for the four button labels (`buttonLabels?: PaginationLabel`).

## Done when

- `PaginationProps extends Omit<HTMLAttributes<HTMLElement>, "children" | "aria-label">`;
  caller `class` merges onto the `<nav>` and other attributes forward via `...rest`.
- The four props `ariaTopLabel` / `ariaLeftLabel` / `ariaRightLabel` /
  `ariaBottomLabel` are gone; a `buttonLabels?: PaginationLabel` prop replaces them
  and `ariaLabel?: string` remains the nav landmark name (default `"Pagination"`).
- Exported `PaginationLabel = { top?: string; left?: string; right?: string; bottom?: string }`.
- The `@component` *Types* block matches the new `PaginationProps` (§11).
- `.ws/decision/004-…md` is created with the body in (C-4).
- `bun run check`, the scoped Pagination test, and scoped `fmt:check` all pass.

## Background / constraints

`Pagination`'s root is a `<nav>` landmark that **is** the `PARTS.WHOLE` element
(unlike Tabs, whose root and role element differ). So `class`/`...rest` land on the
nav root — the strict coding-style §14 default; no deviation, no split.

What must keep working:
- Default accessible names are unchanged: nav `"Pagination"`, buttons
  `"First page"` / `"Previous page"` / `"Next page"` / `"Last page"`.
- All existing paging behavior (clamp, at-bound suppression, ComboBox commit,
  `options`/`thin`, context wiring) is untouched.

## Detailed changes

### (C-1) Extend from nav HTML attributes

**Where:** `PaginationProps` (module `<script>` and the `@component` *Types*
block), the instance `<script>` `$props()` destructure, and the `<nav>` tag.

**Required behavior:**
- `PaginationProps` MUST `extends Omit<HTMLAttributes<HTMLElement>, "children" | "aria-label">`.
  `class` MUST stay out of the `Omit` so it merges.
- The instance destructure MUST take `class: c, ...rest` (in addition to existing props).
- The `<nav>` MUST apply `class={[cls(PARTS.WHOLE, variant), c]}` and spread
  `{...rest}` **before** the component-owned `aria-label` attribute (§14).

**Suggested implementation:**
```ts
// module script: import type
import type { HTMLAttributes, MouseEventHandler } from "svelte/elements";
// interface header
export interface PaginationProps extends Omit<HTMLAttributes<HTMLElement>, "children" | "aria-label"> {
```
```svelte
<nav class={[cls(PARTS.WHOLE, variant), c]} {...rest} aria-label={ariaLabel}>
```

**Risk:** none beyond the API surface widening; `<nav>` has no owned `role`/`type`
to protect, only `aria-label`.

### (C-2) Consolidate the aria labels

**Where:** `PaginationProps`, the `$props()` destructure, and the four button
`aria-label` attributes.

**Required behavior:**
- Add and export `PaginationLabel = { top?: string; left?: string; right?: string; bottom?: string }`.
- `PaginationProps` MUST expose `ariaLabel?: string` (nav landmark) and
  `buttonLabels?: PaginationLabel`, and MUST NOT expose `ariaTopLabel`,
  `ariaLeftLabel`, `ariaRightLabel`, or `ariaBottomLabel`.
- Effective button names MUST fall back per field to the existing defaults:
  `top → "First page"`, `left → "Previous page"`, `right → "Next page"`,
  `bottom → "Last page"`. `ariaLabel` MUST default to `"Pagination"`.
- The `top`/`left`/`right`/`bottom` **snippet** props (button content) are
  unrelated and MUST remain unchanged.

**Suggested implementation:**
```ts
export type PaginationLabel = { top?: string; left?: string; right?: string; bottom?: string };
// destructure (drop the four aria*Label defaults; keep ariaLabel default):
//   ariaLabel = "Pagination", buttonLabels, ... class: c, ...rest
// resolve button names once:
const btn = $derived({
  top: buttonLabels?.top ?? "First page",
  left: buttonLabels?.left ?? "Previous page",
  right: buttonLabels?.right ?? "Next page",
  bottom: buttonLabels?.bottom ?? "Last page",
});
```
Then each button uses `aria-label={btn.top}` / `btn.left` / `btn.right` / `btn.bottom`.

**Notes:** keep `PaginationReqdProps = never` and `PaginationBindProps = "value"`.

```md
# 004. Pagination keeps the nav landmark name as `ariaLabel: string` and groups button labels into `buttonLabels`

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-09
**Affects:** `Pagination` public prop surface (`PaginationProps`)

## Context

`Pagination` renders a `<nav>` landmark root containing four icon control buttons
(first / prev / next / last) plus an embedded `ComboBox`. Accessible naming needs a
name on the nav landmark and on each icon button, so the component previously
exposed five flat props: `ariaLabel` plus `ariaTopLabel` / `ariaLeftLabel` /
`ariaRightLabel` / `ariaBottomLabel`. Migrating `Pagination` to extend the root
element's HTML attributes (matching Modal/Drawer/Toggle/Tabs) and to reduce the
prop-list noise forced a choice about how to shape these labels. The library-wide
convention is `ariaLabel?: string` = the component's own accessible name; folding
all five labels into one grouped object was a live alternative.

## Decision

The nav landmark name stays `ariaLabel?: string` (default `"Pagination"`),
identical in type and meaning to every other component's `ariaLabel`. The four
button labels — internal control names, not the component's landmark name — are
grouped into `buttonLabels?: PaginationLabel`, where
`PaginationLabel = { top?; left?; right?; bottom? }` keyed by `PARTS`. The four
`aria*Label` props are removed. `PaginationProps` extends
`Omit<HTMLAttributes<HTMLElement>, "children" | "aria-label">`; `class` and
`...rest` land on the `<nav>` `PARTS.WHOLE` root — the strict coding-style §14
default, since the nav root *is* the landmark (no Tabs-style split).

## Consequences

- `ariaLabel` keeps one type and one meaning across the whole library; the button
  labels no longer clutter the top-level prop interface.
- Two label surfaces coexist by design — a landmark `string` and a control-label
  object. Reviewers should not "unify" them into one object: the string carries the
  landmark semantics shared with every other component.
- Breaking change for callers using the old `aria*Label` prop names; they migrate
  to `buttonLabels={{ … }}`.
- Merging the two surfaces into one object later would be a breaking API change.

## Rejected alternatives

- **Single `ariaLabel?: PaginationLabel` for all five labels.** Internally tidy
  (keys match `PARTS`), but overloads `ariaLabel` to a non-string type, breaking the
  library-wide `ariaLabel: string` convention and detaching the landmark name from
  its cross-component meaning.
- **Keep the five flat `aria*Label` props.** Rejected: prop-list noise, and
  inconsistent with the extends pattern adopted across the other components.
```

## Invariants / expected behavior

- With no label props, rendered names are exactly the current defaults.
- `ariaLabel="X"` sets the nav's `aria-label`; `buttonLabels={{ left: "Y" }}` sets
  only the prev button's `aria-label`, the other three keep their defaults.
- Caller `class` appears on the `<nav>` alongside the preset/part class; other
  passed attributes (e.g. `id`, `data-*`) appear on the `<nav>`.

## Out of scope

- Paging logic, `thin`/`options`, ComboBox context wiring, the glyph snippets, and
  the `top`/`left`/`right`/`bottom` **content** snippet props.
- Any `.ws/architecture/` change (this shifts no system shape).
