## Goal

`Tabs` currently renders a `role="tablist"` element but exposes **no way to give
it an accessible name** — authors cannot pass `aria-label` / `aria-labelledby`
(nor any other attribute) to the tablist. This is the single remaining WAI-ARIA
APG gap in the component (roving `tabindex`, arrow/Home/End navigation,
`manual`/automatic activation, `aria-selected` / `aria-controls` /
`aria-labelledby` panel wiring, `aria-orientation`, and disabled handling are all
already implemented and MUST keep working).

Add an idiomatic rest-props passthrough — matching `ToggleGroup` — so authors can
name and decorate the tablist. Because the accessible name must sit on the
`role="tablist"` element (not the outer wrapper), the passthrough target is the
**tablist**, a deliberate deviation from coding-style §14's default of spreading
onto the root element; record that judgment as an ADR (C-4).

## Done when

  pass `aria-label` / `aria-labelledby` (and other div attributes) that land on
  the `role="tablist"` element.
- A caller's `class` merges onto the tablist alongside the component's own
  `PARTS.TOP` class; component-owned `role` / `aria-orientation` / `onfocusin`
  cannot be overridden by rest props.
- The `@component` doc block (Types + Anatomy) reflects the new interface and

- `bun run check`, the Tabs test suite, and scoped `fmt:check` all pass.

## Background / constraints

- **Two-level structure.** `Tabs` renders an outer `PARTS.WHOLE` `<div>` (no ARIA
  role) wrapping an inner `PARTS.TOP` `<div role="tablist">` and the panels.
  Unlike `ToggleGroup` (whose single root **is** the role element), Tabs' root and
  its role element differ — so the rest/class target must be the inner tablist,

- **Per-part styling is unchanged.** The `styling` prop and `PARTS.*` classes keep
  owning per-part class generation; the outer `PARTS.WHOLE` wrapper stays
  styling-only with no author passthrough.
- Everything listed under Goal as already-implemented MUST keep working
  (behaviour over cleanliness).

## Detailed changes

### (C-1) Extend `TabsProps` with `HTMLAttributes`

**Where:** the `export interface TabsProps` in `<script module>`, and the type
imports at the bottom of `<script module>`.

**Required behavior:**
- `TabsProps` MUST extend `Omit<HTMLAttributes<HTMLDivElement>, "children" | "role" | "aria-orientation" | "onfocusin">`. Keep `class` **out** of the `Omit` so it flows through as a caller prop (§14).
- The required prop (`tabs`) MUST stay first (§12); existing feature props keep their current relative order.
- Add `import type { HTMLAttributes } from "svelte/elements";` to the `<script module>` type imports.

**Suggested implementation:**

```ts
export interface TabsProps extends Omit<
  HTMLAttributes<HTMLDivElement>,
  "children" | "role" | "aria-orientation" | "onfocusin"
> {
  tabs: TabItem[];
  current?: string; // bindable (first enabled tab value)
  manual?: boolean; // (false) manual activation: Arrow keys only move focus; Enter/Space selects
  orientation?: "horizontal" | "vertical"; // ("horizontal")
  styling?: SVSClass;
  variant?: SVSVariant; // (VARIANT.NEUTRAL)
  // other div attributes are passed to the role="tablist" element via ...rest; `class` is merged onto the tablist
  // aria-label / aria-labelledby name the tablist
}
```

**Notes:** `role`, `aria-orientation`, and `onfocusin` are component-owned on the
tablist; omitting them keeps the type honest (they cannot be set by callers).
`children` is omitted because `Tabs` has no children slot. Do **not** omit
`aria-label` / `aria-labelledby` — enabling them is the point.

### (C-2) Route `class` + rest props to the tablist

**Where:** the `// prettier-ignore` `$props()` destructure line, and the
`<div … role="tablist" …>` element in the markup.

**Required behavior:**
- Destructure `class: c` and `...rest` from `$props()` (§2, single-line, `// prettier-ignore` retained).
- The `role="tablist"` element MUST merge the caller `class` (`c`) after the component's own class, and spread `{...rest}` **before** the component-owned `role` / `aria-orientation` / `onfocusin` so callers cannot override those semantics (§14).
- The outer `PARTS.WHOLE` element MUST NOT receive `c` or `...rest` (it stays styling-only).
- `aria-label` / `aria-labelledby` MUST flow to the tablist **through `{...rest}` only**. `Tabs` computes no merged/derived value for them (it has no context), so they MUST NOT be destructured out and re-emitted as explicit `aria-label={…}` / `aria-labelledby={…}` attributes. Emitting an explicit value (which would be `undefined` in the no-arg case) **after** `{...rest}` would clobber the caller's spread value and erase the attribute — the exact failure in knowledge `svelte-rest-props-reemitted-aria-undefined-clobber.md`. Only destructure-and-re-emit is required for a prop that has a computed value; none here does.

**Suggested implementation:**

```svelte
<!-- $props() line -->
let { tabs, current = $bindable<string | undefined>(undefined), manual = false, orientation = "horizontal", styling, variant = VARIANT.NEUTRAL, class: c, ...rest }: TabsProps = $props();

<!-- tablist element -->
<div class={[cls(PARTS.TOP, variant), c]} {...rest} role="tablist" aria-orientation={orientation} onfocusin={hfocusin}>
```

**Risk:** low. `{...rest}` precedes owned attrs, so `role` / `aria-orientation` /
`onfocusin` win regardless of caller input; owned behaviour is preserved.

**Required behavior:** create the file with **exactly** this body:

```md
# 002. Tabs forwards rest props and caller class to the tablist, not the root wrapper

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-08
**Affects:** `Tabs` public prop surface (`TabsProps`)

## Context

`Tabs` renders a two-level structure: an outer `PARTS.WHOLE` `<div>` (no ARIA
role) wrapping an inner `<div role="tablist">` (`PARTS.TOP`). Coding-style §14's
default is to spread `...rest` and merge the caller's `class` onto the root
(WHOLE) element. But WAI-ARIA APG requires the tablist's accessible name
(`aria-label` / `aria-labelledby`) to sit on the `role="tablist"` element, and
`Tabs` previously exposed no way to supply it. A single-element component like
`ToggleGroup` has no tension here — its root *is* the role element — but `Tabs`
does, because its root and its role element are different nodes.

## Decision

`TabsProps` extends `Omit<HTMLAttributes<HTMLDivElement>, "children" | "role" |
"aria-orientation" | "onfocusin">`. The caller's `class` and `...rest` are applied
to the inner `role="tablist"` element (spread before the component-owned `role` /
`aria-orientation` / `onfocusin`), **not** the WHOLE root. Per-part styling stays
with the `styling` prop and `PARTS.*` classes; the outer wrapper remains
styling-only with no author passthrough.

## Consequences

- Authors can name the tablist and pass arbitrary tablist attributes idiomatically,
  matching `ToggleGroup`'s rest-props convention.
- Caller `class` lands on the tablist, not the outermost element — a deliberate,
  documented deviation from §14's "rest/class on WHOLE" default for a two-level
  component. Reviewers should not "correct" it back to the root.
- The outer WHOLE wrapper stays styling-only; authors cannot attach attributes to it.
- Moving the passthrough target later would be a breaking API change.

## Rejected alternatives

- **Rest/class on the WHOLE root (strict §14).** The accessible name would land on
  a role-less wrapper, failing the APG requirement that it name the tablist.
- **Dedicated `aria-label` / `aria-labelledby` props only.** Minimal and
  a11y-scoped, but diverges from the library's established rest-props idiom and
  forecloses passing other tablist attributes.
- **Split `class`→WHOLE but `rest`→tablist.** Surprising and inconsistent; forces
  two mental models for where author input lands.
```

## Invariants / expected behavior

- Existing keyboard, roving-`tabindex`, activation (`manual`/automatic),
  `aria-selected` / `aria-controls` / panel `aria-labelledby` wiring,
  `aria-orientation`, and disabled-tab handling are unchanged.
- With no `aria-label` / `aria-labelledby` / rest props supplied, the rendered
  output is identical to today (the merged `class` array with `c === undefined`
  and an empty `{...rest}` produce no extra attributes).

## Out of scope

- Any behavioural change to keyboard navigation, activation, focus management, or
  panel rendering.
- Adding attribute passthrough to the outer `PARTS.WHOLE` wrapper or to individual
  `role="tab"` buttons / `role="tabpanel"` panels.
- Any change to other components.

## Accepted risks / tradeoffs

- Caller `class` merges onto the tablist rather than the outermost element; this is
  the intended design (see ADR C-4), consistent with routing all author passthrough
  to the semantic widget.

## Tests

In `tests/Tabs.svelte.test.ts`, add a small block (mirror existing `render(Tabs,
{ … })` + `container.querySelector('[role="tablist"]')` usage):

- **aria-label passthrough:** `render(Tabs, { tabs: baseTabs(), "aria-label": "Sections" })` → the `role="tablist"` element `toHaveAttribute("aria-label", "Sections")`.
- **aria-labelledby passthrough:** likewise for `"aria-labelledby": "ext-id"`.
- **class merge:** `render(Tabs, { tabs: baseTabs(), class: "my-tabs" })` → the tablist's `class` contains `"my-tabs"` (and still carries the component's own `PARTS.TOP` class).
- **owned attrs not overridable:** passing `role: "presentation"` (or `"aria-orientation": "vertical"` while `orientation` stays horizontal) still yields `role="tablist"` / `aria-orientation="horizontal"` on the element.
