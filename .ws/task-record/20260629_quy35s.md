## Goal

Add a headless **simple pagination** component shaped `[«] [‹] [ComboBox] [›] [»]`: built-in
first/prev/next/last buttons around an editable `ComboBox` that shows and edits the current page,
with the page-shortcut list shown as the ComboBox's options. The component is a **dual-mode context
coordinator** (the `ToggleGroupField` pattern), so the ComboBox is either generated internally and
configured via a `comboBox` namespaced bag, or written explicitly as `children` and self-wired via an
optional context. Realizing the `children` form requires `ComboBox` to become context-capable
(additive, standalone behavior unchanged).

This lands during a major-version `dev` cycle, so breaking changes are acceptable where they
simplify structure; but the `ComboBox` change here is deliberately **non-breaking** for standalone use.

## Done when

- `Pagination` renders the anatomy below; first/prev/next/last change `value` and are `aria-disabled`
  at the `[min, max]` bounds; the ComboBox shows the current page and commits an edited page on
  blur/Enter/option-select, reverting invalid input and clamping out-of-range input.
- Both usage forms work: `<Pagination comboBox={...} />` (generated child) and
  `<Pagination><ComboBox/></Pagination>` (caller-written child self-wires via context).
- `ComboBox` used standalone behaves exactly as before (all existing `ComboBox` tests pass unchanged);
  used inside a `ComboBoxContext`, it sources `options`/`value`/`variant`/`styling` from the context
  and routes value writes and commit back to it.
- The shortcut option list defaults to a **magnitude-thinned** sequence over `[min, max]` (dense near
  the start, sparser for larger pages; always includes `min`, `max`, and the current `value`) and is
  overridable via `options?: number[]`.
- `bun run check`, the targeted `vitest` runs, and `bun run fmt` all pass; `.ws/architecture/overview.md`
  reflects the new component.

## Background / constraints

- **Naming (exact):** the component is spelled **`Pagination`** (the draft's "Pagenation" is a
  misspelling). File `Pagination.svelte`, exports `PaginationProps` / `PaginationReqdProps` /
  `PaginationBindProps`, preset `_PAGINATION_PRESET = "svs-pagination"`.
- **Current-page prop is `value`** (not `page`) and the bounds are **`min`/`max`** — matching the
  `value`/`min`/`max` numeric idiom of `Slider` / `NumberInput`.
- **No number-only mode is added to `ComboBox`.** `ComboBox` stays string-based; `Pagination` owns all
  numeric coercion/clamping. (This is the resolution of the draft's open question.)
- **What must keep working:** every existing `ComboBox` standalone behavior and test (search filter,
  open/close, arrow navigation, option apply, overflow flip, `aria-*`). The context wiring is purely
  additive and inert when no context is present.
- `src/lib/index.ts` and `src/lib/_svseeds/dep.json` are **generated** by `bun run build`
  (do not hand-edit); the new `Pagination → ComboBox` import is picked up automatically. They are not
  required for `check`/`test` (tests import via the `#svs/*` subpath), so regenerating them is optional.

## Detailed changes

### `(C-1)` `ComboBoxContext` + accessor pair

**Where:** `ComboBox.svelte` `<script module>`, alongside the existing exports (follow
`NumberInputContext` / `_getNumberInputContext` in `NumberInput.svelte`, and `compound-components.md` §3).

**Required behavior:**
- MUST add `_createContext` to the value import from `./_core`.
- MUST export an interface extending `SVSContext`:
  ```ts
  export interface ComboBoxContext extends SVSContext {
    get options(): SvelteSet<string> | Set<string>;
    get value(): string;
    set value(v: string);
    commit?(): void;
  }
  ```
- MUST export `export const [_getComboBoxContext, _setComboBoxContext] = _createContext<ComboBoxContext>();`.
- `SVSContext` already supplies `get variant` / `get styling`; import `SVSContext` as a type from `./_core`.

**Notes:** Do **not** document the context in the `@component` *Types* block (`coding-style.md` §11 —
internal Context interfaces are not listed). `ComboBoxBindProps` stays `"value" | "expanded" | "element"`.

### `(C-2)` Dual-mode consumption in `ComboBox`

**Where:** `ComboBox.svelte` instance `<script>` and template.

**Required behavior:**
- The component MUST read `const ctx = _getComboBoxContext();` once at init.
- When `ctx` is set, `options`, `value`, and `variant` MUST come from the context, and value writes
  MUST be routed back to the context; when `ctx` is undefined, behavior MUST be identical to today.
- `styling` MUST resolve as `styling ?? ctx?.styling` (the caller's own `styling` still wins), per
  `coding-style.md` §15.
- `options` MUST become **optional** (`options?: SvelteSet<string> | Set<string>;`), so a
  context-embedded `<ComboBox/>` need not be passed options. `ComboBoxReqdProps` MUST become `never`.
  The render guard MUST use the effective options' size (e.g. `effOptions?.size`).
- When the listbox closes (the existing `close()` path — blur, Escape, document scroll, and after an
  option is applied), the component MUST call `ctx?.commit?.()` so a coordinator can finalize the value.
- Standalone, `commit` is unused and MUST have no effect.

**Suggested implementation (illustrative):**
```ts
const ctx = _getComboBoxContext();
const effOptions = $derived(ctx ? ctx.options : options);
const effValue = $derived(ctx ? ctx.value : value);
const effVariant = $derived(ctx ? ctx.variant : variant);
const cls = $derived(_fnClass(_COMBO_BOX_PRESET, styling ?? ctx?.styling));
function setValue(v: string) { if (ctx) ctx.value = v; else value = v; }
```
- Replace internal `value` reads with `effValue` (`view` filter, `open()`'s `view.indexOf`,
  `hinput`'s `options.has(value)`/`view.indexOf`), internal `options` reads with `effOptions`
  (`opts`, `hinput`, the render guard), and `variant` with `effVariant` in the `cls(...)` calls.
- In `apply()`, capture the chosen string locally to avoid reading a not-yet-recomputed derived:
  `const v = view[selected]; setValue(v); if (element) element.selectionStart = v.length; close();`
- In `close()`, after resetting `expanded`/`selected`/`typed`, add `if (ctx) ctx.commit?.();`.
- Template input: `bind:value={() => effValue, setValue}` (function binding, `compound-components.md`
  §8; svelte `^5.9` is within the peer range). Keep `bind:this={element}`.

**Risk:** several internal `value`/`options`/`variant` references must all be switched to the
effective forms consistently — missing one would desync embedded mode. Element mirroring
(`compound-components.md` §7) is **not** required here (Pagination does not need the input element);
do not add a `set element` member.

### `(C-3)` `Pagination.svelte` — new component

**Where:** new file `src/lib/_svseeds/Pagination.svelte`. Mirror `ToggleGroupField.svelte`'s structure
(module exports → instance: `Initialize` / `Initialize Context` / `States` / `Reactive Handlers` /
`Event Handlers` banners → template with `{#snippet}` helpers).

**Props interface (`coding-style.md` §12 — all props optional here, so order is the author's; keep
this order):**
```ts
export interface PaginationProps {
  children?: Snippet<[number, string]>; // Snippet<[value,variant]> — caller-written ComboBox child
  value?: number;            // bindable; current page (default 1)
  min?: number;              // first page (default 1)
  max?: number;              // last page (default = max(min, value))
  options?: number[];        // shortcut list shown in the ComboBox (default: magnitude-thinned min..max)
  top?: Snippet<[number, string]>;    // Snippet<[value,variant]> — FIRST button content
  left?: Snippet<[number, string]>;   // Snippet<[value,variant]> — PREV button content
  right?: Snippet<[number, string]>;  // Snippet<[value,variant]> — NEXT button content
  bottom?: Snippet<[number, string]>; // Snippet<[value,variant]> — LAST button content
  ariaLabel?: string;        // nav landmark label (default "Pagination")
  ariaTopLabel?: string;     // FIRST (default "First page")
  ariaLeftLabel?: string;    // PREV  (default "Previous page")
  ariaRightLabel?: string;   // NEXT  (default "Next page")
  ariaBottomLabel?: string;  // LAST  (default "Last page")
  comboBox?: Omit<ComboBoxProps, ComboBoxReqdProps | ComboBoxBindProps | "options" | "variant" | "styling">;
  styling?: SVSClass;
  variant?: SVSVariant;      // (VARIANT.NEUTRAL)
}
export type PaginationReqdProps = never;
export type PaginationBindProps = "value";
export const _PAGINATION_PRESET = "svs-pagination";
```

**Part-name mapping (settled — unify on `PARTS`, not intuitive role names):** the four buttons reuse
the existing `PARTS` vocabulary by position, not by semantic role —
`PARTS.TOP` = **first**, `PARTS.LEFT` = **prev**, `PARTS.RIGHT` = **next**, `PARTS.BOTTOM` = **last**.
The button-content snippets (`top`/`left`/`right`/`bottom`) and aria labels
(`ariaTopLabel`/`ariaLeftLabel`/`ariaRightLabel`/`ariaBottomLabel`) follow the same mapping. There are
**no** separate decoration slots and no `MIDDLE` wrapper — the snippets ARE the button content.

**Required behavior:**
- MUST import `ComboBox` (default) and `_setComboBoxContext` from `./ComboBox.svelte`, and the types
  `ComboBoxProps` / `ComboBoxReqdProps` / `ComboBoxBindProps` (dependency direction Pagination → ComboBox).
- MUST set a `ComboBoxContext` via `_setComboBoxContext(ctx)` unconditionally, so both the
  generated child and a caller-written `children` child self-wire.
- **Bounds & clamping:** let `effMax = max ?? Math.max(min, value)`. `value` MUST be kept within
  `[min, effMax]` (round to integer) whenever the bounds or `value` change — use an
  `$effect.pre` + `untrack(sync)` (mirror `MonthPicker.svelte`'s clamp). `min` defaults to 1.
- **Buttons** (first/prev/next/last → parts TOP/LEFT/RIGHT/BOTTOM, each a `<button type="button">`):
  - first → set page to `min`; prev → `value - 1`; next → `value + 1`; last → `effMax` (each clamped).
  - **No `disabled`.** Instead, first/prev MUST emit `aria-disabled={value <= min}` and next/last
    `aria-disabled={value >= effMax}`. The button's click handler MUST be a no-op when the button is
    in its at-bound state (mirror `Button.svelte`'s `if (reason) return;`), so an `aria-disabled`
    button stays focusable/in tab order but performs no action. **No `aria-description`** is emitted
    (the `aria-label` + `aria-disabled` state suffice — confirmed; do not add a description prop).
  - A button at its bound MUST render its part class with `VARIANT.INACTIVE` (e.g.
    `cls(PARTS.TOP, atMin ? VARIANT.INACTIVE : variant)`) as a styling hook; otherwise with `variant`.
  - Each MUST carry its `aria-label` from the mapped `aria{Top,Left,Right,Bottom}Label`.
  - Button content renders the mapped snippet if supplied (called as `top(value, variant)` etc.),
    else a default SVG glyph: chevron-left/right for prev(LEFT)/next(RIGHT), double-chevron-left/right
    for first(TOP)/last(BOTTOM). Per the simplified anatomy the glyph/snippet sits **directly** inside
    the `<button>` (no inner `PARTS.EXTRA` span).
- **ComboBox value bridge (commit semantics):** keep a display string `text` (`$state`, init
  `String(value)`). The context's `value` getter returns `text`; its setter assigns `text` (live
  typing updates `text` only, never the committed page). On `ctx.commit()`, parse `text` as an
  integer: a valid in-range integer becomes the new `value` (clamped); empty/non-integer reverts.
  After any committed `value` change (buttons, prop, or commit) `text` MUST resync to `String(value)`
  — implement as `$effect(() => { value; untrack(() => { text = String(value); }); })` so it fires
  only when the committed page changes, never mid-typing.
- **Shortcut options (current-page-centered thinning):** `ctx.options` MUST be a `Set<string>` (or
  `SvelteSet<string>`) built from `(options ?? thin(min, effMax, value)).map(String)`, recomputed via
  `$derived` when inputs change. When `options` is omitted, the default MUST be the `thin` function
  **below** (this is the prescribed default — it is usability-critical, not merely illustrative). It
  keeps the list **dense around the current page** and sparse toward the bounds, always including
  `min`, `effMax`, and the current `value`, returning strictly-increasing unique integers in
  `[min, effMax]`. Use it verbatim (the loop variable is `step`, **not** `d`, to avoid shadowing the
  outer magnitude unit `d` — keep it that way so `svelte-check` does not warn):
  ```ts
  function thin(min: number, max: number, value: number): number[] {
    const v = Math.trunc(value);
    const arr = (add: boolean) => {
      const d = Math.max(1, Math.trunc(max / 10));
      const r = (n: number) => Math.round(n / d) * d;
      const delta = [1, 1, d, d * 2];
      const result: number[] = [];
      for (const step of delta) {
        const next = (result.at(-1) ?? v) + (add ? step : -step);
        result.push(step === 1 ? next : r(next));
      }
      return result;
    };
    return [...new Set<number>([v, min, max, ...arr(true), ...arr(false)])]
      .filter((n) => min <= n && n <= max)
      .sort((a, b) => a - b);
  }
  // thin(1, 100, 50) → 1, 20, 40, 48, 49, 50, 51, 52, 60, 80, 100
  // thin(1, 100, 1)  → 1, 2, 3, 10, 30, 100
  ```
  Assumptions (guaranteed by the component): `min`/`effMax` are integers and `value` is the integer
  current page already clamped into `[min, effMax]`. For `effMax < 10` the unit `d` collapses to `1`
  (near-contiguous list) — acceptable.
- **Dynamic caller `options`:** when a caller supplies `options` and wants it to follow the current
  page (e.g. a window around `value`), they derive it from the bound `value` — `bind:value={page}`
  then `const opts = $derived(windowAround(page))`. The component MUST NOT add an `events` /
  `onchange` notification prop for this: `value` is already bindable, so a plain `$derived` covers it,
  and the `events` veto-hook vocabulary (TagsInput-style add/remove) is a different concern. Document
  this caller pattern in the `@component` *Behavior* (`(C-4)`).
- **Context object** (`compound-components.md` §3–§5; declare any reactive value a getter reads
  *before* the literal):
  ```ts
  const ctx: ComboBoxContext = {
    get options() { return optionSet; },
    get value() { return text; },
    set value(v) { text = v; },
    get variant() { return variant; },
    get styling() { return `${_PAGINATION_PRESET} ${_COMBO_BOX_PRESET}`; },
    commit() { /* parse text → clamp → value */ },
  };
  _setComboBoxContext(ctx);
  ```
  (Import `_COMBO_BOX_PRESET` from `./ComboBox.svelte` for the styling chain, mirroring how
  `ToggleGroupField` chains `_TOGGLE_GROUP_FIELD_PRESET` + `_TOGGLE_GROUP_PRESET`.)

**Anatomy (template):**
```svelte
<nav class="whole" aria-label={ariaLabel}>
  <button class="top"    type="button" aria-label aria-disabled>{top}</button>    <!-- first -->
  <button class="left"   type="button" aria-label aria-disabled>{left}</button>   <!-- prev  -->
  {#if children}{@render children(value, variant)}{:else}<ComboBox {...comboBox} />{/if}
  <button class="right"  type="button" aria-label aria-disabled>{right}</button>  <!-- next  -->
  <button class="bottom" type="button" aria-label aria-disabled>{bottom}</button> <!-- last  -->
</nav>
```
- Outer element is a real `<nav>` landmark (`PARTS.WHOLE`) with `aria-label={ariaLabel}`.
- Button parts use `PARTS.TOP`/`PARTS.LEFT`/`PARTS.RIGHT`/`PARTS.BOTTOM` per the mapping above; the
  snippet/default-glyph sits directly inside the `<button>`. All classes keyed by the per-button
  effective variant (`VARIANT.INACTIVE` at bound, else `variant`).
- The generated child is rendered as a bare `<ComboBox {...comboBox} />` (no `options`/`value`/
  `variant`/`styling` — the context supplies them).

**Risk:** declaration ordering (`compound-components.md` §5) — `optionSet` and `text` (read by the
context getters) MUST be declared before the `ctx` literal.

## Invariants / expected behavior

- Standalone `ComboBox` is byte-for-byte behavior-compatible (no context ⇒ every `eff*` falls back to
  props, `commit` is a no-op).
- `value` is always an integer in `[min, effMax]` after any interaction or prop change.
- An at-bound button is `aria-disabled` yet focusable; activating it (pointer or keyboard) does not
  change `value` — no `disabled` attribute is ever set.
- When `children` is supplied to `Pagination`, the `comboBox` bag is ignored (the caller-written child
  wins), exactly like `ToggleGroupField`'s `children` vs `toggleGroup`.

## Out of scope

- Any number-only mode on `ComboBox`; any change to `ComboBox`'s search/keyboard/overflow logic
  beyond routing through the effective values.
- Classic numbered-page pagination (1 2 3 … N buttons); ellipsis/window logic beyond the combobox
  shortcut list.
- An `events` / `onchange` value-change notification prop (callers derive dynamic `options` from the
  bound `value` via `$derived`).
- Changes to `_core.ts` (confirm none are needed) and to any other component.
- Hand-editing `src/lib/index.ts` / `dep.json` (generated).

## Accepted risks / tradeoffs

- `ComboBox` gains an optional context surface for a single current consumer (`Pagination`); accepted
  because it is the only way to honor the caller-written `<ComboBox/>` form and follows the existing
  `NumberInput`/`ToggleGroup` precedent.
- `options` becoming optional widens `ComboBox`'s API (a forgotten `options` no longer type-errors
  standalone); accepted as part of the major-version reshaping and required for the embedded form.
- For very large page counts the default `min..max` shortcut list is large; the caller mitigates via
  the `options` override.
