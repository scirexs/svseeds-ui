# Task: Restructure Popover arrow into caret parts with flip-following

Target file: `src/lib/_svseeds/Popover.svelte`

- `src/lib/_svseeds/_core.ts` — `PARTS` already contains `WHOLE`/`MAIN`/`TOP`/`BOTTOM`/`LEFT`/`RIGHT`; **`_core.ts` is NOT modified**.
- `.ws/policy/coding-style.md` — §5 (`$effect` + `untrack`), §8 (`hXxx` vs control functions), §11 (`@component` Anatomy), §3 (`const`/`$derived`).

## Goal

Make Popover's `arrow` render a true speech-bubble caret that sticks out past the
panel frame and **follows autoFlip reversal**, without negative margins or dynamic
`rotate`. Split the panel into a decorated `MAIN` body and sibling carets
(`TOP`/`BOTTOM`/`LEFT`/`RIGHT`); render only the one caret that points at the
trigger, chosen from the panel/trigger rect comparison after CSS Anchor resolves
placement. Pre-release allows the breaking styling-contract change.

## Done when

- With `arrow`, the panel renders `<div class="main">…children…</div>` wrapping the
  content, plus exactly **one** sibling caret div chosen by the resolved placement
  (see the placement→caret table in C-2); the old single `EXTRA` hook is gone.
- The caret follows autoFlip: on open, scroll, and resize while open, the shown
  caret switches sides to keep pointing at the trigger, driven by measured rects
  (not written by the user).
- `arrow` off renders the same `MAIN` wrapper with no caret and no measurement
  cost (the `$effect` short-circuits), and the panel positioning styles are
  unchanged from today.
- `data-svs-placement` on the panel reflects the resolved placement when `arrow`
  is on, and is absent when off.
- The new ADR `.ws/decision/001-popover-arrow-caret-split.md` exists with the body
  given in C-5.
- `bun run check`, the scoped Popover test, and the scoped `fmt:check` all pass.

## Background / constraints

- **Why not pure CSS:** `@position-try` fallbacks apply only to the anchored
  element itself and only when *it* overflows; small carets don't overflow in sync
  with the panel (desync), normal-flow children never receive `@position-try`, and
  no CSS selector/inherited property lets a descendant read which fallback the
  panel chose. JS *reading* `getBoundingClientRect()` reliably resolves the
  post-flip side. Positioning itself stays CSS-Anchor-driven; JS only reads the
  result — this is not JS positioning. (This rationale is the ADR in C-5.)
- **Structure vs decoration split (settled):** `panelStyle` already carries
  functional positioning inline (`position-anchor`, `position-area`, gap
  `margin`, `position-try-fallbacks`). The arrow flex layout is the same class of
  *structure derived from the `position` prop* (its `flex-direction` depends on the
  axis, which a user cannot express in static CSS without re-implementing the
  flip logic). So the flex layout stays **component-owned inline** (C-3); only
  **decoration** (border/background/padding/radius/shadow) moves to the user's
  `main` class. This does not violate the "component holds no style" principle —
  that principle is about decoration, and positioning structure has always been
  component-owned.
- **Must keep working:** all existing panel positioning (`position`/`align`/
  `offset`/`matchWidth`/`autoFlip`), trigger wiring, `open` two-way sync,
  `hover`/focus open-close, `manual`, `ariaRole`, `MenuContainerContext`. The
  `arrow: boolean` prop API is unchanged.

## Detailed changes

### (C-1) Panel template — MAIN wrapper + four carets + data attr

**Where:** the `<div bind:this={panel} …>` panel block (currently renders
`{#if arrow}<div class={cls(PARTS.EXTRA, variant)} aria-hidden="true"></div>{/if}`
then `{@render children(variant)}`).

**Required behavior:**
- The panel MUST always render a single wrapper `<div class={cls(PARTS.MAIN, variant)}>`
  containing `{@render children(variant)}` (present whether or not `arrow`).
- When `arrow`, the panel MUST render **exactly one** caret div, chosen by the
  resolved `placement` per this table; each caret is `aria-hidden="true"` and
  classed with its `PARTS.*` part; it MUST be a **sibling** of `MAIN` positioned
  as shown (before `MAIN` for TOP/LEFT, after `MAIN` for BOTTOM/RIGHT):

  | `placement` (side panel emerged) | caret part | position vs MAIN |
  |---|---|---|
  | `bottom` | `TOP` | before MAIN |
  | `top` | `BOTTOM` | after MAIN |
  | `right` | `LEFT` | before MAIN |
  | `left` | `RIGHT` | after MAIN |

- When `arrow` is off, no caret div is rendered.
- The panel MUST set `data-svs-placement={arrow ? placement : undefined}` so the
  attribute is present (resolved side) only when `arrow` is on.

**Suggested implementation:**
```svelte
<div
  bind:this={panel}
  id={pid}
  class={cls(PARTS.WHOLE, variant)}
  popover={manual ? "manual" : "auto"}
  role={ariaRole}
  style={panelStyle}
  data-svs-placement={arrow ? placement : undefined}
  ontoggle={htoggle}
  onpointerleave={hoverEnabled ? panelleave : undefined}
  onfocusout={hoverEnabled ? panelfocusout : undefined}
>
  {#if arrow && placement === "bottom"}<div class={cls(PARTS.TOP, variant)} aria-hidden="true"></div>{/if}
  {#if arrow && placement === "right"}<div class={cls(PARTS.LEFT, variant)} aria-hidden="true"></div>{/if}
  <div class={cls(PARTS.MAIN, variant)}>{@render children(variant)}</div>
  {#if arrow && placement === "top"}<div class={cls(PARTS.BOTTOM, variant)} aria-hidden="true"></div>{/if}
  {#if arrow && placement === "left"}<div class={cls(PARTS.RIGHT, variant)} aria-hidden="true"></div>{/if}
</div>
```

**Notes:** the caret "points at" the trigger — always the opposite side of the
placement the panel emerged toward. Condition-rendering (not display-toggling both)
keeps only one caret in the DOM, avoiding layout shift.

### (C-2) `placement` state + `measure()`

**Where:** the instance `<script>`, near the other `$state` (e.g. by `let panel`).

**Required behavior:**
- Add resolved-side state initialized to the `position` prop:
  `let placement = $state<PopoverPosition>(position);`
- Add a control function `measure()` that reads `panel` and `element`
  `getBoundingClientRect()` and assigns `placement` to the dominant-axis side of
  the panel's center relative to the trigger's center; it MUST no-op when either
  ref is unset.

**Suggested implementation:**
```ts
function measure() {
  if (!panel || !element) return;
  const p = panel.getBoundingClientRect();
  const a = element.getBoundingClientRect();
  const dx = (p.left + p.right - a.left - a.right) / 2;
  const dy = (p.top + p.bottom - a.top - a.bottom) / 2;
  placement = Math.abs(dy) >= Math.abs(dx) ? (dy >= 0 ? "bottom" : "top") : dx >= 0 ? "right" : "left";
}
```
**Notes:** `measure`/`watch` (C-4) are internal control functions → declared as
hoisted `function` (coding-style §8), not `const` arrows. `element` is the
already-bound trigger button; `panel` the already-bound panel div.

### (C-3) `panelStyle` — arrow-only flex layout

**Where:** the `panelStyle` `$derived` array, and a new axis `$derived`.

**Required behavior:**
- When `arrow`, `panelStyle` MUST include a flex layout that lays carets on the
  cross axis centered: `display:flex`, `flex-direction` = `row` for horizontal
  placement (`position` is `left`/`right`) else `column`, and `align-items:center`.
- When `arrow` is off, no flex is added and `panelStyle` is byte-identical to
  today.

**Suggested implementation:**
```ts
const arrowAxis = $derived(position === "left" || position === "right" ? "row" : "column");
// append to the panelStyle array (arrow only):
arrow ? `display:flex;flex-direction:${arrowAxis};align-items:center` : "",
```
**Notes:** the existing `.filter(Boolean).join(";")` drops the `""` when `arrow`
is off, keeping the off-path unchanged.

### (C-4) Measurement `$effect` (coding-style §5)

**Where:** the instance `<script>` reactive section, alongside the existing
open/close `$effect`.

**Required behavior:**
- A `$effect` MUST re-run **only** on `arrow` / `open` changes; its `$effect`
  body's top level MUST be just those trigger reads plus a single
  `untrack(() => …)` call — the actual work (guard, initial `measure()`, listener
  wiring, teardown) lives in an extracted named function, so the reactive surface
  is obvious (coding-style §5). Do not read `panel`/`element`/`placement`
  reactively.
- While `arrow` && `open`: MUST `measure()` once immediately, then keep `placement`
  current by re-measuring on `scroll` (capture + passive) and `resize` (passive),
  rAF-throttled. When `arrow` or `open` goes false (or on teardown) it MUST cancel
  any pending rAF and remove both listeners.

**Suggested implementation:**
```ts
$effect(() => {
  arrow;
  open;
  return untrack(() => watch());
});

function watch() {
  if (!arrow || !open) return;
  measure();
  let raf = 0;
  const onmove = () => {
    if (raf) return;
    raf = requestAnimationFrame(() => {
      raf = 0;
      measure();
    });
  };
  window.addEventListener("scroll", onmove, { capture: true, passive: true });
  window.addEventListener("resize", onmove, { passive: true });
  return () => {
    if (raf) cancelAnimationFrame(raf);
    window.removeEventListener("scroll", onmove, { capture: true });
    window.removeEventListener("resize", onmove);
  };
}
```
**Notes:** `untrack(() => watch())` passes `watch()`'s returned teardown straight
through as the effect cleanup. `watch` re-reads `arrow`/`open` inside `untrack`
for its guard; that adds no reactivity because they are already the declared
triggers outside `untrack`.

**Risk:** getting the reactive triggers wrong (e.g. reading `panel` at top level)
would re-subscribe on every bind. Keep the top level to `arrow; open;` + `untrack`.

### (C-5) ADR

**Where:** create new file `.ws/decision/001-popover-arrow-caret-split.md`
(this path does not exist yet — it is a deliverable of this task).

**Required content (exact body):**
```md
# 001. Popover arrow via structural caret split + JS rect measurement

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-08
**Affects:** Popover public styling contract (parts `whole`/`main`/`top`/`bottom`/`left`/`right`, `data-svs-placement`); no prop-API change

## Context
Popover's `arrow` must render a speech-bubble caret that sticks out past the panel
frame and follows `autoFlip` reversal. Pure CSS cannot follow the flip: CSS Anchor
`@position-try` fallbacks apply only to the anchored element itself and only when
it overflows, so small carets desync from the panel; normal-flow children never
receive `@position-try`; and no CSS selector or inherited property lets a
descendant read which fallback the panel chose. The prior single `EXTRA` hook
could not point at the trigger after a flip.

## Decision
Split the arrow panel into a decorated `MAIN` body and sibling carets
(`TOP`/`BOTTOM`/`LEFT`/`RIGHT`), rendering only the one caret that points at the
trigger. CSS Anchor still owns positioning; a small internal `$effect` reads
`panel`/trigger `getBoundingClientRect()` on open/scroll/resize to determine the
resolved side and pick the caret. Decoration moves from the `whole` part to `main`;
the resolved side is exposed as `data-svs-placement`. Pre-release permits this
breaking styling-contract change.

## Consequences
- The caret follows flips with no user-written logic; users only style static
  per-direction carets and put decoration on `main`.
- Breaking: existing `arrow` users move decoration `whole`→`main` and replace
  `extra` with `top`/`bottom`(/`left`/`right`); non-arrow panels gain one `main`
  wrapper level.
- A minimal JS read path (rect measurement) is introduced; it runs only while
  `arrow` && `open`, rAF-throttled. This is result-reading, not JS positioning.
- `data-svs-placement` becomes part of the public contract.

## Rejected alternatives
- **Pure CSS `@position-try` per caret** — fails: carets don't overflow in sync
  with the panel (desync), and descendants can't read the chosen fallback.
- **Single caret with negative margin + dynamic `rotate`** — the prior shape;
  brittle, needs JS to flip the rotation anyway, and can't sit outside the frame
  cleanly.
- **Full JS positioning** — abandons the standards-first CSS Anchor approach and
  its reduced-motion/perf benefits for no gain; JS is limited to reading the
  resolved side.
```

### (C-6) `@component` comment

**Where:** the leading `@component` comment (Anatomy and Behavior), and the
`arrow` line in both the doc-comment `PopoverProps` and the exported interface.

**Required behavior:**
- Anatomy MUST show the new structure: `main` wrapper always present, and the four
  carets as conditional siblings (coding-style §11 uses the `conditional:` marker).
- The `arrow` prop comment MUST describe it as rendering a per-direction caret part
  (`top`/`bottom`/`left`/`right`) chosen by resolved placement, and mention
  `data-svs-placement`. Behavior MUST note the flip-following measurement.

**Suggested Anatomy:**
```svelte
<button class="label" {...rest} popovertarget aria-haspopup aria-expanded aria-controls style>
  {label}
</button>

  <div class="top" aria-hidden conditional: arrow, placement=bottom></div>
  <div class="left" aria-hidden conditional: arrow, placement=right></div>
  <div class="main">{children}</div>
  <div class="bottom" aria-hidden conditional: arrow, placement=top></div>
  <div class="right" aria-hidden conditional: arrow, placement=left></div>
</div>
```

## Invariants / expected behavior

- `arrow` off: DOM equals today plus a single `main` wrapper; `panelStyle`
