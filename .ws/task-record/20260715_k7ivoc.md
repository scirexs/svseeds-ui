- `.ws/decision/018-dateinput-overlay-default-z-index-via-cssvar.md` — the accepted

## Goal

`ComboBox` renders its listbox (`PARTS.BOTTOM`) as `position:absolute` inside a
`PARTS.WHOLE` `<span>` that is `position:relative`. Two defects follow:

1. **No `z-index`** — the listbox paints in the positioned layer at `auto`, so any
   later positioned sibling covers it. ADR 018 already decided the fix for the
   identical `DateInput` case; `ComboBox` must adopt the same shape.
2. **The wrapper is an inline box, so percentage anchors resolve against the wrong
   height.** A `position:relative` **inline** `<span>` forms a containing block whose
   height comes from font metrics (measured: **17px**), not from the `<input>` it
   wraps (measured: **32px**). Every percentage y-anchor is therefore wrong.

Give `ComboBox` a `cssvar` prop (`x` / `y` / `z`) matching `DateInput`, and fix the
containing block in **both** components so those anchors resolve against the control's
box. This is a **breaking change** and is accepted (see *Accepted risks*).

## Background / constraints

Measured in real chromium against the exact `<span style="position:relative">` +
`<input style="height:32px">` structure both components render:

| wrapper | wrapper box | `top:100%` vs input bottom | `bottom:100%` vs input top | x-axis |
| --- | --- | --- | --- | --- |
| inline `<span>` (**today**) | 183x**17** | **-8.5px (overlaps)** | **-6.5px (overlaps)** | 0.0 ✓ |
| `display:inline-block` | 183x**32** | **0.0 ✓** | **0.0 ✓** | 0.0 ✓ |

Read from this:

- The **x-axis is already correct** — an inline span's bounding box width does hug the
  input, so `left:0%` / `right:0%` resolve correctly. Only the y-axis is broken.
- **`ComboBox`'s flipped anchor is already broken today** (`bottom:100%` overlaps the
  input by 6.5px). This task fixes a live defect, not only a hypothetical one.
- **`ComboBox`'s default (non-flipped) anchor is accidentally correct today** because it
  sets *no* `top`, leaving the listbox at its static position. Moving it to
  `top:var(…,100%)` **without** the wrapper fix would regress it by 8.5px — so C-1 and
  C-2 must land together.
- **`DateInput` is broken on both anchors today** (it already emits `top:var(…,100%)`).
- `display:inline-block` **does not change the wrapper's width** (183px in both rows), so
  the layout footprint is essentially unchanged; the wrapper's height simply becomes the
  control's height instead of the font's.

What must keep working:

- All existing `ComboBox` behavior: focus/blur open+close, keyboard navigation, option
  apply, `search` filtering, context embedding (`ComboBoxContext`, used by `Pagination`),
  `styling` / `variant` / class merging, and ARIA wiring.
- All existing `DateInput` behavior. `DateInput`'s only product change is C-3.
- `Pagination` embeds `ComboBox` and passes a `comboBox` bag typed
  `Omit<ComboBoxProps, ComboBoxReqdProps | ComboBoxBindProps | "options" | "variant" | "styling">`.
  Adding `cssvar` to `ComboBoxProps` flows through automatically — **`Pagination.svelte`
  needs no change**. Do not edit it.

## Detailed changes

### C-1 — `ComboBox`: wrapper establishes the anchor's containing block

**Where:** `ComboBox.svelte`, the `PARTS.WHOLE` `<span>` in the template (the element
carrying `style="position:relative;"`).

**Required behavior:** The wrapper MUST render `display:inline-block` alongside
`position:relative` in its component-owned inline `style` attribute. It MUST NOT be
routed through `styling` / a class / a `<style>` block — `position:relative` is already
imposed inline here, and the display value is part of the same anchoring mechanism the
component owns.

**Suggested implementation** (illustrative):

```svelte
<span class={cls(PARTS.WHOLE, effVariant)} style="position:relative;display:inline-block;">
```

**Notes:** `<input>` is itself `inline-block`, so the wrapper now matches the box it
wraps — this is the least-surprise default.

**Risk:** Callers who relied on the wrapper being a plain inline box may see baseline /
vertical-align shift slightly. Accepted (see *Accepted risks*).

### C-2 — `ComboBox`: `cssvar` bridge and the aligned anchor style

**Where:** `ComboBox.svelte` — the `<script module>` exports, the `$props()` destructure,
the *Reactive Handlers* section (`listboxStyle`), and the `_core` import.

**Required behavior:**

- `ComboBoxProps` MUST gain
  `cssvar?: Partial<Record<ComboBoxCssVar, string>>;`, placed with the other optional
  props and following §12 (required props first).
- The module MUST export `export type ComboBoxCssVar = "x" | "y" | "z";`.
- The names MUST resolve through `_cssVar` (imported from `./_core`) with these defaults —
  identical to `DateInput`, so one caller-side `--svs-position-*` definition governs both
  components:
  - `x` → `--svs-position-x`
  - `y` → `--svs-position-y`
  - `z` → `--svs-position-z`
- `listboxStyle` MUST anchor per axis exactly as `DateInput` does: the y-axis via
  `bottom` when `overflow.y` else `top`, defaulting to `100%`; the x-axis via `right` when
  `overflow.x` else `left`, defaulting to `0%`.
- The listbox MUST carry `z-index:var(<z name>,1)`. The `1` fallback is fixed by ADR 018;
  do not choose a different value.
- The existing `cursor:default`, `user-select:none`, and the `visibility` toggle driven by
  `expanded` MUST be preserved.

**Suggested implementation** (illustrative):

```ts
const xVar = $derived(_cssVar(cssvar, "x", "--svs-position-x"));
const yVar = $derived(_cssVar(cssvar, "y", "--svs-position-y"));
const zVar = $derived(_cssVar(cssvar, "z", "--svs-position-z"));
const listboxStyle = $derived(
  `position:absolute;cursor:default;user-select:none;visibility:${expanded ? "visible" : "hidden"};${overflow.y ? "bottom" : "top"}:var(${yVar},100%);${overflow.x ? "right" : "left"}:var(${xVar},0%);z-index:var(${zVar},1);`,
);
```

**Notes:**

- `cssvar` goes in the `// prettier-ignore` one-line `$props()` destructure with the rest.
- `_cssVar` is a value import from `./_core` — add it to the existing value-import block
  (§1), which already imports `_detectOverflow` / `_fnClass` / `_createContext`.
- Today's `visibility: ${…}` carries a space after the colon; dropping it is fine and
  matches `DateInput`. Do not let tests depend on either spelling — see *Tests*.

**Risk:** The default anchor moves from "static position" to an explicit `top:100%`.
Correct only once C-1 lands; the two must not be separated.

### C-3 — `DateInput`: same wrapper fix

**Where:** `DateInput.svelte`, the `PARTS.WHOLE` `<span>` (the element carrying
`bind:this={root}` … `style="position:relative;"`).

**Required behavior:** identical to C-1 — the inline style MUST become
`position:relative;display:inline-block;`. No other `DateInput` product change is in
scope: its `cssvar`, `overlayStyle`, and `data-svs-flip-*` are already correct.

**Risk:** Same as C-1. This visibly moves `DateInput`'s calendar down by ~8.5px into its
intended position; that is the fix, not a regression.

### C-4 — `ComboBox`: flip state as data attributes

**Where:** `ComboBox.svelte`, the `PARTS.BOTTOM` `<ul>` in the template.

**Required behavior:** The listbox MUST expose `data-svs-flip-x` when `overflow.x` and
`data-svs-flip-y` when `overflow.y`, and MUST omit each attribute otherwise (`undefined`,
not an empty-string attribute that is always present). Match `DateInput`'s spelling.

**Suggested implementation** (illustrative):

```svelte
data-svs-flip-x={overflow.x ? "" : undefined}
data-svs-flip-y={overflow.y ? "" : undefined}
```

### C-7 — ADR 019

**Where:** new file `.ws/decision/019-overlay-anchor-inline-block-wrapper.md`.

**Required behavior:** Create it with **exactly** the body below (exact). It does not
supersede ADR 018 — it applies 018's z-index decision to `ComboBox` and adds a separate
decision about the wrapper's containing block — so ADR 018 MUST NOT be edited.

```md
# 019. Overlay anchors resolve against an inline-block wrapper

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-15
**Affects:** ComboBox public API (`ComboBoxCssVar`, `cssvar`, `data-svs-flip-*`); ComboBox and DateInput `whole` wrapper display; the library's convention for absolute-positioned overlays

## Context

`ComboBox` and `DateInput` are the library's only two components that anchor an
absolutely-positioned overlay inside a `position:relative` wrapper (`PARTS.WHOLE`, a
`<span>`). Both flip the overlay per axis on viewport overflow via `_detectOverflow`.

Applying ADR 018's z-index decision to `ComboBox` required also deciding whether
`ComboBox` should expose the `cssvar` x/y offset keys `DateInput` already has. Measuring
the shared structure in chromium showed the offsets were resting on a broken foundation.

A `position:relative` **inline** `<span>` forms a containing block whose height comes from
font metrics, not from the `<input>` it wraps: measured 17px against a 32px input. Every
percentage y-anchor therefore resolves against the wrong basis:

- `top:100%` lands 8.5px *above* the input's bottom edge — overlapping the control.
- `bottom:100%` lands 6.5px *below* the input's top edge — also overlapping.
- The x-axis is unaffected: the inline box's width does hug the input, so `left:0%` /
  `right:0%` are correct.

This made `DateInput` wrong on both y-anchors, and `ComboBox` wrong on its flipped
anchor. `ComboBox`'s default anchor was accidentally correct only because it set no
`top` at all, leaving the listbox at its static position — which is also why it could not
offer a meaningful y-offset hook: with no explicit anchor, a caller setting the offset
property would jump the listbox to the wrapper's font-height box.

## Decision

Both components render `position:relative;display:inline-block;` on the `whole` wrapper,
as a component-owned inline style. The wrapper's box then coincides with the control it
wraps, so all four percentage anchors resolve against the control's box.

On that foundation, `ComboBox` adopts `DateInput`'s overlay shape wholesale: a `cssvar`
prop with `ComboBoxCssVar = "x" | "y" | "z"` resolved via `_cssVar` against the same
default names (`--svs-position-x` / `--svs-position-y` / `--svs-position-z`), the same
per-axis anchor expression, `z-index:var(--svs-position-z,1)` per ADR 018, and the same
`data-svs-flip-x` / `data-svs-flip-y` attributes.

Sharing the default property names across both components is deliberate: one
`--svs-position-z` definition in caller CSS governs every overlay in the library.

The display value is imposed inline rather than left to `styling`, matching
`position:relative`, which is already imposed there. Both are structural styles that the
anchoring mechanism owns; neither is an opinionated visual style, so this does not breach
the headless principle. A caller who overrides `display` on the wrapper takes ownership of
the resulting anchor geometry.

## Consequences

- Both overlays land where their anchors say, with no caller CSS.
- `ComboBox`'s x/y offset hooks are meaningful for the first time, because there is now an
  explicit anchor to offset from.
- Breaking: the `whole` wrapper is no longer a plain inline box. Its width is unchanged
  (measured 183px before and after), but its height now matches the control rather than
  the font, so baseline / `vertical-align` interactions can shift.
- Breaking: `DateInput`'s calendar and `ComboBox`'s flipped listbox move by ~8.5px /
  ~6.5px into their intended positions. Callers who compensated for the old overlap with
  their own offsets will now over-correct.
- The wrapper's `display` is now effectively public API; callers may override it, but own
  the anchor geometry when they do.
- `ComboBoxCssVar`, the `--svs-position-*` default names, the `1` z-index fallback, and
  the `data-svs-flip-*` attributes are public API for `ComboBox` from now on.
- Any future overlay component follows this same shape.

## Rejected alternatives

- **Add only the `z` key to `ComboBox` and leave positioning alone.** The smallest change
  that satisfies the original request, and it carries no visual risk. Rejected: it leaves
  `ComboBox`'s flipped anchor overlapping the input by 6.5px and `DateInput` wrong on both
  anchors, and it permanently blocks a meaningful y-offset hook, since an offset only has
  meaning relative to an explicit anchor. It also leaves the two structurally identical
  components with divergent implementations.
- **Expose x/y/z on `ComboBox` while keeping the inline wrapper.** Rejected: it regresses
  `ComboBox`'s default anchor by 8.5px — the one case that works today — in exchange for
  hooks that resolve against the font-height box.
- **Expose x/y with an `auto` fallback** (`top:var(--svs-position-y,auto)`), preserving
  today's static-position default. Rejected: it keeps the default correct but makes the
  hook a trap — a caller setting the property gets an offset measured from the wrapper's
  17px box, landing on top of the control. A hook that is wrong whenever it is used is
  worse than no hook.
- **Route the wrapper's `display` through a `cssvar` key or a `<style>` block** so callers
  can override it without inline-style specificity. Rejected as disproportionate:
  `position:relative` is already imposed inline with the same override cost, and coding-style
  §13.1 reserves the `cssvar` bridge for values the caller is expected to tune, not for
  structural styles the mechanism depends on.
- **Move the overlays to the top layer via the native Popover API.** Rejected for the same
  reason as in ADR 018: it would rework the entire positioning, flip, and transition
  surface. Still worth revisiting as its own task.
```

## Invariants / expected behavior

- Default (no overflow, no caller CSS): the overlay's top edge sits flush with the
  control's bottom edge — **zero gap, zero overlap** — and its left edge aligns with the
  control's left edge, in both components.
- Flipped y (`overflow.y`): the overlay's bottom edge sits flush with the control's top
  edge.
- Flipped x (`overflow.x`): the overlay's right edge aligns with the control's right edge.
- The overlay paints above a following sibling's positioned content by virtue of
  `z-index:1`.
- Caller CSS setting `--svs-position-z` retargets both components' overlays; a caller
  passing `cssvar={{ z: "--my-z" }}` retargets only that instance and the default name
  no longer appears in the emitted style.

## Out of scope

- `Pagination.svelte` — no change needed (see *Background / constraints*).
- Any other component; only `ComboBox` and `DateInput` render this wrapper/overlay pair
  (verified by grepping `style="position:relative` across `src/lib/_svseeds/`).
- ADR 018 — must not be edited.
- `DateInput` beyond C-3.
- Exposing x/y offset hooks for the `PARTS.EXTRA` element, or any other positioning rework.
- Version bump / release; the maintainer owns that.
- Unrelated refactors of `ComboBox`'s open/close, filtering, or keyboard logic.

## Accepted risks / tradeoffs

- **Breaking change, accepted by the maintainer.** The wrapper stops being a plain inline
  box, and both overlays shift by ~6.5–8.5px into their intended positions. Callers who
  hand-compensated for the old overlap will over-correct. This is the point of the change.
- The wrapper's `display` becomes public API in practice; a caller overriding it owns the
  anchor geometry. Documented in *Behavior* (C-5) and ADR 019.
- `1` as the z-index fallback and the `--svs-position-*` names are public API (already
  accepted in ADR 018; `ComboBox` now shares them).

## Tests

Both files are browser (`client`) specs — keep them there (§16). Add to
`tests/ComboBox.svelte.test.ts`, mirroring the `_DateInput overlay overflow flip` describe
block's local `overlay` / `overlayStyle` helpers:

- **Default anchor (C-2):** with the listbox expanded, the emitted `style` matches
  `top:\s*var\(--svs-position-y` and `left:\s*var\(--svs-position-x`, and emits no
  `bottom:` / `right:` anchor.
- **Default z-index (C-2):** the style matches `z-index:\s*var\(--svs-position-z\s*,\s*1\)`.
- **Renaming via `cssvar` (C-2):** `cssvar={{ z: "--my-z" }}` yields `z-index:\s*var\(--my-z`
  and the style no longer contains `--svs-position-z`. Cover an x or y rename the same way.
- **Flip attributes (C-4):** `data-svs-flip-x` / `data-svs-flip-y` are absent by default.
- **Preserved behavior (C-2):** the expanded/collapsed `visibility` toggle still works —
  assert via `getComputedStyle` (the existing `style()` helper), **not** by string-matching
  the `style` attribute, so the assertion is independent of the `visibility: ` /
  `visibility:` spacing change.
- **Geometry (C-1) — the regression guard this task exists for:** with the listbox
  expanded and no caller styling, assert from `getBoundingClientRect()` that the listbox's
  `top` equals the input's `bottom` within a small tolerance (≈1px). This test **fails on
  `dev` today by ~8.5px** once C-2 lands without C-1, and is what proves C-1 works.
  A sibling geometry test for `DateInput` (overlay `top` ≈ input `bottom`) is REQUIRED —
  it fails on `dev` today and passes after C-3.

Do not assert on a serialized `style` attribute where the value is read back from the DOM
without accounting for re-serialization — a real browser re-adds a space after each colon
(`.ws/knowledge/browser-style-attribute-serializes-with-spaces.md`). The regex forms above
already tolerate this; keep any new assertion equally tolerant.

Existing `ComboBox` and `DateInput` tests MUST keep passing unchanged. If an existing
assertion breaks purely because of the anchor/`visibility` string change, update that
assertion — do not weaken the behavior it covers.
