## Goal

Let callers write `<DateInput><Calendar … /></DateInput>` (children prop), not only configure the internal Calendar through the `calendar` bag — the same coordinator convention already used by `DateField → DateInput` and `Calendar → MonthPicker`. This completes the date-picker composition family so `<DateField><DateInput><Calendar><MonthPicker/></Calendar></DateInput></DateField>` nests with no special mechanism. Calendar becomes a dual-mode child (reads an optional `CalendarContext`, falls back to its own props standalone); DateInput sets that context and renders `children` in its overlay in place of the default internal Calendar.

## Done when

- Calendar exports `CalendarContext`, `_getCalendarContext`, `_setCalendarContext`; embedded in a context it sources `value`/`min`/`max`/`isDisabled`/`variant`/`styling` from it and routes value writes back through it; standalone behavior is unchanged.
- DateInput accepts `children?: Snippet`; when present it renders in the overlay instead of the default `<Calendar>`, and the child Calendar self-wires (picking a date sets DateInput's value, emits change, and honors `closeOnSelect`) exactly as the internal one does.
- The `calendar` bag still configures the default (no-children) internal Calendar; when `children` is supplied the bag is ignored (children wins).
- `bun run check`, the scoped tests, and scoped `fmt:check` (below) all pass.
- The architecture overview lists `DateInput over Calendar` as a coordinator pairing.

## Background / constraints

- **What must keep working:** every current standalone Calendar behavior (paging, picking, keyboard roving, day selection, constraints, page transitions, the embedded MonthPicker via `MonthPickerContext`) and every current DateInput behavior (readonly vs `parse`, hidden `name` input, overflow-aware overlay placement, Escape/outside dismissal, focus return, `openOnFocus`/`closeOnSelect`, the `DateInputContext` consumer path used by DateField). The context path is **purely additive**: `_getCalendarContext()` returns `undefined` standalone, so the props path is untouched.
- `Temporal` is a global in this codebase (no import); keep using it directly.
- Calendar keeps `display` and `picking` as its own bindable props in **both** modes — the context does **not** carry them (mirrors MonthPicker, whose context carries `value`/`min`/`max`/`variant`/`styling` only). `locale` likewise stays a presentational prop, **not** in the context.

## Detailed changes

### (C-1)/(C-2) Calendar dual-mode child

**Where:** `Calendar.svelte`, `<script module>` (after `CalendarBindProps`, before `_CALENDAR_PRESET`) and the instance `<script>` Initialize/States/logic/template.

**Required behavior:**

- Calendar MUST define, in `<script module>`, a context contract extending `SVSContext` and its accessor pair, and export all three:
  ```ts
  export interface CalendarContext extends SVSContext {
    get value(): Temporal.PlainDate | undefined;
    set value(v: Temporal.PlainDate | undefined);
    get min(): Temporal.PlainDate | undefined;
    get max(): Temporal.PlainDate | undefined;
    get isDisabled(): ((d: Temporal.PlainDate) => boolean) | undefined;
  }
  export const [_getCalendarContext, _setCalendarContext] = _createContext<CalendarContext>();
  ```
  Add `_createContext` to the existing `./_core` value import and `SVSContext` to the type import.
- The instance script MUST read the context once: `const ctx = _getCalendarContext();` and derive effective inputs — `effValue`, `effMin`, `effMax`, `effIsDisabled`, `effVariant` — each `$derived(ctx ? ctx.X : X)` (mirror MonthPicker's `effValue`/`effMin`/`effMax`/`effVariant`).
- The class fn MUST fall back to context styling: `_fnClass(_CALENDAR_PRESET, styling ?? ctx?.styling)`.
- Every read of `value` in logic (`syncFocus`, `selected`, the initial `focused` seed) MUST use `effValue`; the `display`/`focused` seeds (currently `value ?? …`) MUST seed from `ctx?.value ?? value ?? Temporal.Now.plainDateISO()` so an embedded initial value is honored (mirror MonthPicker's `initial`).
- Every `value = …` write (`select`, `clear`) MUST go through a `setValue(v)` helper: `if (ctx) ctx.value = v; else value = v;`.
- `min`/`max` reads (`inRange`, the `ctxMin`/`ctxMax` deriveds) MUST use `effMin`/`effMax`; `isDisabled` (`disabled`) MUST use `effIsDisabled`.
- Every `variant` read that feeds a class or a snippet argument MUST use `effVariant`: the `ctxVariant` derived, all `cls(PARTS.*, variant)` in the template, and the `variant` passed to the `left`/`label`/`right`/`weekday`/`bottom` snippets. Literal `VARIANT.NEUTRAL` uses stay literal; `variantOf` (which returns `VARIANT.*` constants) is unchanged.
- The existing `mpCtx` object MUST expose the effective values to the embedded MonthPicker: its `min`/`max` getters return `effMin`/`effMax` (via the renamed `ctxMin`/`ctxMax` deriveds) and its `variant` getter returns `effVariant`. Its `value` get/set (display-based) is unchanged.

**Suggested implementation:** keep `ctxMin`/`ctxMax` as the `PlainYearMonth` deriveds but base them on `effMin`/`effMax` (`effMin?.toPlainYearMonth()`); replace `ctxVariant` with `effVariant`. Place `ctx`/`eff*` in the Initialize banner alongside the existing `cls`/`uid`, before `mpCtx`.

**Notes:** `display` and `picking` stay owned by Calendar (bindable) in both modes — do not add them to the context. Declaration order (§5): `eff*` deriveds read by `mpCtx` getters must be declared before `mpCtx`; getters are lazy so a `$derived` declared after is tolerable, but declare `eff*` first to match MonthPicker.

**Risk:** wide but mechanical rename surface; missing one `variant`→`effVariant` in the template leaves an embedded Calendar visually out of sync with its coordinator. The embedded-mode test (T-1) guards this.

### (D-1)/(D-2) DateInput coordinator

**Where:** `DateInput.svelte`, `DateInputProps` interface, imports, instance script Initialize, and the `{#if open}` overlay.

**Required behavior:**

- Add `children?: Snippet` to `DateInputProps` (place it near `left`/`right`/`transition`; keep required-first ordering — it is optional so it goes among the optionals). Destructure `children` in `$props()`.
- Import the coordinator context from Calendar: add `_setCalendarContext` to the `import Calendar` value import and `CalendarContext` to the Calendar type import.
- Build a `CalendarContext` and set it at init (Initialize banner), mirroring DateField's `ctx` object + `_setDateInputContext`:
  ```ts
  const calCtx: CalendarContext = {
    get value() { return effValue; },
    set value(v) { onPick(v); },
    get min() { return min; },
    get max() { return max; },
    get isDisabled() { return isDisabled; },
    get variant() { return effVariant; },
    get styling() { return undefined; },
  };
  _setCalendarContext(calCtx);
  ```
  `onPick`/`effValue`/`effVariant` already exist; getters are lazy so declaring `calCtx` in Initialize (before `onPick` is defined lower) is fine — the getter body runs only when the embedded Calendar reads it. `styling` returns `undefined` so the child Calendar keeps its own preset (matches Calendar's `mpCtx.styling`).
- In the overlay `<div id={idOverlay} …>`, render children in place of the default Calendar:
  ```svelte
  {#if children}
    {@render children()}
  {:else}
    <Calendar {...calendar} bind:value={() => effValue, onPick} locale={calendar?.locale ?? locale} {min} {max} {isDisabled} variant={effVariant} />
  {/if}
  ```
  The default branch is exactly today's `<Calendar>` invocation, unchanged.

**Notes:** leave the `calendar` bag type (`Omit<CalendarProps, "value" | "display" | "min" | "max" | "isDisabled" | "variant">`) unchanged — it configures only the default internal Calendar; do not add `children` to that Omit (a caller wanting a custom child writes DateInput's own `children`). `focusCalendar()` queries `[tabindex="0"]` inside `overlayElem`, which a caller-written Calendar also renders, so focus-on-open keeps working.

**Risk:** if the context getters read raw `value`/`variant` instead of `effValue`/`effVariant`, an embedded DateInput (inside DateField) would mis-wire its child Calendar. The children group (T-1) and existing DateField tests cover both.

**@component (D-2):** Update *Usage* to show `<DateInput><Calendar … /></DateInput>` alongside the standalone example; add a *Behavior* line that a declarative `Calendar` child self-wires value/min/max/isDisabled/variant through context and wins over the `calendar` bag. Keep the existing readonly/`parse`/`name`/`format` behavior lines.
