# Task: Restructure NumberInput's spinner props into a `spin` enum and rename the button snippets

Targets to edit:
- `src/lib/_svseeds/NumberInput.svelte` (props, logic, template, `@component` block)
- `tests/NumberInput.svelte.test.ts`
- `tests/NumberField.svelte.test.ts`

References (read before editing):
- `src/lib/_svseeds/NumberField.svelte` — wraps `NumberInput`; passes its bag through
  `numberInput?: Omit<NumberInputProps, …>`. Confirm it needs **no source change** (see Out of scope).
- `.ws/policy/coding-style.md` — §2 `$props()` one-line destructure, §11 `@component`
  (Types / Anatomy), §14 component shape.

## Goal

Collapse the two boolean spinner flags (`spin`, `stack`) into a single tri-state enum
`spin?: "none" | "split" | "stack"`, and rename the two button snippets `decrement` → `left`
and `increment` → `right` so they match the existing `PARTS.LEFT` / `PARTS.RIGHT` class parts.
This removes the meaningless `spin=false, stack=true` combination and aligns the snippet names
with the styling vocabulary already in use.

Additionally, simplify the spinner button markup to match `DarkToggle`: drop the inner
`PARTS.EXTRA` `<span>` wrapper so the snippet / default glyph renders directly inside the
button, and size the default `<svg>` with `style="width:100%;height:100%;"` so its size follows
the `PARTS.LEFT` / `PARTS.RIGHT` button styling instead of a hardcoded `10×10`.

## Done when

- `NumberInput` no longer accepts `stack`, `decrement`, or `increment` props; it accepts
  `spin?: "none" | "split" | "stack"` (default `"none"`), `left?: Snippet<[string]>`, and
  `right?: Snippet<[string]>`.
- The three layouts render as before: `"none"` = no spinner buttons; `"split"` = decrement
  button before the input and increment button after it; `"stack"` = both buttons inside the
  `PARTS.AUX` span after the input.
- `role="spinbutton"` and the `aria-value*` attributes are present whenever `spin !== "none"`
  and absent when `spin === "none"`.
- Each spinner button contains its snippet / default glyph directly (no `PARTS.EXTRA` `<span>`),
  and the default `<svg>` carries `style="width:100%;height:100%;"` (no fixed `width`/`height`).
- `bun run check`, the two scoped tests, and scoped `fmt:check` all pass.

## Background / constraints

- `left` = decrement, `right` = increment — the names denote the **semantic role** (tied to the
  `PARTS.LEFT` / `PARTS.RIGHT` classes), **not** a physical position: in `"stack"` layout the
  buttons are visually top/bottom, yet `left` is still decrement and `right` still increment.
- This is a clean breaking change — no deprecation shims, no backward-compatible aliases.
- `_core.ts` needs no change.
- What must keep working: all current spinner behavior (pointer press-hold repeat, ArrowUp/Down
  keyboard bump, bound-disabling of the buttons, `aligned max` reporting, datalist/options,
  context wiring with `NumberField`). Only the prop **surface** changes, not the mechanics.

## Scope of changes

- (C-1) NumberInput props: replace `spin: boolean` + `stack: boolean` with `spin: "none" | "split" | "stack"`; rename snippet props `decrement`→`left`, `increment`→`right`.
- (C-2) NumberInput logic & template: rewrite every `spin` / `stack` branch against the enum, and the snippet renders against the new names.
- (C-3) NumberInput `@component` comment: sync the Types and Anatomy blocks (enum, rename, and the removed `EXTRA` wrapper).
- (C-4) Tests: update `tests/NumberInput.svelte.test.ts` and `tests/NumberField.svelte.test.ts` to the new prop surface.
- (C-5) NumberInput spinner buttons: drop the `PARTS.EXTRA` `<span>` wrapper and size the default `<svg>` via `style="width:100%;height:100%;"`.

## Detailed changes

### (C-1) Props — `<script module>` interface and `$props()` destructure

**Where:** `NumberInputProps` in `<script module>`, and the one-line `$props()` destructure.

**Required behavior:**
- The `NumberInputProps` interface MUST drop the `stack` property, MUST replace
  `spin?: boolean` with `spin?: "none" | "split" | "stack";`, and MUST rename
  `decrement?: Snippet<[string]>` → `left?: Snippet<[string]>` and
  `increment?: Snippet<[string]>` → `right?: Snippet<[string]>`. Keep the trailing
  `// Snippet<[variant]>` comments; update the `spin` comment to `// ("none")`.
- The `$props()` destructure MUST default `spin` to `"none"`, MUST remove `stack`, and MUST
  destructure `left, right` in place of `decrement, increment`.

**Suggested implementation** (interface):
```ts
spin?: "none" | "split" | "stack"; // ("none")
// (stack line removed)
left?: Snippet<[string]>;  // Snippet<[variant]>
right?: Snippet<[string]>; // Snippet<[variant]>
```
Destructure fragment: `… spin = "none", options, … left, right, onchange: onchangeProp …`
(replacing `spin = false, stack = false, …, decrement, increment`).

**Notes:** Keep required-before-optional order (§12) — all these are optional, so relative order
is unchanged. `NumberInputReqdProps` / `NumberInputBindProps` are unaffected. The enum union is
inlined in the interface; do **not** add a separately exported type alias.

**Risk:** `NumberField`'s `numberInput` bag is `Omit<NumberInputProps, …>`, so the new surface
flows through automatically — no NumberField source edit (verify at C-4/Out of scope).

### (C-2) Logic & template — enum branches and snippet renders

**Where:** `hkeydown`, the `<span class={cls(PARTS.WHOLE …)}>` template block, and the
`decBtn` / `incBtn` snippet bodies in `NumberInput.svelte`.

**Required behavior:** every place that currently tests `spin` (truthy) MUST test
`spin !== "none"`, and every place that distinguishes the two layouts MUST test the specific
enum value. Specifically:
- `hkeydown` guard `if (!spin || ev.isComposing) return;` → `if (spin === "none" || ev.isComposing) return;`.
- Pre-input decrement render `{#if spin && !stack}` → `{#if spin === "split"}`.
- `<input>` attributes: `role={spin ? … }`, `aria-valuenow={spin ? … }`,
  `aria-valuemin={spin ? … }`, `aria-valuemax={spin && max !== undefined ? … }` MUST each use
  `spin !== "none"` in place of the truthy `spin` test.
- Post-input block `{#if spin && stack} … {:else if spin} …` → `{#if spin === "stack"} … {:else if spin === "split"} …`.
- The `decBtn` snippet MUST render `left` (was `decrement`); the `incBtn` snippet MUST render
  `right` (was `increment`) — i.e. `{#if left}{@render left(effVariant)}{:else}…svg…{/if}` and
  the mirror for `right`. Keep the fallback `<svg>` glyph **paths**, `PARTS.LEFT`/`PARTS.RIGHT`
  classes, `aria-label`s, and `hdown`/`hup` wiring unchanged. (The surrounding `PARTS.EXTRA`
  `<span>` wrapper and the `<svg>` sizing change per C-5.)

**Notes:** The `decBtn` / `incBtn` snippet identifiers and the `PARTS.LEFT` / `PARTS.RIGHT`
classes stay as-is; only the *prop* the snippet body checks/renders is renamed.

**Risk:** miss one truthy `spin` test → aria attributes leak or disappear in a layout; the tests
in C-4 cover each branch.

### (C-3) `@component` comment — Types & Anatomy

**Where:** the leading `@component` block of `NumberInput.svelte`.

**Required behavior:**
- The Types block MUST mirror C-1 exactly (`spin?: "none" | "split" | "stack"; // ("none")`,
  `stack` line removed, `left?` / `right?` replacing `decrement?` / `increment?`).
- The Anatomy block MUST update the button `conditional:` triggers to the enum
  (`conditional: spin === "split"` for the two side buttons, `conditional: spin === "stack"`
  for the `aux` span), remove the `<span class="extra">…</span>` wrappers inside each button
  (per C-5), and show the button content as the rendered snippet — `{left}` for the decrement
  button, `{right}` for the increment button.

**Suggested implementation** (Anatomy button shape, both split and stack):
```svelte
<button class="left" type="button" conditional: spin === "split">{left}</button>
…
<span class="aux" conditional: spin === "stack">
  <button class="right" type="button">{right}</button>
  <button class="left" type="button">{left}</button>
</span>
<button class="right" type="button" conditional: spin === "split">{right}</button>
```

**Notes:** this is a rename/label + enum + wrapper-removal sync of the doc block, kept otherwise
minimal.

### (C-4) Tests

**Where:** `tests/NumberInput.svelte.test.ts`, `tests/NumberField.svelte.test.ts`.

**Required behavior:** update every reference to the old surface so the suites exercise the new
one. Apply this mapping wherever it occurs (anchors are hints; line numbers drift):
- `spin: true` → `spin: "split"` (all occurrences except the stacked case below).
- `spin: false` (the `props.spin = false` reassignment in the "pointer hold … arrows are native"
  test) → `props.spin = "none"`.
- The stacked-layout renders — `render(NumberInput, { spin: true, stack: true })` and the
  NumberField `numberInput: { spin: true, decrement, increment, stack: true }` — MUST become
  `{ spin: "stack" }` (drop the separate `stack` key) with the snippets passed as `left` / `right`.
- Snippet keys `decrement:` / `increment:` (NumberInput "renders labelled snippets…" test) →
  `left:` / `right:`. The NumberField test's local `decrement` / `increment` snippet consts MAY
  keep their variable names but MUST be passed via `left` / `right` keys; renaming the consts to
  `left` / `right` for clarity is acceptable.

**Notes:** the "renders labelled snippets and supports split and stack layouts" assertions,
the `aria-value*` / `role` assertions, and the disabled-at-bounds assertions all stay — only the
prop inputs change. Do not weaken any assertion.

### (C-5) Spinner buttons — remove the `EXTRA` wrapper, size the default svg

**Where:** the `decBtn` and `incBtn` snippet bodies in `NumberInput.svelte`.

**Required behavior:**
- Each button MUST no longer wrap its content in `<span class={cls(PARTS.EXTRA, effVariant)}>…</span>`.
  The `{#if left}…{:else}…svg…{/if}` (resp. `right`) block MUST become a direct child of the
  `<button>`. `NumberInput` no longer emits any `PARTS.EXTRA` element.
- The default `<svg>` in each button MUST carry `style="width:100%;height:100%;"` and MUST drop
  the fixed `width="10" height="10"` attributes. All other svg attributes (`viewBox="0 0 16 16"`,
  `aria-hidden`, `focusable`, `xmlns`) and the `<path>` glyphs MUST stay unchanged.

**Suggested implementation** (decrement button, increment mirrors it with its two-path glyph):
```svelte
<button class={cls(PARTS.LEFT, effVariant)} type="button" tabindex="-1" aria-label={ariaDecLabel}
  disabled={decDisabled} onpointerdown={hdown(-1)} onpointerup={hup} onpointerleave={hup} onpointercancel={hup}>
  {#if left}
    {@render left(effVariant)}
  {:else}
    <svg style="width:100%;height:100%;" aria-hidden="true" focusable="false" xmlns="http://www.w3.org/2000/svg" viewBox="0 0 16 16">
      <path d="M0 7h16v2H0z" />
    </svg>
  {/if}
</button>
```

**Notes:** follows `DarkToggle.svelte`'s default-svg pattern (`style="width:100%;height:100%;"`,
no wrapper). Reference it for the intended shape.

**Risk:** removing the `PARTS.EXTRA` element drops the `svs-number-input__extra` styling hook for
the buttons — an intended, accepted styling-surface change (see Accepted risks). No test asserts
that element, so tests stay green.

## Invariants / expected behavior

- `spin === "none"`: no buttons, no `role`/`aria-value*`, ArrowUp/Down not intercepted.
- `spin === "split"`: decrement before input, increment after; `role="spinbutton"` + `aria-value*` present.
- `spin === "stack"`: single `PARTS.AUX` span after the input containing increment then decrement; spinbutton semantics present.

## Out of scope

- `src/lib/_svseeds/NumberField.svelte` **source** — no edit. Its `numberInput` bag references
  `NumberInputProps`, so the new surface propagates by type; only its **test** changes.
  NumberField's own `left` / `right` **field-area snippet props** are unrelated and MUST stay.
- `_core.ts`, any other component, and any runtime behavior/mechanics beyond the changes in
  C-1..C-5 (the spinner internals rework is limited to those items).
- No exported enum type alias, no deprecation aliases for the removed props.

## Accepted risks / tradeoffs

- Breaking change to a public component API, taken deliberately for a cleaner surface; no shims.
- `left`/`right` name a semantic role, not a physical position (documented in Background); accepted
  for consistency with the `PARTS.LEFT`/`PARTS.RIGHT` styling parts.
- Removing the `PARTS.EXTRA` `<span>` drops NumberInput's `svs-number-input__extra` styling hook
  for the spinner buttons; accepted to match `DarkToggle` and let the svg size follow the button.
  Button-glyph sizing now comes from the `LEFT`/`RIGHT` button CSS rather than a fixed `10×10`.
