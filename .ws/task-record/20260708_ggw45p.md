# Task: Rename Calendar's `prev`/`next` snippet props to `left`/`right`

Target files:
- `src/lib/_svseeds/Calendar.svelte` (public props)
- `tests/Calendar.svelte.test.ts` (follow the rename)

References (read before editing):
- `src/lib/_svseeds/Calendar.svelte` — the doc-comment interface mirror, the
  exported `CalendarProps`, the `$props()` destructuring, and the two
  `{@render}` sites all use these props.
- `.ws/policy/coding-style.md` — project conventions (the doc-comment `CalendarProps`
  block at the top of the file mirrors the exported interface; keep them in sync).

## Goal

Rename Calendar's two page-navigation snippet props `prev` → `left` and
`next` → `right`. Every other component uses `left`/`right`, and Calendar's own
navigation buttons already carry the `PARTS.LEFT`/`PARTS.RIGHT` styling parts, so
`prev`/`next` is the lone inconsistency. This is a pure public-API rename with no
behavioral change.

## Done when

- `CalendarProps` exposes `left`/`right` (not `prev`/`next`), and the two
  navigation buttons render the renamed snippets.
- The doc-comment `CalendarProps` mirror at the top of `Calendar.svelte` matches
  the exported interface.
- `tests/Calendar.svelte.test.ts` passes without a `prev`/`next` prop reference.
- `bun run check` and the scoped Calendar test are green.

## Background / constraints

- **Behavior must not change.** Only prop *names* move; the button rendering
  logic, keyboard handling, and month navigation stay identical.
- **DateInput / DateField need no edit.** Both consume Calendar via
  `Omit<CalendarProps, …>` bags that do **not** name `prev`/`next`, so the rename
  flows through their public types automatically — do not touch those files.
- The internal handlers/functions (`prevMonth`, `nextMonth`, `hprev`, `hnext`)
  and the keyboard-handler local `next` are private implementation and out of
  scope — leave them as-is.
- The button fallback text `Previous`/`Next` is default rendered content, not a
  prop name — leave it unchanged.

## Scope of changes

- [ ] (C-1) `Calendar.svelte`: rename props `prev` → `left`, `next` → `right`.
- [ ] (C-2) `Calendar.svelte.test.ts`: follow the rename in the snippet-override
      test.

## Detailed changes

### (C-1) Rename the props in `Calendar.svelte`

**Where:** four sites, all keyed on the prop names `prev`/`next`:
1. The doc-comment `interface CalendarProps` mirror near the top of the file
   (`prev?: Snippet<[string]>;` / `next?: Snippet<[string]>;`).
2. The exported `export interface CalendarProps` block (same two lines).
3. The `$props()` destructuring (the `prettier-ignore` line — the props appear as
   `…, prev, next, …`).
4. The two `{@render}` sites in the top navigation row.

**Required behavior:**
- The prop keys MUST be `left` and `right`; `prev` and `next` MUST NOT remain as
  prop names anywhere in the file. The `Snippet<[string]>` types are unchanged.
- `left` MUST render in the `PARTS.LEFT` button (the `onclick={hprev}` button) and
  `right` in the `PARTS.RIGHT` button (the `onclick={hnext}` button) — i.e. the
  left/right positions are preserved; only the prop identifier changes.
- The doc-comment mirror MUST match the exported interface after the change.

**Suggested implementation** (illustrative):
```svelte
    left?: Snippet<[string]>;
    right?: Snippet<[string]>;
```
```svelte
    <button class={cls(PARTS.LEFT, variant)} type="button" onclick={hprev}
      >{#if left}{@render left(variant)}{:else}Previous{/if}</button
    >
    …
    <button class={cls(PARTS.RIGHT, variant)} type="button" onclick={hnext}
      >{#if right}{@render right(variant)}{:else}Next{/if}</button
    >
```
In the `$props()` destructuring, replace `prev, next` in place with `left, right`.

**Notes:** Keep the `hprev`/`hnext` handler names and the `Previous`/`Next`
fallback text as-is (out of scope).

**Risk:** Low — mechanical rename; TypeScript/`svelte-check` catches any missed
reference.

### (C-2) Follow the rename in `Calendar.svelte.test.ts`

**Where:** the snippet-override test titled
`"day/weekday/label/prev/next/bottom snippets override defaults"`.

**Required behavior:**
- The test MUST pass the snippets under the `left`/`right` prop keys (not
  `prev`/`next`).
- The test title and the `snippet0(…)` testids MUST use `left`/`right` so the
  vocabulary stays consistent.

**Suggested implementation** (illustrative):
- Title → `"day/weekday/label/left/right/bottom snippets override defaults"`.
- `prev: snippet0("prev", "<")` → `left: snippet0("left", "<")`.
- `next: snippet0("next", ">")` → `right: snippet0("right", ">")`.
- `getByTestId("prev")` → `getByTestId("left")`; `getByTestId("next")` →
  `getByTestId("right")`.

**Notes:** The separate test titled
`"prev/next change display by one month, not value"` exercises **month
navigation** (it clicks the `PARTS.RIGHT` button and does not use these snippet
props); leave that test — including its title — unchanged.
