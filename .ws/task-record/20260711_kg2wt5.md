## Goal

The Sortable component announces keyboard-DnD and multi-select actions through a `aria-live="polite"` SR-only region, but the wording is hard-coded English. Let consumers supply their own message text (e.g. for localization) through a typed, type-safe API, while the current English wording remains the default when nothing is supplied.

## Done when

- A new exported `SortableMessages` interface defines all seven announcement formatters; `createSortableGroup` accepts a third `messages?: Partial<SortableMessages>` argument, and both `Sortable` and `SortableGroup` accept a `messages?: Partial<SortableMessages>` prop.
- Supplying `messages` overrides the corresponding announcements; every unspecified formatter falls back to the current English wording, so existing behavior (and existing test assertions) is unchanged when `messages` is omitted.
- All seven announcement call sites go through the resolved message set — none keep an inline literal string.
- ADR `.ws/decision/010-sortable-announce-messages.md` is created with the body specified in C-5.
- `bun run check` and the two Sortable test files pass; the new custom-message tests pass.

## Background / constraints

- The live region and its wiring (`aria-live`, `messageOwnerId` routing, focus handling) are **not** changing — only where the message *text* comes from.
- Five messages are built inside `SortableController` (which already receives `presentation` / `motion` config through `createSortableGroup`); two selection messages are built in the component's `toggleSelection`, which already holds a `controller` reference. Centralize the resolved message set on the controller so both surfaces read one config.
- The controller is obtained three ways: an explicit `group` prop, a `SortableGroup` context, or a standalone fallback the component creates itself (`Sortable.svelte`: `const controller = group ?? context ?? createSortableGroup()`). A component-level `messages` prop can only configure the controller the component **creates itself**; when an external `group`/context controller is in effect, that controller's own messages win and the local prop is ignored — exactly as `presentation`/`motion` already behave. This precedence MUST be documented, not enforced with new runtime plumbing.
- **Default wording must stay byte-identical** to today's strings — the existing tests assert them verbatim (e.g. `tests/Sortable.svelte.test.ts` checks `"Grabbed a"`, `"Moved to position 2 of 3"`, `"a selected, 1 items selected"`).

## Detailed changes

### (C-1) `SortableMessages`, defaults, and controller threading — `Sortable.svelte` module

**Where:** module `<script module>` (near the `SortableGroupController` interface / `createSortableGroup` / `SortableController`).

**Required behavior:**

- MUST export an interface with exactly these seven formatter members (parameter order and semantics are the contract; `index` is the **1-based** display position):

  ```ts
  export interface SortableMessages {
    grabbed(key: string, index: number, total: number): string;
    atPosition(index: number, total: number): string;
    moved(index: number, total: number): string;
    dropped(index: number, total: number): string;
    cancelled(index: number, total: number): string;
    selected(key: string, count: number): string;
    deselected(key: string, count: number): string;
  }
  ```

- MUST provide a module-level default whose output is **byte-identical** to the current strings (exact):

  ```ts
  const defaultSortableMessages: SortableMessages = {
    grabbed: (key, index, total) => `Grabbed ${key}, position ${index} of ${total}. Use arrow keys to move, space to drop, escape to cancel.`,
    atPosition: (index, total) => `At position ${index} of ${total}.`,
    moved: (index, total) => `Moved to position ${index} of ${total}.`,
    dropped: (index, total) => `Dropped at position ${index} of ${total}.`,
    cancelled: (index, total) => `Cancelled, returned to position ${index} of ${total}.`,
    selected: (key, count) => `${key} selected, ${count} items selected.`,
    deselected: (key, count) => `${key} deselected, ${count} items selected.`,
  };
  ```

- `createSortableGroup` MUST take a third optional parameter `messages?: Partial<SortableMessages>`, resolve it as `{ ...defaultSortableMessages, ...messages }`, and pass the resolved `SortableMessages` into the `SortableController`.
- `SortableGroupController` MUST gain `readonly messages: SortableMessages;`, backed by the resolved set on `SortableController`.
- The five keyboard call sites MUST call the resolved formatters instead of inline literals, preserving their current arguments — passing the **1-based** position where the literal currently used `index + 1` / `p.index + 1`:
  - `grabKeyboard`: `this.messages.grabbed(key, index + 1, member.items.length)`
  - `stepKeyboard` (blocked branch): `this.messages.atPosition(p.index + 1, p.total)`
  - `stepKeyboard` (moved branch): `this.messages.moved(p.index + 1, p.total)`
  - `dropKeyboard`: `this.messages.dropped(p.index + 1, p.total)`
  - `cancelKeyboard`: `this.messages.cancelled(p.index + 1, p.total)`

**Suggested implementation:** add a `#messages` field set from a new constructor parameter, with `get messages()` returning it; mirror how `presentation` is already stored and exposed via `get variant()/get styling()`.

**Risk:** an off-by-one if a call site passes the 0-based index — the defaults expect 1-based. Keep the `+ 1` at the call sites.

### (C-5) ADR — create `.ws/decision/010-sortable-announce-messages.md`

Create the file with **exactly** this body:

```md
# 010. Sortable announcements are customized via a `SortableMessages` formatter-function object

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-11
**Affects:** `Sortable` / `SortableGroup` public prop surface (`messages`), the `createSortableGroup` factory signature, and the new exported `SortableMessages` type

## Context

Sortable announces keyboard drag-and-drop and multi-select actions through an
`aria-live="polite"` SR-only region. The wording was hard-coded English built at
seven call sites (five in the group controller, two in the component's selection
toggle). Consumers need to localize or reword these announcements. Each message
interpolates a small set of runtime values — an item key, a 1-based position, a
total, or a selection count — so the customization surface must carry those
values to the consumer, not just accept a static string. How to shape that
surface was the open choice.

## Decision

Introduce an exported `SortableMessages` interface: one formatter function per
announcement, each receiving the values it needs (`grabbed(key, index, total)`,
`atPosition(index, total)`, `moved`, `dropped`, `cancelled`,
`selected(key, count)`, `deselected(key, count)`; `index` is 1-based). A
module-level default provides the existing English wording. Consumers pass a
`Partial<SortableMessages>` that is merged over the default
(`{ ...defaults, ...messages }`), so any subset can be overridden and unspecified
formatters keep the English default.

The resolved message set lives on the group controller (`SortableGroupController.messages`),
because five of the seven announcements originate there. It is configured through
`createSortableGroup(presentation, motion, messages?)` (a third optional
argument) and surfaced as a `messages` prop on both `Sortable` and `SortableGroup`,
which forward it into the controller they create. When an external `group`
controller or a `SortableGroup` context is in effect, that controller's own
messages win and a local `messages` prop is ignored — identical to how
`presentation`/`motion` precedence already works.

## Consequences

- Announcements are fully localizable/customizable, with the runtime values
  exposed to the consumer's formatter and type-checked against `SortableMessages`.
- Partial override plus default merge means a consumer overrides only the strings
  they care about; omitting `messages` reproduces the prior wording byte-for-byte.
- One config surface (the controller) feeds both the keyboard and selection
  announcements, so there is no second place to configure.
- The prop/argument shape is now public API; changing the formatter signatures or
  the merge semantics later would be a breaking change.
- The precedence rule (external controller's messages win over a local prop) is a
  documented behavior reviewers should preserve, not "fix".

## Rejected alternatives

- **Template strings with placeholders** (e.g. `"{key} at {index}/{total}"`).
  Simpler to type, but cannot type-check the available placeholders, handles
  plural/positional grammar poorly, and pushes string-templating into the
  component. Formatter functions keep grammar in the consumer's hands and are
  type-safe.
- **A single `announce(type, params)` override callback.** Flexible, but forces
  every consumer to switch over all message types and makes per-message fallback
  to the English default awkward; the per-message object with partial merge gives
  clean, incremental overrides.
- **Configure messages only on `createSortableGroup`.** Minimal API, but standalone
  and `SortableGroup`-context consumers never call the factory, leaving them no way
  to customize. Component props on both `Sortable` and `SortableGroup` cover every
  usage form.
```

## Invariants / expected behavior

- With no `messages` supplied anywhere, every announcement is byte-identical to today.
- A `messages` formatter supplied on a standalone `Sortable`, or on the controlling `createSortableGroup`/`SortableGroup`, replaces the matching announcement; unspecified formatters stay English.
- Selection announcements read from the same resolved set as keyboard announcements.

## Out of scope

- The live region markup, `messageOwnerId` routing, focus management, and any DnD behavior.
- Enforcing/merging a local `messages` prop against an external controller's messages (precedence is documented, not runtime-merged).
- Any change to `.ws/architecture/overview.md` — adding one prop + type to an existing component does not shift the system's shape.

## Accepted risks / tradeoffs

- When an external `group`/context controller is used, a `messages` prop on the child `Sortable` is silently ignored. This matches existing `presentation`/`motion` precedence and is documented in the doc comments and ADR.
