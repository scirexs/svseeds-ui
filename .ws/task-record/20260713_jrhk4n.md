# Task: Make FileField's `content` prop optional

Target file: `src/lib/_svseeds/FileField.svelte`
Also created: `.ws/decision/013-filefield-content-optional-top-level.md` (new ADR — does not exist yet)
Also edited: `tests/FileField.svelte.test.ts`

References (read before editing):

- `src/lib/_svseeds/FileInput.svelte` — the primitive FileField wraps; note its
  `children` prop is **required** (`FileInputReqdProps = "children"`) and typed
  `Snippet<[File[], boolean, string]>`.
- `.ws/policy/coding-style.md` — §11 (`@component` comment), §12 (Props order),

- `src/lib/_svseeds/TextField.svelte` — gold-standard field for style reference.

## Goal

`FileField.content` is a `Snippet` that supplies the **default** `<FileInput>`'s
zone content. It is currently a **required** prop, but when the caller passes
their own control as `children` (the wrapper usage
`<FileField><FileInput/></FileField>`), `content` is unused yet still forced by
the type. Make `content` **optional** so the wrapper usage no longer has to pass
a throwaway `content`, and stop rendering the default `<FileInput>` when no
`content` is available to feed its required `children`.

## Done when

- `content` is optional in `FileFieldProps`; `FileField` renders with `children`
  and **no** `content` without a type error.
- When `children` is provided, the default `<FileInput>` is not rendered
  (unchanged); when neither `children` nor `content` is provided, no default
  `<FileInput>` is rendered (new).
- The `@component` Types block, the trailing note, and the Anatomy sketch match
  the new optionality.
- ADR `013` is created with the exact body in (C-4).
- `bun run check`, the scoped tests, and the scoped `fmt:check` all pass.

## Background / constraints

- `content` is passed straight through as the default `<FileInput>`'s `children`
  (`<FileInput ... children={content} />`). Because `FileInput.children` is
  **required and non-nullable**, a bare `content?` passed there would fail
  `svelte-check`. The render must therefore be guarded on `content` (C-2).
- **Behaviour that must keep working:** the default-input path
  (`<FileField content={...} />`) is the primary usage and must be unchanged —
  same DOM, same `content` snippet arguments `(files, dragover, variant)`, same
  `fileInput` bag forwarding, same `multiple` / `name` forwarding. The wrapper
  path (`children` provided) is likewise unchanged.
- `_core` needs no change.

## Scope of changes

- [ ] (C-1) Make `content` optional in the exported `FileFieldProps` and set
  `FileFieldReqdProps` to `never`.
- [ ] (C-2) Guard the default `<FileInput>` render on `content` in the template.
- [ ] (C-3) Sync the `@component` Types block, the trailing note, and the
  Anatomy sketch with the new optionality.
- [ ] (C-4) Create ADR `013` recording *why top-level-optional and not folded
  into the `fileInput` bag*.
- [ ] (C-5) Add a test covering the wrapper usage with `children` and **no**
  `content`.

## Detailed changes

### (C-1) `content` optional + `FileFieldReqdProps = never`

**Where:** the exported `interface FileFieldProps` and the `FileFieldReqdProps`
type alias in `<script module>`.

**Required behavior:**

- `content` MUST become optional: `content?: Snippet<[File[], boolean, string]>;`
  (keep the existing trailing `//` annotation).
- `FileFieldReqdProps` MUST become `never` (no required props remain) — matching
  coding-style §14 (`export type NameReqdProps = "children"; // …, or never`).

**Notes:** §12 requires only that required props precede optional ones; with zero
required props the current position of `content` (first) is compliant, so leave
its position unchanged — this keeps the diff minimal. Do **not** reorder the
interface.

**Risk:** low; widening a required prop to optional is source-compatible for
existing callers.

### (C-2) Guard the default `<FileInput>` render

**Where:** the `{#if children} … {:else} … {/if}` block inside
`<div class={cls(PARTS.MIDDLE, variant)}>`.

**Required behavior:** The default `<FileInput ... children={content} />` MUST NOT
be rendered when `content` is `undefined` (so a possibly-undefined `content` is
never passed to `FileInput`'s required `children`, and `svelte-check` stays
clean). When `children` is provided it still wins; when neither is provided,
nothing is rendered in that slot.

**Suggested implementation** (illustrative):

```svelte
{#if children}
  {@render children()}
{:else if content}
  <FileInput {...fileInput} {multiple} {name} children={content} />
{/if}
```

**Risk:** low. The only behavioural change is the degenerate
"no `children`, no `content`" case, which previously could not occur (content was
required) and now renders no default input — an acceptable no-op for a misuse.

### (C-3) Sync `@component` doc

**Where:** the leading `@component` HTML comment.

**Required behavior:**

- The Types block MUST mark `content` optional (`content?:`), matching (C-1).
- The trailing note (currently: "`fileInput` configures the default `<FileInput/>`
  only; when `children` is provided the bag is ignored and the child control owns
  its own props.") MUST also state that `content` is the default control's zone
  content only — ignored when `children` is provided, and that the default
  control is not rendered when neither `content` nor `children` is given.
- The Anatomy sketch line
  `{#if children}{@render children()}{:else}<FileInput />{/if}` SHOULD be updated
  to reflect the `content` guard (e.g. `{:else if content}<FileInput />`).

**Notes:** Prettier does not lint inside the `@component` comment, so this sync is
verified by hand — keep the Types block identical in shape to the exported
`interface`.

### (C-4) ADR 013

**Where:** create `.ws/decision/013-filefield-content-optional-top-level.md`
(this path does not exist yet — it is a deliverable of this task, not a missing
reference). Use this exact body:

```md
# 013. Keep FileField's `content` optional at the top level, not folded into `fileInput`

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-13
**Affects:** FileField public props API (`content` becomes optional)

## Context

`FileField` wraps a default `<FileInput>` and passes the `content` snippet through
as that input's required `children` (its drop-zone content). `content` was a
**required** top-level prop. But `FileField` also supports a wrapper usage where
the caller supplies their own control as `children`
(`<FileField><FileInput/></FileField>`); in that mode `content` (and the
`fileInput` config bag) are unused, yet the required type still forced callers to
pass a throwaway `content`.

Two shapes could fix this:

1. Keep `content` at the top level and make it optional.
2. Remove top-level `content` and move it into the existing `fileInput` bag as
   `fileInput.children` (the bag already forwards to the default `<FileInput>`),
   so all default-input configuration lives in one place and the bag's required
   `children` gives a tighter compile-time guarantee.

## Decision

Take option 1: `content` stays a top-level prop and becomes optional
(`content?`), and the default `<FileInput>` is rendered only when `content` is
present. `FileFieldReqdProps` becomes `never`.

## Consequences

- The primary usage `<FileField content={...} />` keeps its concise, flat form;
  no caller has to nest zone content under `fileInput={{ children: … }}`.
- The change is source-compatible: existing callers that already pass `content`
  are unaffected.
- The compile-time guarantee is softer than option 2: the type no longer forces
  `content` for the default-input path. Supplying neither `content` nor
  `children` renders no control instead of erroring — an accepted trade for the
  ergonomics of the common path.

## Rejected alternatives

- **Fold `content` into the `fileInput` bag (`fileInput.children`).** Conceptually
  cleaner — one bag for all default-input config, with a required `children`
  enforcing zone content whenever the default input is configured. Rejected
  because it penalises the *common* case: the primary, most-frequent usage
  (default input with zone content) would have to nest into
  `fileInput={{ children: … }}`, trading everyday ergonomics for a guarantee that
  only matters in the rarer degenerate case. It is also a larger breaking change
  across callers, tests, and the type annotations.
```

**Notes:** ADR files under `.ws/` are workflow artifacts, not product code — keep
them out of the `fmt:check` scope (do not run Prettier over `.ws/`).

### (C-5) Test: wrapper usage without `content`

**Where:** `tests/FileField.svelte.test.ts`, in the
`"FileField layout and default child"` describe block (a sibling of the existing
`"reserves bottom, replaces child, and forwards content snippet"` test, which
already builds a `children` `createRawSnippet` and asserts `input` is null when
`children` is passed).

**Required behavior:** Add a case that renders `FileField` with a `children`
snippet and **no** `content` prop, asserting that: the child renders, no default
`<input>` is present, and no `[data-testid="content"]` node is present. This is
the regression guard for the bug being fixed (wrapper usage must type-check and
render without `content`).

**Suggested implementation** (illustrative):

```ts
test("renders children without a content prop", () => {
  const children = createRawSnippet(() => ({ render: () => "<span data-testid='replacement'>custom</span>" }));
  const { container, getByTestId } = render(FileField, { children });
  expect(getByTestId("replacement")).toBeTruthy();
  expect(container.querySelector("input")).toBeNull();
  expect(container.querySelector('[data-testid="content"]')).toBeNull();
});
```

**Notes:** Keep the existing tests as-is — they all pass `content` and remain
valid. Do not remove `content` from tests that assert default-input behaviour.

## Invariants / expected behavior

- Default-input path (`content` present, no `children`): unchanged rendering and
  `content` snippet arguments `(files, dragover, variant)`.
- Wrapper path (`children` present): unchanged; `content` and `fileInput` ignored.
- New: neither present → no default `<FileInput>` rendered.
