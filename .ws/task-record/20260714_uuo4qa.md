# Task: extend Calendar's `bottom` snippet with a control object

Target file: `src/lib/_svseeds/Calendar.svelte`
Test file: `tests/Calendar.svelte.test.ts`

References (read before editing):

- `src/lib/_svseeds/DateInput.svelte` — the pattern to mirror. It bundles control
  functions into a `DateInputCtl` object (`const ctl: DateInputCtl = { toggle, show, hide, clear };`),
  exports the type from `<script module>`, and passes it as the snippet's first
  positional arg alongside state and variant: `Snippet<[DateInputCtl, boolean, string]>`
  → `(ctl, open, variant)`. Follow the same shape.
- `.ws/policy/coding-style.md` — project conventions (§11 `@component` Types/Behavior
  blocks, §14 `<script module>` exports).

## Goal

Give the `bottom` snippet the information and actions it needs to control its own
content: (1) whether the calendar grid or the year/month picker is currently shown,
and (2) a `clear` action that resets the selected `value`. To avoid a long
positional arg list, bundle the snippet's control functions into a `CalendarCtl`
object, mirroring `DateInput`'s `DateInputCtl` pattern exactly.

## Done when

- The `bottom` snippet signature is `Snippet<[CalendarCtl, boolean, string]>` =
  `(ctl, picking, variant)`, where `CalendarCtl = { setToday: () => void; clear: () => void }`.
- A new `clear()` control sets `value` to `undefined`; it is reachable through
  `ctl.clear` in the `bottom` snippet.
- The `bottom` snippet receives the current `picking` state as its second arg so it
  can tell the grid view (`false`) from the year/month picker view (`true`).
- `CalendarCtl` is exported from `<script module>` (like `DateInputCtl`) and appears
  in the `@component` Types block; the Behavior note for `bottom` reflects the new shape.
- `bun run check`, the scoped Calendar test, and the scoped format check all pass.

## Background / constraints

- Current signature is `bottom?: Snippet<[string, () => void]>` = `(variant, setToday)`.
  This is a **breaking change** to that snippet's public API — that is accepted
  (see Accepted risks).
- `setToday` (existing) selects today, jumps to its month, and closes the
  MonthPicker (`picking = false`). Its behavior is unchanged; it moves from a bare
  positional arg into `ctl.setToday`.
- Behavior that MUST keep working: prev/next paging, the label toggling the embedded
  MonthPicker, day selection/roving focus, transitions, and all existing `bottom`
  interactions via `setToday`.

## Detailed changes

### (C-1) `bottom` type + `CalendarCtl` export

**Where:** `<script module>` — the `CalendarProps` interface and the type exports
alongside `DayCtx` / `TransitionProp`; and the mirrored `@component` Types block at
the top of the file.

**Required behavior:**

- The `bottom` prop MUST be typed `Snippet<[CalendarCtl, boolean, string]>`.
- `CalendarCtl` MUST be `{ setToday: () => void; clear: () => void }` and MUST be
  exported from `<script module>` (mirroring `export type DateInputCtl` in DateInput).
- The `@component` Types block MUST list `CalendarCtl` and MUST show the updated
  `bottom` signature, staying in sync with the interface (coding-style §11 — Prettier
  does **not** check inside the `@component` comment, so verify this by hand; see
  `.ws/knowledge/svelte-component-comment-block-skips-prettier.md`).

**Suggested implementation:**

```ts
bottom?: Snippet<[CalendarCtl, boolean, string]>; // Snippet<[ctl,picking,variant]>
```
```ts
export type CalendarCtl = {
  setToday: () => void;
  clear: () => void;
};
```

### (C-2) `clear()` and the `ctl` object

**Where:** instance `<script>`, near the existing `setToday()` control function
(the `ctl` const goes at the end of the Event Handlers section, as DateInput places
`const ctl: DateInputCtl = { … };`).

**Required behavior:**

- `clear()` MUST set the bindable `value` to `undefined` and MUST do nothing else.
- `ctl` MUST be a plain `const` (stable references; not `$derived`) of type
  `CalendarCtl` containing `{ setToday, clear }`.

**Suggested implementation:**

```ts
function clear() {
  value = undefined;
}
```
```ts
const ctl: CalendarCtl = { setToday, clear };
```

**Notes:** `setToday` and `clear` are `function` declarations (hoist), so `const ctl`
may reference them regardless of ordering — keep them as `function`s per coding-style
§8 (internal control functions stay `function`).

### (C-3) Render call

**Where:** the `{#if bottom}` block near the end of the template.

**Required behavior:** the snippet MUST be rendered as `{@render bottom(ctl, picking, variant)}`
(ctl first, then the current `picking` state, then `variant`).

**Suggested implementation:**

```svelte
{#if bottom}
  <div class={cls(PARTS.BOTTOM, variant)}>{@render bottom(ctl, picking, variant)}</div>
{/if}
```

### (C-4) `@component` Behavior note

**Where:** the `@component` Behavior list, the line currently reading

**Required behavior:** the note MUST describe the new shape — that `bottom` receives a
`CalendarCtl` ({ setToday, clear }), the current `picking` state, and `variant`; that
`setToday` behaves as before; and that `clear` resets `value` to `undefined`.

**Required behavior:**

## Invariants / expected behavior

- `setToday` still selects today, jumps to its month, and closes the MonthPicker.
- `clear` only clears `value`; it does not touch `display`, `picking`, or focus.
- `picking === true` ⇔ the MonthPicker (year/month selection) is shown; `false` ⇔ the
  day grid is shown.

## Out of scope

- No change to `DateInput.svelte` — it does not itself pass `bottom`; its
  `calendar?: Omit<CalendarProps, …>` propagates the new `bottom` type automatically.
- No change to any other snippet (`label`/`left`/`right`/`weekday`/`day`), to
  `MonthPicker`, or to the `_core` primitives.
- No CSS / styling / parts changes.

## Accepted risks / tradeoffs

- The `bottom` snippet signature change is breaking for any caller currently using
  `(variant, setToday)`. Accepted: the library is pre-release and the consolidation
  is the point of the task.
