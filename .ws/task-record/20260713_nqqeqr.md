## Goal

Let callers animate a left/right **slide when the shown month changes while the grid is
visible** (prev/next, keyboard month-cross, date-select month-cross, external `display`
change), via a new opt-in `pageTransition` prop — without changing the existing
`transition` (picker open/close) behavior. A month change made from *inside* the
MonthPicker is deliberately outside the slide's scope (the grid is unmounted while picking;
see Out of scope). Separately, make the `bottom` snippet's `setToday` return from the
MonthPicker to the grid.

## Done when

- A new optional `pageTransition?: TransitionProp` prop exists; when unset the DOM and
  behavior match today exactly (no animation), and existing `transition` is unchanged.
- On a month change made while the grid is shown (prev/next, keyboard, date-select cross,
  external `display`), the `.main` grid re-mounts and `pageTransition.fn` is invoked with
  `params.dir` set to `-1` (earlier month) or `+1` (later month); reduced motion falls
  back to no animation. A MonthPicker-originated month change does **not** slide.
- Invoking `setToday` (the second arg of the `bottom` snippet) selects today, jumps to
  today's month, **and** sets `picking = false` so the grid is shown.
- `@component` Types / Anatomy / Behavior reflect `pageTransition` and the container
  styling requirement.
- ADR `.ws/decision/011-calendar-page-transition.md` is created with the exact body in
  (C-7).
- `bun run check`, the scoped Calendar test, and the scoped format check all pass.

## Background / constraints

- The existing `transition` fires only on the `{#if picking}` switch (MonthPicker ⇔ grid
  mount/unmount) applied to `.middle`. Month navigation only updates the derived `weeks`,
  so nothing re-mounts and no direction is available. The fix adds a remount trigger and a
  direction signal, scoped to `.main`.
- `TransitionProp` already exists in this component's `<script module>` — **reuse it**,
  do not add a new type.
- **Must keep working:** current `transition` behavior, all keyboard/roving-focus logic,
  the grid model (`weeks`/`cellKeys`/`trimCells`), and every existing test. Behavior over
  cleanliness; `_core.ts` needs **no** change.

## Detailed changes

### (C-1) `pageTransition` prop

**Where:** `export interface CalendarProps` (`<script module>`), the `@component` Types
block (the leading comment — §14 keeps the two in sync), and the `$props()` destructure.

**Required behavior:** MUST add an optional `pageTransition?: TransitionProp` to
`CalendarProps` and destructure it in `$props()`. MUST add the identical line to the
`@component` Types block's `interface CalendarProps`. Both are optional, so existing
callers are non-breaking. Place it adjacent to the existing `transition?: TransitionProp`.

### (C-2) Direction detection

**Where:** the States section and the first `$effect.pre` (the one reading `display` /
`value` and calling `syncFocus()`).

**Required behavior:** A `dir` state MUST hold the sign of the last month change: `-1`
when the new shown month is earlier, `+1` when later, unchanged (`0`) otherwise. It MUST
cover **all** navigation paths (prev/next buttons, keyboard, date-select month cross,
MonthPicker select, and external two-way `display` binding). Detection MUST happen in the
existing `display`/`value` `$effect.pre` (do not add per-handler logic), and MUST NOT
alter the existing `syncFocus()` behavior. Detection is universal even though the *slide*
renders only while the grid is mounted (C-4, Out of scope): keeping `dir` correct for the
MonthPicker path is harmless and keeps the detection single-sourced.

**Suggested implementation** (per coding-style §5 — body >1 line ⇒ extract a named function
called inside `untrack`):

```ts
// States
let dir = $state(0);
let prevYm = initialDisplay;   // plain let, non-reactive; updated inside untrack
```

```ts
$effect.pre(() => {
  display;
  value;
  untrack(() => sync());
});

function sync() {
  const cur = currentDisplay();
  if (!cur.equals(prevYm)) {
    dir = Temporal.PlainYearMonth.compare(cur, prevYm); // -1 | 0 | 1
    prevYm = cur;
  }
  syncFocus();
}
```

**Notes:** `Temporal.PlainYearMonth.compare` returns only the sign, so multi-month jumps
(year paging, a distant MonthPicker pick) still yield `-1` / `+1`.

### (C-3) `pfn` / `pparams` derived

**Where:** next to the existing `tfn` / `tparams` derivations.

**Required behavior:** `pfn` MUST be `pageTransition.fn` when a fn is provided and motion is
not reduced, else `noop` (reuse the existing `noop` and `reduced`). `pparams` MUST be the
caller's `pageTransition.params` with `dir` merged in — the library injects `dir`; the
caller does not supply it.

**Suggested implementation:**

```ts
const pfn = $derived(!reduced && pageTransition?.fn ? pageTransition.fn : noop);
const pparams = $derived({ ...((pageTransition?.params ?? {}) as object), dir });
```

### (C-4) Re-key `.main`

**Where:** the `{:else}` branch of `{#if picking}`, inside `.middle`.

**Required behavior:** The `.main` grid MUST be wrapped in
`{#key currentDisplay().toString()}` and carry `transition:pfn|local={pparams}`, so it
re-mounts on every month change and plays the page transition. The `|local` qualifier MUST
be kept so the picker open/close switch (the parent `{#if}`) does **not** trigger the page
transition — only a month (key) change does. `.middle`'s existing `transition:tfn|local`
stays as-is. No other element or attribute changes.

**Suggested implementation** (only the changed wrapper shown; the grid's inner content is
unchanged):

```svelte
{:else}
  <div class={cls(PARTS.MIDDLE, variant)} transition:tfn|local={tparams}>
    {#key currentDisplay().toString()}
      <div class={cls(PARTS.MAIN, variant)} role="grid" tabindex="-1" aria-labelledby={idCaption} onkeydown={hkeydown} transition:pfn|local={pparams}>
        <!-- weekday header + weeks: unchanged -->
      </div>
    {/key}
  </div>
{/if}
```

**Note (deliberate boundary — do not "fix"):** because `.main` lives only in the `{:else}`
branch and `|local` suppresses the intro on mount, a month change made *while the MonthPicker
is open* does not slide — the grid is unmounted during picking, and the later picker→grid
return is a `|local`-suppressed mount. This is intended (see Out of scope); every month
change made while the grid is shown does slide. Do not drop `|local` or keep the grid mounted
during picking to force a picker-path slide.

**Risk:** during a cross-slide two `.main` grids are briefly mounted; with `outsideDays`
enabled, a day shared between adjacent months has the same `bind:this={cells[key]}` key in
both grids and can transiently clash. This affects focus only *during* the transition —
once it settles, `trimCells` + the single remaining grid restore a correct `cells` map. See
Accepted risks.

### (C-5) `setToday` closes the picker

**Where:** the `setToday` control function.

**Required behavior:** `setToday` MUST set `picking = false` (in addition to its current
`displayOf(today)` + `select(today)`), so pressing "Today" while the MonthPicker is open
returns to the grid.

```ts
function setToday() {
  displayOf(today);
  select(today);
  picking = false;
}
```

### (C-7) ADR

**Where:** create the new file `.ws/decision/011-calendar-page-transition.md` (this path
does not exist yet — it is the deliverable, not a missing reference).

**Required behavior:** MUST create it with **exactly** this body:

```md
# 011. Calendar month-page slide transition

**Status:** Accepted
**Supersedes:** none
**Date:** 2026-07-13
**Affects:** public API (Calendar `pageTransition` prop), Calendar.svelte grid anatomy

## Context
Calendar's existing `transition` prop only fires on the MonthPicker ⇔ grid mount/unmount (the `{#if picking}` switch); month navigation merely updates the derived `weeks`, so no enter/leave runs and the transition fn never receives a left/right direction. Users want a left/right slide when the shown month changes while the grid is visible (prev/next, keyboard, date-select month cross, external `display` change). Realizing it needs both a remount trigger on month change and a direction signal, without breaking the existing `transition` behavior.

## Decision
Add a separate optional `pageTransition?: TransitionProp` prop for the month slide, leaving `transition` (picker open/close) untouched. Detect direction centrally in the existing `display`/`value` `$effect.pre` by comparing the previous shown month with `Temporal.PlainYearMonth.compare` (`-1`/`0`/`+1`), store it in `dir`, and re-key the whole `.main` grid with `{#key currentDisplay().toString()}`, applying `transition:pfn|local={pparams}` where the library injects `dir` into the params. No default `fn` is bundled — the feature is opt-in, and an unset `pageTransition` resolves to `noop` (prior behavior).

## Consequences
- Existing `transition` users are unaffected; `pageTransition` is additive and backward-compatible.
- Direction is detected in one place, covering every navigation path (buttons, keyboard, date-select, picker, two-way `display`).
- A month change made from within the MonthPicker updates `display` and `dir` but does not itself slide: the grid is unmounted while picking, and the picker→grid return is a `|local`-suppressed mount. The slide covers month changes made while the grid is shown.
- Because the whole `.main` is re-keyed, the weekday-header row slides together with the week rows.
- The caller owns the slide `fn` and its container CSS (`position: relative; overflow: hidden`, to overlay the leaving grid); the library documents the requirement but does not enforce it.
- During a cross-slide two `.main` grids are briefly mounted; with `outsideDays`, shared day keys can transiently clash on `bind:this`, affecting focus only mid-transition.

## Rejected alternatives
- **Overload the existing `transition` prop** (synthesize `dir` into its params and branch on `dir === 0` vs `±1`): zero API surface, but mixes two distinct motions on one fn and changes month-navigation behavior for current `transition` users, weakening backward compatibility.
- **Keep the weekday header fixed via a keyed `role="rowgroup"` wrapper around the week rows**: visually cleaner, but the wrapper cannot be given a proper SVS styling part without adding a Calendar-specific entry to the shared `PARTS` enum (used across the component set); re-keying the whole `.main` is the more minimal, convention-consistent change, at the cost of the header sliding along.
```

## Invariants / expected behavior

- `pageTransition` unset ⇒ output and behavior are byte-for-byte the prior behavior
  (`pfn = noop`); the `{#key}` remount is harmless.
- The page transition plays on month (key) change only, never on the picker open/close
  switch (guaranteed by `|local`).
- A month change originating in the MonthPicker updates `dir` but does not itself play the
  slide (grid unmounted during picking); the subsequent picker→grid return does not slide.
- `dir` sign matches the direction of change across every navigation path; a no-op re-run
  leaves `dir` unchanged.
- After a settled navigation, roving focus (`focusDate` via `cells[d]`) resolves correctly.

## Out of scope

- No default slide `fn` bundled into `_core.ts` (caller supplies `fn`); `_core.ts` is not
  touched.
- No change to `transition`, the grid model, keyboard handling, or MonthPicker.
- **Sliding on a MonthPicker-originated month change is out of scope.** The grid is unmounted
  while the picker is open, and the picker→grid return is a deliberately `|local`-suppressed
  mount. `dir` is still detected for this path, but no slide renders — do **not** add wiring,
  restructure the anatomy, or drop `|local` to force it (that would break C-4/ADR 011 and the
  existing `[role="grid"]`-is-null-while-picking tests).
- No anatomy restructure (no `role="rowgroup"`, no new `PARTS` entry).
- README and demos are not updated (README carries no per-component prop docs; no Calendar
  demo exists in this repo).

## Accepted risks / tradeoffs

- The weekday-header row slides along with the weeks (whole-`.main` re-key). Accepted per
  the design decision (ADR 011); keeping it fixed was rejected.
- With `outsideDays` enabled, mid-transition `bind:this` key clashes between the two briefly
  co-mounted grids can perturb focus *during* the slide only; the settled state is correct.
- With `fixedWeeks: false`, months with differing row counts can show vertical jitter mid-
  slide; a cross-slide `fn` that absolutely-positions the leaving grid keeps the container
  height determined by the entering grid.
