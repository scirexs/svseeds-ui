- `.ws/policy/coding-style.md` — §7 section banners, §8 `hXxx` handler naming and

## Goal

Two related changes to `DateInput`:

1. When the input keeps focus after the overlay closes (Escape dismissal, or a
   calendar pick that returned focus with `closeOnSelect` true), a **pointerdown
   on the already-focused input** currently does nothing — no `focus` event fires
   because focus never left. Add an `onpointerdown` handler so a re-click reopens
   the calendar, matching the "click the field to see the calendar" expectation.
2. Change the `openOnFocus` default from `false` to `true`, so the calendar opens
   on focus/click out of the box.

## Done when

- Clicking the focused input while the overlay is hidden reopens the overlay when
  `openOnFocus` is true (covers the post-Escape and post-pick-return states).
- A pointerdown on the input does **not** open the overlay when `openOnFocus` is
  false, and does **not** close it when it is already open.
- A caller-supplied `onpointerdown` still fires.
- `openOnFocus` defaults to `true`; the `@component` doc and the exported
  interface both say `// (true)`.
- `bun run check`, the scoped `DateInput` test, and the scoped format check pass.

## Background / constraints

- Focus-based opening is governed by `openOnFocus` (`hfocus` calls `show()` when
  `openOnFocus && !returning`). The `returning` flag suppresses the reopen during
  the programmatic `returnFocus()` after a pick, so picks don't immediately
  re-pop. Pointer events do **not** fire during `returnFocus()` (it is a
  programmatic `element.focus()`), so gating the new click-to-open on the same
  `openOnFocus` prop keeps that suppression intact — only a genuine user click
  reopens.
- The overlay's `houtside` document listener only exists while `open` is true and
  ignores pointerdowns inside `root` (which includes the input), so it never
  fights the new handler.
- Opening on pointerdown is idempotent with the focus-based open: on a first click
  from an unfocused state, pointerdown opens and the following `focus` calls
  `show()` again harmlessly. Behaviour that must keep working: pick → close →
  return focus (no spurious reopen); Escape → close with focus retained; outside
  pointerdown → close.

## Detailed changes

### (C-1) `onpointerdown` handler

**Where:** instance `<script>` — the `$props()` destructure, the Event Handlers
section (near `hfocus` / `hkeydown`), the `svelte/elements` type import, and the
`<input>` element.

**Required behavior:**
- The input MUST call `show()` on `pointerdown` when `openOnFocus` is true **and**
  the overlay is hidden (`!open`). This reopens the calendar when focus was
  retained on the input after a close (Escape or pick-with-return).
- When `openOnFocus` is false, pointerdown MUST NOT open the overlay.
- When the overlay is already open, pointerdown MUST NOT toggle it closed
  (no-op).
- A caller-supplied `onpointerdown` MUST still be invoked, and it MUST NOT be
  clobbered by the component's handler (follow the existing re-dispatch pattern:
  destructure it out of `...rest` as `onpointerdown: onpointerdownProp` and call
  it in the handler).

**Suggested implementation:**
```ts
// $props() destructure — add alongside onfocus/onblur/onkeydown:
onpointerdown: onpointerdownProp,

// type import (svelte/elements) — add PointerEventHandler:
import type { FocusEventHandler, FormEventHandler, KeyboardEventHandler, PointerEventHandler } from "svelte/elements";

// Event Handlers section:
const hpointerdown: PointerEventHandler<HTMLInputElement> = (ev) => {
  onpointerdownProp?.(ev);
  if (openOnFocus && !open) show();
};
```
```svelte
<!-- <input> — add after {...rest}, alongside the other component-owned handlers -->
onpointerdown={hpointerdown}
```

**Notes:** `pointerdown` fires before `focus`; on a first click from unfocused,
both paths call `show()` (idempotent). `show()` hoists (a `function`), so ordering
inside the arrow is fine. Match §8 typing — annotate the `const`, not the
parameter.

**Risk:** low; the handler only ever calls `show()`, never `hide()`.

### (C-2) Default `openOnFocus` to `true`

**Where:** the `$props()` destructure, and the two doc-comment occurrences of
`openOnFocus?: boolean; // (false)`.

**Required behavior:**
- The `$props()` default MUST be `openOnFocus = true`.
- The `@component` *Types* block comment (top-of-file) and the exported
  `DateInputProps` interface comment MUST both read `// (true)`.

**Suggested implementation:** change `openOnFocus = false` → `openOnFocus = true`
in the destructure, and both `// (false)` → `// (true)` on the `openOnFocus` line.

**Notes:** keep the `@component` doc in sync with the interface (coding-style §11);
Prettier does not lint inside the `@component` comment, so verify that edit by
hand.

**Risk:** behavioural — an editable (`parse`) input now opens the calendar on
focus by default, and opening moves focus into the calendar (`focusCalendar()`).
This is the intended new default. Its only in-repo fallout is test expectations
(see C-3).

## Invariants / expected behavior

- `openOnFocus` remains the single switch governing "open on interaction" —
  neither focus nor click opens the overlay when it is false.
- Picking a date with `closeOnSelect` true still closes and returns focus without
  the overlay flashing back open.

## Out of scope

- `_core.ts` (no change needed).
- The `hfocus` / `returning` / `returnFocus()` logic, `hkey`/`houtside`
  dismissal, and the overlay flip/positioning.
- `Calendar.svelte`, `DateField.svelte`, and any `focusCalendar()` focus-stealing
  behaviour — do not alter how the overlay grabs focus when it opens.
- Adding a "toggle on click" behaviour — clicking an open overlay must stay a
  no-op.

## Accepted risks / tradeoffs

- With the new default, focusing/clicking an editable date input opens the
  calendar and moves focus into it. Accepted: this is the requested default;
  callers wanting the field to stay closed pass `openOnFocus={false}`.
