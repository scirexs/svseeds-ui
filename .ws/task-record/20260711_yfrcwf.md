# Task: Refine Sortable — clear selection after a move, add `ariaRoleDescription`, and align internals with conventions

Target file: `src/lib/_svseeds/Sortable.svelte` (only file changed)

References (read before editing):
- `.ws/policy/coding-style.md` — esp. §8 (event handlers vs control functions),
  §12 (Props order), §14 (component shape / `_core`), §11 (`@component`).
- `src/lib/_svseeds/_core.ts` — for `SR_ONLY`, the `Vector` type.
- `src/lib/_svseeds/TextField.svelte`, `src/lib/_svseeds/Disclosure.svelte` —
  gold-standard style for `hXxx` const-arrow handlers typed from `svelte/elements`.
- `tests/Sortable.svelte.test.ts` — existing multi-select / keyboard tests to extend.

## Goal

Improve `Sortable` usability and maintainability without changing unrelated
behavior:
1. In multi-select mode, clear the whole selection after a move actually commits.
2. Make the fixed `aria-roledescription` on items overridable via a new prop.
3. Fold three internal cleanups into the same change: use `_core`'s `SR_ONLY` for
   the live region, use `_core`'s `Vector` in place of the local `Point` type, and
   bring the `onPointerXxx` handlers into the project's `hXxx` const-arrow
   convention (coding-style §8).

## Done when

- After a committed multi-select move (pointer drop or keyboard drop), every item
  in the **source** list is deselected; a cancelled drag and a plain
  click-to-select do **not** clear the selection.
- `Sortable` accepts `ariaRoleDescription?: string` (placed right after
  `ariaLabel`), defaulting to `"Sortable Item"`, and each item's
  `aria-roledescription` reflects it.
- The polite live-region `<span>` uses `SR_ONLY` from `_core`; the local `Point`
  type is gone and `Vector` from `_core` is used instead; the five pointer
  handlers are `hpointerdown` / `hpointerover` / `hpointerout` / `hpointerenter` /
  `hpointerup`, defined as typed `const` arrows.
- `bun run check` is clean and the Sortable test suites pass.

## Background / constraints

- **Behaviour over cleanliness.** The move/clone/swap logic, keyboard model,
  followers, confirm delay, FLIP motion, ghost/shadow rendering, and group
  coordination must all keep working exactly as today. Only the four items above
  change.
- Selection state (`selected: SvelteSet<string>`) lives on each `Sortable`
  instance; the shared `SortableController` drives commit. So the controller must
  reach the **source** instance to clear its selection — do it through the
  `SortableMember` abstraction (which already exposes `touch()`), not by having
  the controller own selection.
- The default announced roledescription intentionally changes from the current
  literal `"sortable item"` to `"Sortable Item"` (the new prop default). No test
  asserts the old literal, so this is safe.
- `SortableGroup.svelte` is **not** changed (see Out of scope).

## Scope of changes

- [ ] (C-1) Clear the source list's selection when a move commits (pointer + keyboard).
- [ ] (C-2) Add the `ariaRoleDescription` prop and apply it to each item.
- [ ] (C-3) Replace the live-region inline style with `_core`'s `SR_ONLY`.
- [ ] (C-4) Replace the local `Point` type with `_core`'s `Vector`.
- [ ] (C-5) Convert the `onPointerXxx` functions to `hXxx` typed const arrows (§8).

## Detailed changes

### (C-1) Deselect the source list after a committed move

**Where:** `SortableController.commit()` and `SortableController.dropKeyboard()`;
the `SortableMember` type; the instance `member` object literal.

**Required behavior:**
- The `SortableMember` type MUST gain a no-arg method (suggested name `deselect`)
  that clears that member's selection. The instance `member` object MUST implement
  it by clearing its `selected` set (`selected.clear()`).
- `commit()` MUST call `this.#active.source.<deselect>()` **only inside the
  existing `if (this.#active)` block** (i.e. only when a real drag was active),
  after `#commitFollowers()`. A pointer-up with no active drag (a plain
  click-to-select) MUST NOT clear the selection.
- `dropKeyboard()` MUST call the source member's deselect (`active.source`) after
  `#commitFollowers()`, before/around the existing `#clearDrag()`.
- `cancel()` / `cancelKeyboard()` MUST NOT clear the selection.

**Suggested implementation:**
- Add to the member type: `deselect(): void;`
- In the instance `member` literal, next to `touch()`:
  `deselect() { selected.clear(); }`
- `commit()`:
  ```ts
  if (this.#active) {
    this.#commitSwap();
    this.#commitFollowers();
    this.#active.source.deselect();
  }
  ```
- `dropKeyboard()`: after `this.#commitFollowers();`, add
  `active.source.deselect();`

**Notes:** clearing is unconditional across modes (move/clone/swap) and regardless
of whether followers existed — "after a committed move, nothing stays selected".
Clearing an already-empty set (non-multiple lists) is a harmless no-op.

**Risk:** low. Guarded by `if (this.#active)`, so the no-move click-select path is
untouched; cancel paths do not call it.

### (C-2) `ariaRoleDescription` prop

**Where:** `SortableProps` interface (`<script module>`), the `@component`
*Types* block, the `$props()` destructure, and the `<li>` in the template.

**Required behavior:**
- Add `ariaRoleDescription?: string;` to `SortableProps`, positioned immediately
  after `ariaLabel?: string;` (keep it among the optional props — §12 requires
  required-first, which still holds).
- Mirror the same line in the `@component` *Types* interface block (§11 keeps
  *Types* and `SortableProps` in sync).
- Destructure with a default: `ariaRoleDescription = "Sortable Item"` (place it
  right after `ariaLabel` in the `$props()` list).
- The `<li>` MUST render `aria-roledescription={ariaRoleDescription}`, replacing
  the current hard-coded `aria-roledescription="sortable item"`.

**Notes:** `aria-roledescription` is on `<li>`, not on the `<ul>` that receives
`{...rest}`, so no rest-prop clobber concern. The Anatomy sketch already shows the
bare `aria-roledescription` marker on `<li>` — no Anatomy edit needed.

**Risk:** low; purely additive prop with a default preserving today's behavior
(modulo the intended `"sortable item"` → `"Sortable Item"` casing).

### (C-3) Use `SR_ONLY` for the live region

**Where:** the `<span aria-live="polite">` after the `<ul>`, and the `_core` value
import.

**Required behavior:**
- Add `SR_ONLY` to the existing value import from `./_core`.
- Replace the span's inline `style="position:absolute;…clip:rect(0,0,0,0)…"` with
  `style={SR_ONLY}`.

**Notes:** `_core`'s `SR_ONLY` uses the modern `clip-path:inset(50%)` rather than
the deprecated `clip:rect(...)`; the visual (visually-hidden) result is
equivalent. Matches `FileInput.svelte` / `WheelPicker.svelte` usage.

**Risk:** low.

### (C-4) Replace local `Point` with `_core`'s `Vector`

**Where:** the `type Point = { x: number; y: number };` declaration, its
references, and the `_core` type import.

**Required behavior:**
- Remove `type Point = { x: number; y: number };`.
- Add `Vector` to the type import from `./_core`.
- Replace every `Point` reference with `Vector` (e.g. `SortableShadowState.point`,
  the `#standby.anchor` shape, `PointerVector`'s `#initial` / `#current`, and the
  `point()` / `PointerVector.vector()` return types).

**Notes:** keep the `point(x, y)` factory function (now returning `Vector`) — only
the *type name* changes, not the helper. `_core`'s `Vector` is structurally
identical (`{ x: number; y: number }`).

**Risk:** low; type-only rename.

### (C-5) `onPointerXxx` → `hXxx` typed const arrows (coding-style §8)

**Where:** the five instance functions `onPointerDown` / `onPointerOver` /
`onPointerOut` / `onPointerEnter` / `onPointerUp`, their `<ul>` attribute
bindings, and the `svelte/elements` type import.

**Required behavior:**
- Rename to `hpointerdown` / `hpointerover` / `hpointerout` / `hpointerenter` /
  `hpointerup` and define each as a `const` arrow (not `function`).
- Type each `const` with the matching handler type — `PointerEventHandler<HTMLUListElement>`
  from `svelte/elements` — so `ev` / `ev.currentTarget` are inferred (§8 "type the
  arrow, not the body"). Do not annotate the parameter and cast inside.
- Update the `<ul>` bindings accordingly
  (`onpointerdown={hpointerdown}`, …). These handlers stay attached to the `<ul>`
  in the same positions/order; `{...rest}` still precedes them.

**Suggested implementation:**
```ts
const hpointerdown: PointerEventHandler<HTMLUListElement> = (ev) => { … };
const hpointerover: PointerEventHandler<HTMLUListElement> = (ev) => { … };
const hpointerout: PointerEventHandler<HTMLUListElement> = (ev) => { … };
const hpointerenter: PointerEventHandler<HTMLUListElement> = () => { … };
const hpointerup: PointerEventHandler<HTMLUListElement> = () => { … };
```
Add `PointerEventHandler` to the `svelte/elements` type import (which already
imports `EventHandler`, `HTMLAttributes`, `KeyboardEventHandler`).

**Notes:**
- No TDZ risk: these are referenced only in the template, well after their
  declarations, and none is forward-referenced — the §8 exception (control/action
  functions that hoist) does not apply, so `const` arrows are correct here.
- With `currentTarget` now typed `HTMLUListElement`, the bodies' existing
  `ev.currentTarget instanceof HTMLElement` runtime guards remain valid; you MAY
  drop that now-redundant narrowing, but keeping it is acceptable (behaviour over
  cleanliness). Do not otherwise rewrite the bodies.
- Leave the other handlers as they are: `hblur` / `hkeydown` already follow §8;
  the module-level `ondragstart = () => false` and the `handle` attachment stay
  untouched.

**Risk:** low; naming + typing, no logic change.

## Invariants / expected behavior

- Followers, cross-list move/clone/swap, confirm delay, keyboard grab/step/drop/
  cancel, `dragging` bindable, ghost/shadow, and FLIP motion behave as before.
- Cancelling a drag (pointer `pointercancel`, keyboard `Escape`) preserves the
  current selection.
- A plain click-to-select in multiple mode still toggles selection and is not
  wiped by the subsequent `pointerup`.

## Out of scope

- `SortableGroup.svelte` — no change. `role="group"` needs no `aria-roledescription`;
  it is named via `aria-label` / `aria-labelledby`. The Sortable-vs-group
  asymmetry is accepted.
- Any change to `_core.ts`.
- The `point()` factory, `hblur`, `hkeydown`, `ondragstart`, the `handle`
  attachment, and all controller move/clone/swap algorithms.
- No architecture-overview update: this adds one optional prop and a localized
  behavior; it does not shift the system's shape.

## Accepted risks / tradeoffs

- The default item roledescription text changes `"sortable item"` → `"Sortable Item"`
  (maintainer-requested). Accepted.
- `SR_ONLY` swaps `clip` for `clip-path`; equivalent visually-hidden result.
